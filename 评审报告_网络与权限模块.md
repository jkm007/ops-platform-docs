## 一、网络模块评审

### 1.1 Linux Bridge + VLAN 方案：是否与现有环境兼容？有没有更好的方案？

**结论：方案兼容但存在显著局限，建议 P1 阶段迁移至 OVN/OVS。**

**兼容性分析：**

| 维度 | 评审结论 |
|------|---------|
| 现有环境兼容 | Linux Bridge 与现有 32 台 VM 的 KVM + qcow2 环境完全兼容，无需改造存量。Bridge 命名规范 `vn-{id前8位}` 避免了与现有 Bridge 冲突。 |
| 创建/销毁操作 | PRD 对 Bridge 的创建、NAT 规则配置、dnsmasq 启停、回滚机制描述清晰（4.1.2 节）。 |
| 存量共存 | PRD 1.3 节明确"不破坏现有 32 台 KVM 虚拟机的网络配置"，通过新建 Bridge + 路由互通实现隔离，策略正确。 |

**风险与不足：**

1. **缺少 VLAN 隔离机制**：PRD 中未提及 VLAN ID 的分配与管理。当前方案依赖纯 Bridge 隔离，二层域之间通过"不配置 NAT/路由"来实现隔离，这在高密度租户场景下存在 ARP 广播风暴风险。NAT 网络和隔离网络仅靠 Bridge 隔离，无 VLAN 标签，所有 Bridge 的广播域在物理网卡层面仍可能互相影响。
2. **跨宿主机通信**：PRD 在 TI-09 提到 agent 通过 gRPC 通信，但未说明不同宿主机上属于同一虚拟网络的 VM 之间如何实现二层互通。Linux Bridge 本身无法跨宿主机，需要 VXLAN/GRE overlay 或物理交换机 VLAN trunk。这是 PRD 中**最大的技术盲区**。
3. **推荐方案演进路径**：
   - **P0 阶段**：保持 Linux Bridge（满足 6 台 Dell R740 小规模场景）
   - **P1 阶段**：评估 OVS + VXLAN（利用 KVM 原生支持，跨宿主机二层互通，与 Ceph/K8s 混部兼容）
   - **P2 阶段**：引入 OVN（原生多租户、ACL 即安全组、分布式 DHCP、与 K8s CNI 对接）

**建议补充：**

- 明确跨宿主机二层互通方案（VXLAN 隧道或物理交换机 VLAN trunk）
- 增加 VLAN ID 字段到 virtual_networks 表，避免广播域污染
- 在桥接网络创建时增加"VLAN ID"可选参数

### 1.2 安全组实现：iptables/nftables 方案在混部环境下的性能影响？

**结论：方案可行，但存在性能衰减和规则爆炸风险，建议补充压测指标和 nftables 优先策略。**

**详细分析：**

| 维度 | 评审结论 |
|------|---------|
| 方案正确性 | PRD TI-02/03 描述的 chain 命名规范（`SG-{sg_id}-{ingress/egress}`）和原子下发（临时 chain → 替换 → 回滚）设计合理，是业界成熟做法。 |
| 规则生成逻辑 | 5.3 节流程图描述清晰：遍历安全组规则 → 按优先级排序 → 生成 iptables → agent 下发 → 原子替换。 |
| 多安全组叠加 | AC-SG-08 验收条件覆盖了多安全组叠加场景。 |

**性能风险：**

1. **规则膨胀问题**：每台 VM 最多 5 个安全组，每个安全组可能包含几十条规则。在最坏情况下，单台 VM 的 iptables 规则可达数百条。Linux Bridge 混部场景下（KVM + Ceph + K8s 共享 6 台宿主机），iptables 规则匹配是 O(n) 的线性遍历，大量规则会显著增加网络延迟。
2. **iptables vs nftables**：PRD 中提到"iptables/nftables"但未明确优先级策略。nftables 在内核 4.x+ 中性能优于 iptables（规则匹配从 O(n) 优化为集合查找），且支持原子规则替换。建议**以 nftables 为主、iptables 为 fallback**。
3. **Ceph 存储网络干扰**：Ceph OSD 之间的复制流量、K8s pod 网络流量与 VM 流量共享物理网卡。安全组规则在宿主机 netfilter 层处理，大量 VM 的高频规则匹配可能挤占 CPU，影响 Ceph 的延迟敏感性。
4. **缺失性能基准**：PRD 未给出安全组规则数量与吞吐量/延迟的压测指标。建议补充：
   - 单 VM 100 条规则时的吞吐量衰减率
   - 单宿主机 50 VM 时的 iptables 规则总数上限
   - P99 网络延迟在规则加载后的变化

**安全风险：**

1. **默认规则安全**：PRD 4.2.1 明确默认入方向 DROP、出方向 ALLOW，符合最小权限原则。但需确保默认 DROP 规则在宿主机层面正确配置（不仅是 VM 内部 iptables）。
2. **规则绕过风险**：安全组规则应用于 VM 网卡对应的 Linux Bridge 接口。需确认规则是在 Bridge 的 FORWARD chain 还是 VM tap 接口的 INPUT/OUTPUT chain。建议在 Bridge 的 FORWARD chain 层面控制，避免 VM 内部绕过。
3. **存量 VM 保护**：需明确存量 32 台 VM 的默认安全策略。建议在引入安全组时，为存量 VM 自动绑定一个"存量兼容"安全组（入方向全放行），防止误拦截现有业务。

**建议补充：**

- 明确 nftables 优先策略，iptables 仅作为内核兼容性 fallback
- 增加性能基准指标：100/500/1000 条规则时的吞吐量衰减
- 明确安全组规则生效位置（Bridge FORWARD chain vs VM tap interface）
- 为存量 VM 设计兼容迁移策略

### 1.3 CLB 方案：Nginx/HAProxy 作为内网 CLB 是否合理？高可用如何保证？

**结论：方案合理但高可用设计严重不足，P0 阶段即存在单点故障风险。**

**合理性分析：**

| 维度 | 评审结论 |
|------|---------|
| 代理选型 | Nginx 适合七层 HTTP/HTTPS（URL 转发、SSL 卸载），HAProxy 适合四层 TCP/UDP 和七层 HTTP。PRD 同时支持两种代理类型（TI-07），设计合理。 |
| 配置管理 | PRD 7.13 节明确 `nginx -t` / `haproxy -c` 语法检查 + 回滚机制，graceful reload（4.3.3），安全可靠。 |
| SSL 管理 | 私钥 AES-256-GCM 加密存储（TI-08），证书过期提醒（4.3.5），安全合规。 |
| 健康检查 | 完整覆盖 TCP/HTTP/HTTPS，健康/不健康阈值可配置，全不健康时返回 503（7.4 节）。 |

**高可用风险（严重）：**

1. **单点故障**：PRD 7.6 节明确承认"V1 阶段不自动迁移，需运维人工介入"，并计划"P2 迭代考虑 VRRP 方案"。**这是整个网络模块最大的风险点**——CLB 实例部署在单台宿主机，宿主机故障 = CLB 不可用 = 所有后端服务不可达。
2. **6 台 R740 场景下的影响**：如果 CLB 部署在 node-01，node-01 故障后所有依赖该 CLB 的业务中断，直到人工介入重建。对于生产环境不可接受。
3. **VIP 绑定方式不明**：PRD 提到"配置 VIP 绑定"但未说明 VIP 是如何绑定到 CLB 宿主机的。如果是 `ip addr add` 到 lo 或物理网卡，VIP 不具备漂移能力。需要明确：
   - VIP 是绑定到 loopback 还是物理接口？
   - 跨宿主机迁移时如何更新 ARP 表？
4. **缺失冗余方案**：即使是 V1 阶段，也应至少支持以下方案之一：
   - **方案 A（推荐 P0）**：主备模式 — 在不同宿主机部署 CLB 主备实例，通过 keepalived VRRP 实现 VIP 漂移
   - **方案 B（推荐 P1）**：多实例 + DNS 轮询 — 多个 CLB 实例绑定不同 VIP，通过内部 DNS 做负载均衡
   - **方案 C（推荐 P2）**：基于 OVS/OVN 的分布式 LB（类似 K8s kube-proxy IPVS 模式）

**建议补充（P0 优先级）：**

- 将 CLB 高可用方案从 P2 提升至 P0：至少实现主备 + keepalived VRRP
- 明确 VIP 绑定机制和漂移流程
- 增加"CLB 主备切换"验收条件

### 1.4 浮动 IP 设计：跨宿主机漂移的 ARP/GARP 方案在 Linux Bridge 环境下是否可行？

**结论：方案原理正确，但实现细节缺失严重，跨宿主机漂移在纯 Linux Bridge 环境下面临核心挑战。**

**原理分析：**

| 维度 | 评审结论 |
|------|---------|
| 设计对标 | 对标阿里云 EIP 的浮动 IP 设计理念正确（4.4 节），IP 与 VM 生命周期解耦。 |
| ARP/GARP 方案 | 4.4.5 节提到"发送 GARP 通知网络交换机更新 MAC 地址表"，这在二层网络（同一 VLAN/Bridge 域）下是标准做法。 |
| 绑定/解绑流程 | 4.4.2-4.4.3 节的 `ip addr add/del` + ARP 配置流程描述合理。 |

**核心问题：**

1. **跨宿主机漂移的核心矛盾**：浮动 IP 跨宿主机漂移的关键是二层可达性。在纯 Linux Bridge（无 VXLAN overlay）环境下，不同宿主机上的 VM 属于不同 Bridge 域，浮动 IP 从一个宿主机的 Bridge 迁移到另一个宿主机的 Bridge 后，**其他宿主机的 ARP 缓存不会自动更新**。GARP 只能更新同一二层域内的交换机 MAC 表，但无法解决跨 Bridge 的路由问题。
2. **物理交换机依赖**：PRD 假设"发送 GARP 通知网络交换机"，但这要求：
   - 所有宿主机连接到同一物理交换机
   - 虚拟网络使用桥接模式（Bridge 绑定物理网卡）
   - 交换机支持并正确响应 GARP
   如果使用 NAT 网络或隔离网络，GARP 无法跨越 Bridge 边界。
3. **漂移时间目标不现实**：PRD 设定"整个漂移过程目标在 10 秒内完成"，在 Linux Bridge 环境下，ARP 缓存老化时间通常为 30-300 秒，10 秒内完成二层路径切换需要额外手段（如主动清理所有相关宿主机的 ARP 缓存）。
4. **IP 冲突风险**：绑定期间使用 `ip addr add` 添加辅助 IP，如果同一 IP 被意外配置到两台 VM（agent 通信失败等异常场景），将导致 IP 冲突和网络不可用。PRD 的"强制解绑"场景（4.4.3）可能导致此问题。
5. **缺失实现细节**：
   - 浮动 IP 漂移时如何确保旧宿主机不再响应 ARP？（需要 agent 清理 ARP 条目）
   - 如何通知同网络内其他 VM 更新 ARP 缓存？（需要广播 ARP announcement）
   - NAT 网络下浮动 IP 漂移后，SNAT 规则如何更新？

**可行性评估矩阵：**

| 漂移场景 | 可行性 | 条件 |
|---------|--------|------|
| 同宿主机不同 VM | 可行 | 仅需本地 ARP 更新 |
| 跨宿主机 — 桥接网络 | 可行 | 依赖物理交换机 GARP 支持，需确认交换机型号 |
| 跨宿主机 — NAT 网络 | 不可行（V1） | 需要 overlay 网络（VXLAN）或集中式网关 |
| 跨宿主机 — 隔离网络 | 不可行（V1） | 二层隔离域无法跨宿主机 |

**建议：**

- 明确浮动 IP 的适用网络类型限制（P0 仅支持桥接网络）
- 增加跨宿主机漂移的前置条件校验
- 漂移流程中增加：旧宿主机 ARP 条目清理 → 新宿主机 IP 绑定 → GARP 广播（3次，间隔1s）→ 确认可达性
- 将漂移时间目标调整为 30 秒（更现实）
- 考虑引入分布式 ARP 代理（如 OVN 的 ARP responder）解决跨宿主机问题

### 1.5 与存量网络共存：不破坏现有 32 台 VM 的网络，方案是否足够安全？

**结论：PRD 声明了存量兼容原则，但缺乏具体的迁移策略和安全隔离验证机制。**

**安全性分析：**

| 维度 | 评审结论 |
|------|---------|
| 设计原则 | PRD 1.3 节明确"不破坏现有 32 台 KVM 虚拟机的网络配置，新网络与存量网络通过路由互通"。 |
| 命名规范 | TI-01 定义 Bridge 命名 `vn-{network_id前8位}`，可避免与现有 Bridge 冲突。 |
| CIDR 冲突检测 | 4.1.2 节实现 CIDR 重叠检测，防止新建网络与存量网络 IP 冲突。 |

**安全风险：**

1. **存量 VM 的安全组真空**：32 台存量 VM 没有安全组保护，在新平台上创建的新 VM 均有安全组。如果新老 VM 在同一路由域内，攻击者可能通过存量 VM 作为跳板攻击新 VM。需要为存量 VM 提供安全组绑定路径。
2. **存量网络拓扑未建模**：PRD 假设存量网络与新建网络"通过路由互通"，但未描述如何发现和建模存量网络的拓扑。如果存量 VM 使用默认的 virbr0 或自定义 Bridge，管理平台需要能够发现这些网络并展示。
3. **iptables 规则冲突**：存量环境可能已有手工配置的 iptables 规则（防火墙/NAT）。新平台下发安全组规则时可能与存量规则冲突或互相覆盖。PRD 7.10 节覆盖了 VM 迁移场景，但未覆盖存量规则冲突场景。
4. **DHCP 冲突**：存量 VM 可能依赖已有的 dnsmasq 实例。新平台的 dnsmasq 实例需确保监听在正确的 Bridge 接口上，不与存量 dnsmasq 冲突。
5. **路由表污染**：新平台在宿主机上配置 NAT 规则（SNAT/MASQUERADE），可能影响存量 VM 的出站路由行为。

**建议补充：**

- 增加"存量网络发现与导入"功能：扫描宿主机上现有 Bridge/iptables/dnsmasq，自动建模
- 为存量 VM 提供"导入并绑定默认安全组"的一键操作
- 增加 iptables 规则冲突检测（在 agent 下发前执行 `iptables -C` 检查）
- 定义新平台使用的 iptables chain 命名空间（如统一前缀 `NP-`），与存量规则隔离
- 增加存量兼容性验收条件

## 二、权限与审计模块评审

### 2.1 多租户隔离粒度：资源池/网络/数据三层隔离是否充分？

**结论：三层隔离设计充分，落地细节较为完整，但部分隔离边界存在隐患。**

**隔离层次分析：**

| 隔离层 | PRD 实现 | 评审结论 |
|--------|---------|---------|
| **资源池隔离** | Ceph 独立 RBD pool + RGW namespace（REQ-TENANT-001）、K8s 独立 Namespace | 充分。存储和计算在底层物理隔离，防止数据泄露和数据面攻击。 |
| **网络隔离** | 虚拟网络按租户归属、安全组按租户可见（PRD-03 8.1 节） | 充分。但需注意：网络模块 PRD 中安全组支持跨租户只读引用（3.1 节），可能成为信息泄露渠道——租户 A 可通过引用租户 B 的安全组来推断租户 B 的 VM 数量。 |
| **数据隔离** | API 中间件注入 tenant_id（REQ-TENANT-004）、审计日志独立存储 | 充分。但需补充数据库级别的隔离验证（如 PostgreSQL Row-Level Security）。 |

**隐患与建议：**

1. **安全组跨租户引用风险**：PRD-03 允许安全组规则引用其他租户的安全组作为流量来源。这虽然是合理的网络互通需求，但存在信息泄露风险——攻击者可通过创建引用规则来探测目标安全组是否存在（时序侧信道）。建议：
   - 引用前需目标租户管理员审批
   - 限制引用频率，防止枚举攻击
2. **数据库层面隔离**：PRD 描述的是应用层 tenant_id 过滤。建议增加 PostgreSQL Row-Level Security（RLS）策略作为纵深防御，防止应用层 bug 导致的跨租户数据泄露。
3. **配额隔离**：REQ-TENANT-002 的配额校验在中间件层，但未描述配额计算是实时查询还是缓存。如果使用缓存（如 PRD-03 提到的 Redis 权限缓存），可能存在配额检查的时间窗口漏洞——两个并发创建请求可能同时通过配额检查导致超额分配。建议配额校验使用数据库行级锁或 Redis 原子操作。

### 2.2 5 个预定义角色：权限边界是否合理？有没有遗漏？

**结论：5 个预定义角色边界清晰、符合最小权限原则，但缺少安全管理员独立角色。**

**角色设计评审：**

| 角色 | 权限范围 | 评审结论 |
|------|---------|---------|
| 平台管理员 (platform_admin) | 全局所有权限 | 合理。需确保至少有 1 个活跃平台管理员（PRD 7.11 节已覆盖）。 |
| 租户管理员 (tenant_admin) | 本租户全部权限 + 用户/角色管理 | 合理。权限继承规则（8.2 节）清晰。 |
| 租户运维 (tenant_operator) | 本租户资源管理 | 合理。与开发者权限边界清晰（运维可管理全部资源，开发者仅管理自己创建的）。 |
| 租户开发者 (tenant_developer) | 本租户查看 + 自助创建 | 合理。"仅能删除自己创建的资源"（8.4 节）是正确的设计。 |
| 只读用户 (tenant_viewer) | 仅查看 | 合理。 |

**遗漏角色：**

1. **安全管理员 (security_admin)**：PRD-03 的权限矩阵（8.1 节）中存在"安全管理员"角色——"可查看所有租户的安全组和规则，可批量封禁 IP，但不可创建/删除网络和 CLB"。但在 PRD-04 的预定义角色列表（REQ-ROLE-001）中，安全管理员**并未列为预定义角色**。两个 PRD 存在不一致。
   - 如果安全管理员是独立角色，需要加入预定义角色列表并定义完整权限矩阵
   - 如果安全管理员是平台管理员的职责子集，需要在平台管理员权限中明确安全操作边界
2. **审计员 (auditor)**：PRD-04 的 US-05 提到"审计员"角色，但在预定义角色和权限矩阵中均未出现。审计员应独立于平台管理员，仅具有审计日志查看权限，无任何资源操作权限。这是等保三级的重要要求（三权分立：系统管理员、安全管理员、审计管理员）。
3. **网络管理员 (network_admin)**：如果网络运维需要独立角色（仅管理网络/安全组/CLB，不管理 VM 和存储），当前角色体系无法满足。建议作为可扩展的自定义角色，暂不纳入预定义。

**建议：**

- 将"安全管理员"和"审计员"纳入预定义角色列表，实现三权分立
- 统一 PRD-03 和 PRD-04 中的角色定义
- 在权限矩阵中为安全管理员和审计员补充完整行

### 2.3 审计日志方案：独立存储、不可篡改的方案是否可行？

**结论：方案设计总体可行且安全，但 ClickHouse/ES 选型需澄清，append-only 的实现细节需补充。**

**安全性分析：**

| 维度 | 评审结论 |
|------|---------|
| 存储分离 | REQ-AUDIT-002 要求审计日志与业务数据库物理分离、独立账号、仅审计模块可写。符合等保三级要求。 |
| 不可篡改 | 仅支持 append-only，不提供修改/删除 API。设计正确。 |
| 数据完整性 | 使用 UUID v7（按时间排序），便于顺序校验和完整性检查。 |
| 保留策略 | 180 天在线 + 归档到 Ceph RGW + 3 年保留（7.4 节）。合规且可落地。 |

**实现风险：**

1. **ClickHouse vs Elasticsearch 选型未定**：PRD 同时提到 ClickHouse 和 Elasticsearch。两者架构差异显著：
   - ClickHouse：列存、压缩率高、时序查询快，但更新/删除能力弱（恰好符合 append-only 需求）
   - Elasticsearch：全文搜索强、支持复杂聚合，但存储成本高、运维复杂
   - **推荐 ClickHouse**：更符合审计日志"写多读少、时序查询"的特征，且 6 台 R740 的资源有限，ClickHouse 的压缩率更有优势
2. **append-only 的完整性校验**：PRD 未提及如何防止审计日志被绕过直接操作数据库。建议：
   - 在 ClickHouse 表上设置 `ENGINE = MergeTree()` + 禁止 DELETE/UPDATE 的权限
   - 使用区块链式哈希链（每条日志包含前一条日志的哈希），实现防篡改验证
   - 定期导出日志快照并签名，存入离线存储
3. **审计日志写入的可靠性**：审计日志写入失败不应阻塞业务操作。PRD 需明确写入策略：
   - 同步写入（阻塞业务，但保证完整性）vs 异步写入（不阻塞，但可能丢失）
   - 写入失败时的重试和降级策略
   - **推荐方���**：异步写入 + 本地缓冲队列 + 失败告警
4. **敏感信息脱敏**：审计日志中 request_params 包含"脱敏后"的请求参数（REQ-AUDIT-001），但未定义脱敏规则。密码、Token、私钥等敏感字段必须在写入审计日志前脱敏。建议明确脱敏字段清单和规则。

**建议补充：**

- 明确审计日志存储选型为 ClickHouse（推荐）
- 增加审计日志完整性校验机制（哈希链或 Merkle Tree）
- 明确脱敏字段清单（密码/Token/私钥/手机号/邮箱）
- 定义审计日志写入可靠性策略（异步 + 缓冲 + 告警）

### 2.4 LDAP 预留接口：设计是否合理？

**结论：接口预留思路正确，降级方案务实，但集成细节有改进空间。**

**合理性分析：**

| 维度 | 评审结论 |
|------|---------|
| 降级策略 | 7.10 节设计了 LDAP 不可用时的本地认证降级，并明确 5 秒超时和自动重连。设计务实。 |
| 用户导入 | REQ-USER 功能列表中包含"LDAP/AD 导入用户，同步组织架构"（P1），方向正确。 |
| 密码策略兼容 | 本地密码策略（90 天过期、复杂度校验）与 LDAP 域策略可能存在冲突，需定义优先级。 |

**风险与建议：**

1. **LDAP 用户的 tenant 映射**：LDAP 用户导入时，如何映射到平台租户？是通过 LDAP 组织单元（OU）映射还是手动指定？PRD 未说明。建议：
   - 支持基于 LDAP 属性（如 `department` 或 `memberOf`）自动映射到租户
   - 提供映射规则配置界面
2. **密码策略冲突**：LDAP 域控制器通常有自己的密码策略。当平台本地策略（90 天过期）与 AD 策略不一致时，以谁为准？建议明确：LDAP 用户密码策略以域控制器为准，本地策略仅用于纯本地用户。
3. **LDAP 同步周期**：未说明用户和组的同步频率（实时/定时）。建议：
   - 实时同步：用户登录时验证 LDAP 凭据并更新本地信息
   - 定时全量同步：每日凌晨执行，同步新增/删除用户和组织架构变更
4. **纯 LDAP 用户的 fallback**：7.10 节正确指出"纯 LDAP 用户（无本地密码），LDAP 不可用时无法登录"。建议为 LDAP 用户可选配置本地密码哈希同步（用户首次 LDAP 登录成功后缓存密码哈希），提升高可用性。
5. **LDAPS/TLS**：未提及 LDAP 通信是否加密。**必须使用 LDAPS（636 端口）或 StartTLS**，明文 LDAP 不可接受。

**建议补充：**

- 明确 LDAP 用户到租户的映射规则
- 明确 LDAP 密码策略与本地策略的优先级
- 补充 LDAPS/TLS 要求
- 增加 LDAP 用户本地密码哈希缓存选项

### 2.5 会话安全和登录策略：是否足够？

**结论：基本安全措施到位，但 JWT 设计存在风险，MFA 缺失是明显短板。**

**安全措施评审：**

| 安全措施 | PRD 实现 | 评审结论 |
|---------|---------|---------|
| 密码策略 | 8位+三类字符+90天过期+历史5次（REQ-USER-003） | 符合基线要求。建议增加密码黑名单检测（如 Have I Been Pwned API）。 |
| 登录锁定 | 5次失败/30分钟（REQ-USER-004） | 合理。建议增加递增锁定时间（第1次锁定5分钟，第2次15分钟，第3次30分钟）。 |
| 会话超时 | 30分钟无操作 + 前端5分钟续期提醒（REQ-USER-006） | 合理。 |
| 异地登录检测 | IP 归属地城市变化 + 邮件告警（REQ-SEC-001） | 务实。纯内网环境（10.x IP）的异地检测意义有限，但对 VPN/远程接入场景有价值。 |
| JWT Token | RS256 签名 + 2小时有效期（10.7 节） | 存在风险，见下文。 |

**JWT 安全风险（重要）：**

1. **无 Token 刷新机制细节**：PRD 附录 B 列出了 `/api/v1/auth/refresh` 接口，但未说明 refresh token 的存储和失效策略。refresh token 通常需要更长的有效期和更强的保护（httpOnly cookie 或 secure storage）。
2. **无 Token 吊销机制**：JWT 是无状态的，2 小时有效期内无法主动吊销。如果管理员禁用用户或修改角色，用户的旧 Token 仍然有效（直到过期）。PRD 虽然提到"禁用用户后已登录会话在下次请求时失效"（10.1 节），但这与无状态 JWT 的设计矛盾——除非每次请求都查数据库验证用户状态。
3. **建议**：
   - 引入 Token 黑名单（Redis 缓存被吊销的 Token ID，TTL = Token 剩余有效期）
   - 或使用有状态会话（Session ID + Redis），彻底解决吊销问题
   - 如果坚持无状态 JWT，将有效期缩短至 15 分钟 + refresh token 1 小时，降低吊销窗口
4. **RS256 vs HS256**：PRD 选择 RS256（非对称签名），这是正确的选择（便于微服务间验证，私钥集中管理）。但需确保私钥安全存储（如 Vault 或加密配置文件）。

**缺失的安全措施：**

1. **MFA 多因素认证**：PRD 将 MFA 标记为 P2，但考虑到平台管理员和租户管理员的高权限，**建议将 MFA 提升至 P1**（至少对管理员角色强制启用）。
2. **IP 白名单**：PRD 将 IP 白名单标记为 P2。考虑到内网运维平台的敏感性，建议 P1 支持租户级别的 IP 访问控制（如仅允许公司内网 IP 段访问）。
3. **操作频率限制**：PRD 未提及 API 速率限制。缺少速率限制可能导致暴力破解（虽然已有登录锁定）和资源滥用。建议在 API Gateway 层增加速率限制。
4. **会话并发控制**：未提及是否允许同一用户多处同时登录。建议支持配置：允许/禁止并发会话，默认允许但限制数量（如最多 3 个设备）。

**建议补充：**

- 明确 JWT 吊销机制（Token 黑名单或缩短有效期）
- 将 MFA 从 P2 提升至 P1（至少对管理员角色）
- 增加 API 速率限制（全局 + 用户级别）
- 增加会话并发控制配置
