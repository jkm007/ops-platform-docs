# 运维内网管理系统 — 存储模块 PRD 评审报告

> **评审角色**：Ceph 分布式存储专家  
> **评审日期**：2026-07-14  
> **评审对象**：`PRD_02_存储模块.md` v1.0  
> **实际环境**：Ceph Reef 18.2.5，6 节点，HEALTH_OK

---

## 评审总览

PRD 整体质量较高，结构完整，EARS 格式规范，边界场景考虑充分。但存在一个**关键定位偏差**——块存储模块的假设与实际情况严重不符，需要重大调整。以下逐条分析。

---

## 1. 块存储定位

### 现状分析

| 维度 | PRD 假设 | 实际情况 |
|------|---------|---------|
| VM 磁盘 | 基于 Ceph RBD | 本地 qcow2 文件（`/home/kvm/images/`） |
| 块存储驱动 | `rbd map` + libvirt attach | `virsh attach-disk` 使用本地路径 |
| 云盘定位 | 对标阿里云 EBS | 实际上 KVM 从未使用 Ceph RBD |

### 评审结论

**PRD 中块存储模块的定位需要做重大调整，当前方案与现状完全脱节。**

#### 问题拆解

1. **块存储模块存在的基础被架空**

   PRD 将块存储定位为"虚拟机持久化高性能块设备"，对标阿里云 EBS，并设计了完整的"创建云盘 → 挂载 VM → 扩容 → 通知 OS 识别"流程（REQ-BLK-001 ~ REQ-BLK-010）。但实际情况是：**所有 KVM 虚拟机磁盘使用本地 qcow2 文件**，Ceph RBD 从未被用作 VM 主存储。

   这意味着 PRD 中块存储模块的全部用户故事（US-03 "为虚拟机挂载云盘"）和 12 个功能点（BLK-01 ~ BLK-12）在当前的用户场景中**无对应需求**。

2. **仅有 `kvm-backup` pool 含 251 GiB RBD 数据**

   集群中唯一的 RBD pool 是 `kvm-backup`，251 GiB。其用途需要进一步确认（见第 2 条），但无论如何，这个量级远不足以支撑一个"云盘全生命周期管理"模块。

3. **存储池分层的前提不存在**

   PRD 在 REQ-BLK-003 中设计了"存储池选择"功能（ssd-pool / hdd-pool / archive-pool 三级分层），但实际集群**只有 8 块 HDD OSD**，不存在 SSD Pool 或归档 Pool。这个交互设计完全是空想。

4. **`virsh blockresize` 流程不适用于 qcow2**

   PRD 中扩容流程依赖 `rbd resize` + `virsh blockresize` / QEMU Guest Agent（REQ-BLK-007），这是 RBD 场景的流程。对于本地 qcow2 文件，扩容流程完全不同（`qemu-img resize` + VM 内部 resize2fs/xfs_growfs），需要重新设计。

### 调整建议

#### 方案 A：重新定位块存储模块（推荐）

将块存储模块从"Ceph RBD 云盘管理"调整为 **"虚拟机存储管理"**：

- **P0 核心功能**：展示 VM 的 qcow2 磁盘列表（路径、大小、实际占用、格式）、磁盘使用率监控、空间告警
- **P1 扩展功能**：qcow2 磁盘在线扩容（`qemu-img resize` + Guest Agent）、快照管理（qcow2 snapshot，非 RBD snapshot）
- **P2 远期规划**：Ceph RBD 作为 VM 存储后端的迁移方案（需与虚拟化团队协商）

#### 方案 B：明确 Ceph RBD 当前仅用于备份

如果团队确认 `kvm-backup` pool 仅用于备份场景，则：

- 将块存储模块降级为 **"备份存储管理"** 子模块
- 删除"挂载到虚拟机"、"扩容通知 OS"等流程
- 保留：RBD Image 列表查看、容量监控、快照管理（用于备份版本管理）

#### 方案 A vs B 对比

| 维度 | 方案 A（VM 存储管理） | 方案 B（备份存储管理） |
|------|----------------------|----------------------|
| 业务价值 | 高（解决真实痛点） | 中（功能单一） |
| 开发成本 | 中等（需适配 qcow2） | 低（大幅裁剪 PRD） |
| 远期扩展性 | 好（可平滑迁移到 RBD） | 差（后续需重构） |
| PRD 改动量 | 大（块存储全量重写） | 中（删除 VM 相关流程） |

**建议采用方案 A**，因为"虚拟机磁盘空间不足"才是运维日常的真实痛点，Ceph RBD 的云盘管理在无 VM 使用场景下是伪需求。

## 2. kvm-backup Pool

### 现状分析

- Pool 名称：`kvm-backup`
- 数据量：251 GiB
- 类型：副本池（3 副本）
- 关联：RBD（含 RBD Image）

### 评审结论

**Pool 命名表明其用途是 KVM 虚拟机备份，PRD 需要根据实际用途调整备份策略设计。**

#### 关键判断

1. **251 GiB 的数据量**说明已有一定规模的备份数据在此 pool 中。需要确认：
   - 备份方式：是 RBD Image 直接写入备份数据，还是 VM 通过某种备份工具将数据写入此 pool？
   - 备份频率和保留策略是什么？
   - 是否有自动清理机制？

2. **PRD 中未涉及备份策略设计**

   PRD 仅在基础设施概况表格中提到了 `kvm-backup` pool（1.3 节），但在功能清单和详细需求中完全没有针对此 pool 的专门设计。这是一个明显的遗漏。

3. **PRD 中块存储模块的快照功能（RBD snap）≠ 备份**

   PRD 设计的"云盘快照"（REQ-BLK-008 ~ REQ-BLK-010）是 RBD 快照，属于存储层的 Crash-Consistent Snapshot，与 `kvm-backup` pool 中可能存在的应用级备份是不同层面的东西。

### 调整建议

1. **明确 `kvm-backup` pool 的实际用途**

   需要运维团队确认：
   - 251 GiB 数据的具体内容（备份工具？手动导出的 qcow2 文件？）
   - 备份策略（增量/全量、保留周期）
   - 是否期望通过平台统一管理备份策略

2. **在 PRD 中新增"备份管理"子模块**

   至少包含以下功能：
   - 备份任务列表查看（来源 VM、备份时间、大小）
   - 备份空间使用趋势
   - 备份保留策略配置（如保留最近 N 份备份）
   - 备份过期自动清理
   - 备份恢复操作（恢复到指定 VM 或导出下载）

3. **将 PRD 中块存储的"快照"功能与备份功能区分**

   - **RBD 快照**：存储层的即时快照，用于快速回滚（保留在块存储模块）
   - **备份管理**：应用层的数据备份，涉及 `kvm-backup` pool（新建模块）

## 3. 对象存储方案

### 现状分析

- RGW 已启用，825 GiB 数据
- 4 个 RGW 用户：`manager-images`、`dashboard`、`inner-img`、`ark-video-backup`
- S3-compatible API 已可用

### 评审结论

**对象存储方案整体合理可行，PRD 覆盖了 RGW 管理的核心场景。但存在以下 5 个需要关注的问题。**

#### 问题 1：ACL 设计过于简化（中风险）

PRD 的 ACL 设计（REQ-OBJ-015）仅支持三个级别：私有、公共读、公共读写。这是**Bucket-level ACL**，粒度太粗。

实际 RGW 支持：
- Bucket Policy（JSON 策略，类似 AWS IAM Policy，可按用户/IP/操作精细化控制）
- IAM User Policy
- 对象级 ACL

**建议**：
- P0 保留三个预设级别（快速配置入口）
- P1 增加"高级策略编辑器"（直接编辑 Bucket Policy JSON），满足精细化权限需求
- 公共读写选项增加红色高危警告（PRD 已有提及，需强化）

#### 问题 2：RGW User 与平台用户的映射关系未定义（高风险）

PRD 中"创建 Bucket 后自动生成 AccessKey/SecretKey"（REQ-OBJ-005）暗示：**每个 Bucket 对应一个 RGW User**。

这与 RGW 的实际模型不一致：
- RGW User 是顶层实体，一个 User 可以拥有多个 Bucket
- 现有 4 个 RGW User 已经存在，每个 User 下可能有多个 Bucket
- 如果每个 Bucket 创建一个 RGW User，会导致 User 数量膨胀，管理混乱

**建议**：
- 明确 RGW User 的创建策略：每个**项目**对应一个 RGW User（而非每个 Bucket）
- 在该 User 下创建多个 Bucket，通过 Bucket Policy 实现项目内隔离
- 或者：每个**应用**对应一个 RGW User（更细粒度）
- 文档中补充 RGW User 与平台项目的映射关系图

#### 问题 3：用量统计的数据来源不明确（低风险）

PRD 要求展示"最近 24 小时请求次数趋势（GET/PUT/DELETE 分类）"（REQ-OBJ-007），但 RGW 原生不直接提供按 Bucket 维度的请求分类统计。

**建议**：
- 使用 RGW Usage Log（`radosgw-admin usage show`）获取用量数据
- 或启用 RGW 的 Bucket Logging 功能，将访问日志写入专门的 log bucket
- 或通过 Ceph exporter + Prometheus 采集 RGW 指标
- 在 PRD 中标注数据采集方案选型

#### 问题 4：文件上传大小限制过于乐观（中风险）

PRD 设定单文件最大 100 GiB（边界场景 1），但这是 RGW 配置的理论上限。在 8 块 HDD、混部环境的实际场景下：
- 100 GiB 文件通过 Web 上传需要数小时，期间网络波动、浏览器崩溃风险高
- 分片上传（10 MiB 分片）意味着 100 GiB 需要 10240 个分片，管理开销大

**建议**：
- Web 界面上传限制为 5 GiB（行业常见上限）
- 超过 5 GiB 的文件引导用户使用 S3 CLI/SDK 工具上传
- 或者 P1 实现"断点续传 + 分片上传"作为 Web 大文件上传方案

#### 问题 5：文件预览的技术边界不清晰（低风险）

PRD 要求"支持常见格式在线预览（图片、文本、PDF、视频）"（OBJ-12，P1），但没有定义：
- 预览是通过生成预签名 URL 直接在浏览器打开，还是后端转码后展示？
- 视频预览是否需要转码？（850 MiB 的 MP4 文件直接加载不现实）
- PDF 预览是否需要后端渲染？

**建议**：
- 图片/文本：生成预签名 URL，浏览器直接渲染（零成本）
- PDF：可集成 pdf.js 或后端转图片
- 视频：P1 仅支持小视频（< 50 MiB）的 HTML5 播放，大视频提供下载链接；P2 可考虑集成转码服务

## 4. Ceph 集群管理

### 现状分析

- 6 节点、5 MON + 3 MGR、8 OSD
- 单集群，规模不大
- Ceph Dashboard 已内置（Reef 版本自带）

### 评审结论

**自研 Dashboard 有其价值，但需明确与 Ceph 原生 Dashboard 的分工和权限边界。**

#### 4.1 Dashboard 嵌入 vs 自研

| 维度 | Ceph 原生 Dashboard | 自研 Dashboard |
|------|-------------------|----------------|
| 集群健康 | 原生支持，实时 | 需自建数据采集 |
| 性能图表 | 内置 Prometheus + Grafana | 需自建时序存储和图表 |
| 告警 | 内置 Alertmanager | 需自建告警引擎 |
| 用户体系 | 简单的 local user | 可与企业 IAM 集成 |
| 多集群 | 不支持 | PRD 设计支持（P2） |
| 操作审计 | 无 | PRD 设计了完整审计 |
| 审批流程 | 无 | PRD 设计了高危操作审批 |
| 中文界面 | 部分支持 | 完全自定义 |

**建议：分层策略**

| 层级 | 功能 | 方案 |
|------|------|------|
| **展示层** | 集群健康、容量、OSD/Pool 状态 | 自研 Dashboard（与企业 UI 统一） |
| **数据层** | IOPS、带宽、延迟等性能指标 | 复用 Ceph Prometheus Exporter + Prometheus + Grafana（嵌入 iframe），自研负责告警规则 |
| **操作层** | OSD out/in、Pool 创建/删除 | 自研封装 + 审批流程 |
| **管理入口** | 高级排障 | 提供"Ceph 原生 Dashboard"链接（SSO 跳转），保留原生工具作为兜底 |

**核心理由**：8 块 OSD 的单集群，自研全套性能监控（时序数据库 + 图表渲染 + 数据采集）的 ROI 很低。直接嵌入 Grafana iframe 是更务实的方案，将开发资源投入到审批流程、审计日志、IAM 集成等差异化能力上。

#### 4.2 权限边界评估

PRD 的权限矩阵（8.2 节）设计合理，但有以下建议：

| 操作 | PRD 设计 | 建议调整 | 理由 |
|------|---------|---------|------|
| 删除 OSD | 审批后执行 | **保持** | 正确，OSD 删除不可逆 |
| 删除 Pool | 审批后执行 | **保持** | 正确，需安全校验 + 关联资源检查 |
| OSD out/in | 存储管理员直接执行 | **增加操作窗口限制** | out 操作会触发数据重平衡，建议限制在业务低峰期（如凌晨 2-6 点）执行 |
| 修改 PG 数 | 仅允许增加 | **增加变更窗口限制** | PG 调整会触发大量数据迁移，同样需要低峰期窗口 |
| 创建 Pool | 存储管理员直接执行 | **保持** | 创建操作对集群影响较小 |
| MON 节点管理 | 仅查看 | **保持** | MON 操作应在原生环境执行 |

#### 4.3 OSD 故障处理设计评估

REQ-CLS-006（OSD 磁盘故障处理）的"一键隔离"按钮设计是亮点，但需注意：

- **"OSD Down > 5 分钟 → 高亮闪烁 + 一键隔离"** 是合理的。但需考虑：短暂网络抖动也可能导致 OSD Down 5 分钟，不一定代表磁盘故障
- **建议增加**：在"一键隔离"前展示 OSD 的 SMART 信息（如果可获取），帮助判断是否为硬件故障
- **自动 out 风险**：不建议自动执行 out 操作，因为 out 会触发数据重平衡，在错误判断的情况下代价很高。保持"手动确认"是正确的设计

## 5. 性能影响

### 现状分析

- 6 节点同时运行 KVM 虚拟机（qcow2）和 Ceph（OSD/MON/MGR/RGW）
- 8 块 HDD OSD，29 TiB 总容量
- Ceph 使用率仅 10.89%，负载较轻

### 评审结论

**当前负载下影响可控，但需在 PRD 中补充混部环境的操作约束。**

#### 5.1 各操作的影响评估

| 操作 | 对 KVM 影响 | 对 Ceph 影响 | 风险等级 |
|------|-----------|------------|---------|
| **Dashboard 查看** | 几乎为零 | 几乎为零（`ceph status` 是轻量操作） | 无风险 |
| **OSD out/in** | 中等（backfill/recovery 占用磁盘 IO 和网络） | 高（数据迁移期间性能下降） | **中风险** |
| **调整 PG 数** | 中等（同上） | 高（大量数据迁移） | **中风险** |
| **创建 Pool** | 几乎为零 | 低 | 低风险 |
| **RBD 快照** | 几乎为零 | 低（元数据操作） | 低风险 |
| **RGW 大文件上传** | 中等（占用网络带宽） | 中（占用 OSD IO） | 低风险 |
| **性能数据采集** | 几乎为零 | 低（60 秒间隔合理） | 无风险 |

#### 5.2 混部环境的关键风险

1. **OSD 操作触发 backfill/recovery 时**，HDD 的 IO 会被数据迁移大量占用，可能影响：
   - KVM 虚拟机的 qcow2 磁盘 IO（如果 qcow2 文件在同一个物理磁盘上）
   - RGW 的对象读写响应延迟

2. **磁盘带宽竞争**：8 块 HDD 在 backfill 期间，每个 OSD 的 IO 能力会显著下降

#### 5.3 调整建议

1. **PRD 中增加"混部环境操作约束"章节**

   ```
   - OSD out/in 操作限制在每日 02:00-06:00 窗口执行
   - PG 调整操作限制在周末低峰期执行
   - 执行数据迁移操作前，系统自动检测当前 IOPS 和带宽使用率，
     如果超过正常水平的 50%，延迟执行
   - 创建大量 RBD Image / 上传大量对象时，增加并发限流
   ```

2. **增加 backfill/recovery 的限速控制**

   在 Pool 配置中建议设置：
   ```
   osd_max_backfills = 1
   osd_recovery_max_active = 1
   osd_recovery_op_priority = 1
   ```
   在混部环境下，这些参数可以防止数据迁移"饿死"正常业务 IO。PRD 中可以增加"集群调优建议"章节。

3. **监控混部影响**

   在 Dashboard 中增加"节点级别的 CPU/内存/磁盘 IO/网络"指标（当前 PRD 只关注 Ceph 集群级别的指标），帮助发现混部资源竞争。

## 6. 存量 RGW 用户

### 现状分析

| RGW User | 推测用途 | 数据量 |
|----------|---------|--------|
| `manager-images` | 管理后台图片存储 | 未知 |
| `dashboard` | Dashboard 相关资源 | 未知 |
| `inner-img` | 内部图片服务 | 未知 |
| `ark-video-backup` | 方舟视频备份 | 未知 |

4 个用户共 825 GiB 数据。

### 评审结论

**存量 RGW 用户需要"纳管"而非"迁移"，但 PRD 的存量纳管流程（5.3 节）对 RGW 用户的处理过于笼统。**

#### 6.1 关键问题

1. **RGW User 与 Bucket 的关系需要扫描清楚**

   存量纳管流程（5.3 节 Step 3）提到"扫描 RGW Bucket（关联用户、用量）"，但没有说明如何建立 Bucket 与 User 的关联关系。需要补充：
   - 扫描每个 RGW User 下有哪些 Bucket
   - 扫描每个 Bucket 的 ACL / Policy
   - 将存量 Bucket 归属到对应的 RGW User

2. **AccessKey/SecretKey 已在外部使用**

   4 个现有 RGW User 的 AccessKey/SecretKey 可能已经被业务系统硬编码使用。如果平台"吊销"或"重新生成"这些凭证，会导致业务中断。

   **PRD 中的凭证管理（REQ-OBJ-016）需要明确**：
   - 存量 User 的凭证**仅查看，不可吊销**（或吊销前需确认业务影响）
   - 吊销操作增加"影响范围分析"：展示使用该凭证的 Bucket 列表和最近的访问记录

3. **纳管后的权限归属**

   存量 RGW User 对应的 Bucket 需要分配归属项目：
   - `ark-video-backup` → 方舟项目
   - `inner-img` → 内部图片服务项目
   - `manager-images` → 管理后台项目
   - `dashboard` → 运维项目

   分配后，项目成员才能看到对应 Bucket（符合 PRD 的项目隔离设计）。

#### 6.2 调整建议

1. **在存量纳管流程中增加 RGW User 专项步骤**

   ```
   Step 3.5: RGW 用户专项扫描
   - 列出所有 RGW User（radosgw-admin user list）
   - 获取每个 User 的详细信息（radosgw-admin user info --uid=xxx）
   - 建立 User → Bucket 的关联映射
   - 记录每个 User 的 AccessKey 列表（不暴露 SecretKey）
   - 标记 User 类型：存量纳管 / 平台创建
   ```

2. **增加"RGW 用户管理"页面**

   当前 PRD 只有 Bucket 列表，没有 RGW User 列表。建议增加：
   - RGW User 列表（用户 ID、显示名称、所属项目、Bucket 数量、用量汇总、创建时间）
   - 查看 User 详情（关联 Bucket、AccessKey 列表（脱敏）、配额、权限策略）
   - 创建平台 RGW User（自动关联到项目）

3. **迁移策略**

   - **不需要迁移数据**：存量 Bucket 中的 825 GiB 数据保持不动
   - **不需要迁移凭证**：存量 User 的 AK/SK 保持不变，业务不受影响
   - **需要纳管元数据**：将 User-Bucket 关联关系导入平台数据库，建立归属关系
   - **新增 Bucket 使用平台创建**：平台新建 Bucket 时创建新的 RGW User（或复用项目级 RGW User）

## 7. 容量规划

### 现状分析

| 指标 | 当前值 |
|------|--------|
| 总容量 | 29 TiB（8 × 3.6 TiB HDD） |
| 已用量 | 3.2 TiB（10.89%） |
| 剩余容量 | 25.8 TiB |
| 增长趋势 | 未知（需确认历史增长速率） |
| OSD 扩容能力 | 6 节点可继续增加 OSD |

### 评审结论

**当前容量充裕，但 PRD 的告警阈值设置需要调整，容量规划需要补充。**

#### 7.1 告警阈值分析

PRD 设计的容量告警阈值（REQ-CLS-003 / REQ-CLS-014）：

| 阈值 | PRD 行为 | 评估 | 建议 |
|------|---------|------|------|
| 80% | 容量环图变黄色 | **偏高** | 建议降为 **70%** 黄色预警（HDD 扩容周期长，需提前规划） |
| 90% | 容量环图变橙色 | **合理** | 保持，触发邮件通知 |
| 95% | 环图变红色 + 持久化横幅 | **合理** | 保持，触发紧急告警 + 限制写操作 |

**调整理由**：
- HDD 集群扩容涉及硬件采购、上架、OSD 部署、数据重平衡，全流程可能需要数周
- 70% 预警给运维团队留出充足的采购和部署时间
- Ceph 在容量超过 85% 时可能出现性能下降（nearfull ratio 默认为 85%）

#### 7.2 扩容路径评估

| 扩容方式 | 可行性 | 影响 | 建议 |
|---------|--------|------|------|
| 增加 HDD OSD | **可行** | 数据重平衡（backfill），持续数天 | 首选方案，6 节点均有空余盘位 |
| 替换大容量 HDD | 可行但风险高 | 逐个 OSD 替换，每块需完整重平衡 | 备选方案 |
| 增加节点 | 可行 | 需部署 MON/OSD/MGR，周期较长 | 长期方案 |
| 增加 SSD OSD（分层） | 可行 | 需创建 SSD Pool + CRUSH rule | P2 规划，可显著提升 RGW 小文件性能 |

#### 7.3 容量规划补充建议

PRD 中缺少容量规划功能，建议增加：

1. **容量趋势预测**

   基于历史用量数据（90 天），展示：
   - 过去 90 天用量增长趋势
   - 预计容量耗尽可能日期（基于线性/指数回归）
   - 按 Pool 维度的增长趋势分解

2. **OSD 扩容模拟**

   - 输入"新增 N 块 HDD（每块 X TiB）"，系统估算：
     - 扩容后总容量
     - 预计数据重平衡时长
     - 扩容后各 OSD 的使用率分布

3. **容量报表**

   - 按项目维度展示对象存储/块存储的用量趋势
   - 支持导出月度/季度容量报告

4. **Pool 级别的容量告警**

   当前 PRD 只有集群级别的容量告警（80%/90%/95%），建议增加：
   - 单个 Pool 使用率超过 70% → 警告
   - 单个 Pool 使用率超过 85% → 严重告警

## 附录：PRD 亮点

尽管存在上述需要调整的问题，PRD 在以下方面做得很好：

1. **EARS 格式规范**：需求描述遵循 Event-driven / State-driven / Ubiquitous / Unwanted 四类模式，可测试性强
2. **边界场景覆盖全面**：13 个边界场景覆盖了上传、删除、挂载、扩容、故障、并发等典型异常情况
3. **存量纳管设计合理**：只读纳管 → 手动分配 → 持续同步的三阶段流程设计成熟
4. **高危操作防护充分**：删除 OSD/Pool 需要审批流程、多重确认、安全校验
5. **权限矩阵清晰**：5 种角色、17 个功能模块的权限定义完整
6. **HEALTH_WARN/ERR 的自动化处理**：自动拦截高危操作、调整刷新频率等设计是生产级的考虑
