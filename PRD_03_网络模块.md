## 一、模块概述

### 1.1 模块定位

网络模块是运维内网统一管理平台的核心基础设施模块，负责为虚拟机、容器及其他云资源提供可软件定义的网络能力。模块对标阿里云 VPC / 安全组 / CLB 控制台体验，底层基于现有 Linux Bridge + iptables/nftables + Nginx/HAProxy 架构实现，确保与存量网络环境平滑共存。

### 1.2 模块组成

```
网络模块
├── 子模块 1：虚拟网络（Virtual Network）
│   ├── 网络管理（列表/创建/编辑/删除）
│   ├── 子网管理（CIDR/网关/DHCP）
│   └── 网络拓扑可视化
├── 子模块 2：安全组（Security Group）
│   ├── 安全组管理（列表/创建/编辑/删除/克隆）
│   ├── 规则管理（入方向/出方向）
│   └── 实例绑定/解绑
└── 子模块 3：负载均衡（CLB）
    ├── 实例管理（列表/创建/编辑/删除）
    ├── 监听器管理（HTTP/HTTPS/TCP/UDP）
    ├── 后端服务器管理
    ├── 健康检查
    └── SSL 证书管理
├── 子模块 4：浮动 IP（Float IP）
    ├── 浮动 IP 管理（列表/申请/释放）
    ├── 绑定/解绑虚拟机
    └── 带宽限速
```

### 1.3 核心设计原则

1. **存量兼容**：不破坏现有 32 台 KVM 虚拟机的网络配置，新网络与存量网络通过路由互通
2. **租户隔离**：网络资源按租户隔离，安全组可跨租户共享（只读引用）
3. **声明式管理**：用户声明期望状态，系统负责状态收敛
4. **渐进式复杂度**：P0 功能覆盖 80% 日常场景，P1/P2 逐步迭代

### 1.4 术语定义

| 术语 | 说明 |
|------|------|
| **虚拟网络** | 一个二层隔离域，对标 VPC。每个虚拟网络拥有独立的路由表和 DHCP 服务 |
| **子网** | 虚拟网络内的 CIDR 划分，承载虚拟机网卡 |
| **NAT 网络** | 虚拟机通过宿主机 NAT 访问外部，外部无法直接访问虚拟机 |
| **隔离网络** | 纯二层隔离，不与外部互通，适用于内部测试环境 |
| **桥接网络** | 虚拟机直接桥接到物理网卡，与物理网络同级 |
| **VLAN** | 虚拟局域网，通过在物理网络上打标签实现二层隔离，VLAN ID 范围 1-4094 |
| **VLAN Trunk** | 物理交换机端口允许多个 VLAN 的标记帧通过，实现跨宿主机二层互通 |
| **安全组** | 虚拟状态防火墙，作用于网卡级别 |
| **CLB 实例** | 负载均衡实例，包含公网/内网两种类型 |
| **监听器** | CLB 上的流量入口，监听特定协议+端口 |
| **后端服务器** | 被 CLB 代理的真实服务器（虚拟机 IP + 端口） |

## 二、用户故事

### US-01：运维工程师创建隔离测试网络

> 作为**运维工程师**，我希望**创建一个隔离网络用于内部渗透测试**，以便**测试流量不会泄露到生产网络，且测试虚拟机之间可以互通**。

**验收条件**：
- 创建隔离网络 `pentest-net`，指定 CIDR `10.100.0.0/16`
- 在该网络下创建 3 台虚拟机，虚拟机之间可以 ping 通
- 虚拟机无法访问外部网络，外部也无法访问这些虚拟机

---

### US-02：应用开发者在 NAT 网络中部署服务

> 作为**应用开发者**，我希望**在 NAT 网络中创建虚拟机并绑定安全组**，以便**虚拟机可以访问外网下载依赖，同时受安全组保护**。

**验收条件**：
- 创建 NAT 网络 `dev-net`，CIDR `172.16.0.0/20`
- 创建子网 `dev-subnet-01`，CIDR `172.16.1.0/24`，启用 DHCP（范围 `172.16.1.100-200`）
- 创建安全组 `dev-sg-web`，允许入方向 TCP 80/443
- 创建虚拟机并选择 `dev-net` + `dev-subnet-01` + `dev-sg-web`
- 虚拟机自动获取 IP `172.16.1.x`，可访问外网，外部可访问其 80/443 端口

---

### US-03：安全管理员批量管理安全组规则

> 作为**安全管理员**，我希望**统一管理所有租户的安全组规则**，以便**快速响应安全事件（如封禁恶意 IP）**。

**验收条件**：
- 安全管理员可查看所有租户的安全组列表
- 可批量向多个安全组添加拒绝规则（如封禁 `10.0.99.0/24`）
- 规则变更后 5 秒内生效于已绑定的虚拟机
- 操作记录写入审计日志

---

### US-04：应用开发者配置七层负载均衡

> 作为**应用开发者**，我希望**创建 CLB 实例并配置 HTTP 监听器**，以便**将流量分发到多台 Web 服务器**。

**验收条件**：
- 创建内网 CLB 实例 `web-lb-01`，选择网络 `prod-net`
- 添加 HTTP 监听器（端口 80），转发规则按 URL 路径分发
- 添加 3 台后端服务器（`10.0.1.10:8080`, `10.0.1.11:8080`, `10.0.1.12:8080`），权重分别为 50/30/20
- 配置健康检查（HTTP GET `/health`，间隔 5s，超时 3s，不健康阈值 3 次）
- 访问 CLB VIP 的 80 端口，请求按权重分发到健康的后端

---

### US-05：运维工程师管理 SSL 证书并绑定 HTTPS 监听器

> 作为**运维工程师**，我希望**上传 SSL 证书并绑定到 HTTPS 监听器**，以便**为 Web 服务启用 HTTPS 加密**。

**验收条件**：
- 上传 PEM 格式证书和私钥，系统验证证书有效性
- 创建 HTTPS 监听器（端口 443），绑定已上传的证书
- 客户端访问 `https://<CLB_VIP>` 时，CLB 完成 SSL 卸载
- 证书到期前 30 天，系统发送告警通知
- 支持证书更新（不中断现有连接）

---

### US-06：租户管理员查看网络拓扑

> 作为**租户管理员**，我希望**可视化查看本租户的网络拓扑**，以便**快速了解虚拟机与网络、安全组、CLB 的关联关系**。

**验收条件**：
- 拓扑图展示：网络 → 子网 → 虚拟机（含安全组标签）
- 虚拟机节点展示其绑定的安全组和所属 CLB 后端池
- 点击节点可跳转到详情页
- 支持导出拓扑图为 PNG

## 四、详细需求描述（EARS 格式）

### 4.1 虚拟网络

#### 4.1.1 虚拟网络列表

- **[Ubiquitous]** 系统应始终以表格形式展示虚拟网络列表，每行包含：网络名称、网络类型（NAT/隔离/桥接）、CIDR、子网数量、关联虚拟机数量、所属租户、创建时间、状态（可用/创建中/异常）。
- **[Event-driven]** 当用户输入关键词并点击搜索时，系统应在 500ms 内返回筛选结果，支持按名称模糊匹配和按类型/租户精确匹配。
- **[State-driven]** 在加载网络列表期间，系统应展示骨架屏加载状态。
- **[Unwanted]** 如果列表请求失败（超时或服务端错误），则系统应展示错误提示"加载失败，请重试"，并提供重试按钮。

#### 4.1.2 创建虚拟网络（NAT 类型）

- **[Event-driven]** 当用户在创建页面选择"NAT 网络"类型时，系统应展示 NAT 网络配置表单，包含：网络名称、CIDR（如 `172.16.0.0/16`）、描述、租户选择。
- **[State-driven]** 在填写 CIDR 期间，系统应实时校验 CIDR 格式合法性，并检测是否与已有网络 CIDR 重叠。若重叠，应展示警告"C类CIDR与已有网络 xxx 重叠，请调整"。
- **[Event-driven]** 当用户提交创建请求时，系统应在宿主机上执行以下操作：
  1. 创建 Linux Bridge（命名规则 `vn-{network_id}`）
  2. 配置 iptables NAT 规则（SNAT/MASQUERADE）
  3. 启动 dnsmasq 实例（绑定到 Bridge）
  4. 创建网络记录并写入数据库
- **[State-driven]** 在创建过程中，系统应展示进度条：创建 Bridge → 配置 NAT → 启动 DHCP → 完成。
- **[Unwanted]** 如果任一创建步骤失败，则系统应回滚已完成步骤（删除已创建的 Bridge 和 NAT 规则），并返回具体失败原因。
- **[Optional]** 当用户启用"高级配置"时，系统应展示额外选项：MTU 设置（默认 1500）、DNS 服务器（默认宿主机 DNS）、是否启用 IPv6（默认关闭）。

#### 4.1.3 创建虚拟网络（隔离类型）

- **[Event-driven]** 当用户在创建页面选择"隔离网络"类型时，系统应展示隔离网络配置表单，包含：网络名称、CIDR、描述、租户选择。
- **[Event-driven]** 当用户提交创建请求时，系统应执行：
  1. 创建 Linux Bridge（命名规则 `vn-{network_id}`）
  2. **不配置**任何 NAT 或路由规则（纯二层隔离）
  3. 启动 dnsmasq 实例（可选，取决于是否启用 DHCP）
  4. 创建网络记录
- **[Unwanted]** 如果 CIDR 与已有隔离网络重叠，则系统应拒绝创建并提示"C类CIDR已被隔离网络 xxx 使用，隔离网络之间 CIDR 可以重叠但不建议，确认继续？"（提供强制创建选项）。

#### 4.1.4 创建虚拟网络（桥接类型）

- **[Event-driven]** 当用户在创建页面选择"桥接网络"类型时，系统应展示桥接网络配置表单，包含：网络名称、物理网卡选择（下拉列表展示宿主机可用网卡）、描述、租户选择。
- **[Event-driven]** 当用户提交创建请求时，系统应执行：
  1. 创建 Linux Bridge
  2. 将选定的物理网卡加入 Bridge
  3. 迁移物理网卡 IP 到 Bridge
  4. 创建网络记录
- **[Unwanted]** 如果物理网卡已被其他桥接网络使用，则系统应拒绝创建并提示"物理网卡 eth0 已被桥接网络 xxx 使用"。
- **[Unwanted]** 如果迁移 IP 过程中物理网卡失去连接，则系统应在 30 秒内自动回滚配置。

#### 4.1.5 跨宿主机二层互通（VLAN 配置）

不同宿主机上属于同一虚拟网络的 VM 需要二层互通。PRD 提供两种方案，一期采用方案 A。

**方案 A（推荐，一期）：物理交换机 VLAN Trunk**

- **[Event-driven]** 当用户在创建虚拟网络（NAT/隔离类型）时，系统应自动为该网络分配一个唯一的 VLAN ID（从预配置的 VLAN 池中选取，范围 2-4094）。
- **[Event-driven]** 当 VLAN ID 分配完成后，系统应在宿主机上执行以下操作：
  1. 创建带 VLAN tag 的 Linux Bridge：`ip link add name vn-{network_id} type bridge`
  2. 创建 VLAN 子接口绑定到物理网卡：`ip link add link eth0 name eth0.{vlan_id} type vlan id {vlan_id}`
  3. 将 VLAN 子接口加入 Bridge：`ip link set eth0.{vlan_id} master vn-{network_id}`
  4. 激活所有接口
- **[Event-driven]** 当网络创建成功后，系统应生成"物理交换机配置指引"，提示运维人员在交换机上执行以下操作：
  - 创建 VLAN `{vlan_id}`
  - 将所有宿主机物理网卡对应端口配置为 Trunk 模式，允许 VLAN `{vlan_id}` 通过
  - 操作完成后，在平台确认"交换机已配置"
- **[State-driven]** 在交换机配置确认之前，网络状态应展示为"交换机待配置"，并醒目提示"请完成物理交换机 VLAN Trunk 配置后确认"。
- **[Unwanted]** 如果分配的 VLAN ID 已被其他虚拟网络使用，则系统应自动分配下一个可用 VLAN ID。
- **[Optional]** 当用户在创建页面展开"高级配置"时，系统应允许用户手动指定 VLAN ID（覆盖自动分配），手动指定时需校验 VLAN ID 范围（2-4094）且未被占用。

**方案 B（备选，二期）：VXLAN Overlay**

- 不依赖物理交换机 VLAN 配置，通过 VXLAN 隧道在不同宿主机之间建立二层 overlay 网络。
- 每个虚拟网络分配唯一的 VNI（VXLAN Network Identifier，范围 1-16777215）。
- 宿主机之间通过 VXLAN 隧道封装原始二层帧，实现跨宿主机的二层互通。
- **局限性**：VXLAN 封装/解封装有性能损耗（约 5-10% 吞吐量下降），且需要额外配置 VTEP（VXLAN Tunnel Endpoint）。
- 二期迁移至 OVS/OVN 架构时统一支持。

**VLAN ID 管理规则**

- **[Ubiquitous]** 系统应维护全局 VLAN ID 池（范围 2-4094），已分配的 VLAN ID 不可重复使用。
- **[Event-driven]** 当用户删除虚拟网络时，系统应自动回收其 VLAN ID，归还到可用池。
- **[State-driven]** 在 VLAN 池利用率超过 80% 时，系统应发送告警通知平台管理员。
- **[Unwanted]** 如果 VLAN ID 池耗尽（所有 2-4094 均已分配），则系统应拒绝创建新网络并提示"VLAN ID 资源已耗尽，请联系平台管理员扩展或清理未使用的网络"。

#### 4.1.6 删除虚拟网络

- **[Event-driven]** 当用户点击删除按钮时，系统应展示依赖检查结果弹窗：
  - 存在子网：列出所有子网名称，提示"请先删除所有子网"
  - 存在虚拟机：列出所有虚拟机名称，提示"请先迁移或删除所有虚拟机"
  - 无依赖：展示确认删除弹窗"删除网络 xxx 后，相关网络配置将被清理，此操作不可恢复"
- **[Event-driven]** 当用户确认删除且依赖检查通过时，系统应执行：
  1. 停止 dnsmasq 实例
  2. 删除 iptables NAT 规则
  3. 从 Bridge 移除 VLAN 子接口并删除子接口
  4. 删除 Linux Bridge
  5. 回收 VLAN ID 到可用池
  6. 删除数据库记录
- **[Unwanted]** 如果删除过程中任意步骤失败，则系统应记录失败日志并触发告警，提示管理员手动清理。

#### 4.1.7 子网创建与 DHCP 配置

- **[Event-driven]** 当用户在虚拟网络详情页点击"创建子网"时，系统应展示子网创建表单，包含：子网名称、CIDR（必须在父网络 CIDR 内）、网关 IP、是否启用 DHCP、DHCP 地址池范围。
- **[State-driven]** 在用户输入子网 CIDR 期间，系统应实时校验：CIDR 必须是父网络 CIDR 的子集、不能与已有子网重叠。
- **[Event-driven]** 当用户启用 DHCP 时，系统应展示 DHCP 配置区域：
  - 地址池起始 IP / 结束 IP
  - 租约时间（默认 24 小时）
  - DNS 服务器（默认继承网络 DNS）
- **[Event-driven]** 当用户提交创建请求时，系统应：
  1. 校验 CIDR 合法性和非重叠性
  2. 更新 dnsmasq 配置（添加 DHCP 范围到对应 Bridge）
  3. 重载 dnsmasq
  4. 创建子网记录
- **[Unwanted]** 如果 DHCP 地址池超出子网 CIDR 范围，则系统应拒绝并提示"D类CP 地址池必须在子网 CIDR 范围内"。

### 4.2 安全组

#### 4.2.1 安全组创建

- **[Event-driven]** 当用户点击"创建安全组"时，系统应展示创建表单：安全组名称、描述、所属租户。
- **[Event-driven]** 当用户提交创建时，系统应创建安全组记录，并自动添加以下默认规则：
  - 出方向：允许所有（TCP/UDP/ICMP，目标 0.0.0.0/0）
  - 入方向：拒绝所有（优先级最低的默认规则）
- **[Ubiquitous]** 安全组名称在同一租户内必须唯一。
- **[Unwanted]** 如果同一租户下已存在同名安全组，则系统应提示"安全组名称已存在"，并建议修改名称。

#### 4.2.2 安全组规则编辑（入方向/出方向）

- **[Event-driven]** 当用户在安全组详情页点击"添加规则"并选择"入方向"时，系统应展示入方向规则表单：

| 字段 | 类型 | 说明 |
|------|------|------|
| 协议 | 下拉 | TCP / UDP / ICMP / ALL |
| 端口范围 | 文本 | 单个端口 `80` 或范围 `8000-9000`，协议为 ALL 时禁用 |
| 来源类型 | 下拉 | IP/CIDR / 安全组 |
| 来源值 | 文本/下拉 | IP/CIDR 输入框 或 安全组选择器 |
| 策略 | 下拉 | 允许 / 拒绝 |
| 优先级 | 数字 | 1-100，数字越小优先级越高 |
| 描述 | 文本 | 规则说明（可选） |

- **[Event-driven]** 当用户选择"安全组"作为来源类型时，系统应展示安全组选择器（支持跨租户只读引用），规则将匹配来源安全组关联的所有虚拟机 IP。
- **[Event-driven]** 当用户提交规则时，系统应校验：
  - 端口范围合法性（1-65535，起始 <= 结束）
  - 来源 CIDR 格式合法性
  - 优先级不重复（同方向同优先级时自动调整）
- **[State-driven]** 在规则保存期间，系统应展示"规则同步中"状态，规则新增后应在 5 秒内下发到已绑定的虚拟机。
- **[Unwanted]** 如果规则下发到某台虚拟机失败（如虚拟机已关机），则系统应记录失败状态并在规则列表标注"部分生效"，提供重试按钮。

#### 4.2.3 安全组绑定/解绑虚拟机

- **[Event-driven]** 当用户在安全组详情页点击"绑定实例"时，系统应展示虚拟机选择器（列表/搜索），支持多选，仅展示安全组所属租户的虚拟机。
- **[Event-driven]** 当用户选择虚拟机并确认绑定时，系统应：
  1. 检查虚拟机网卡数量（每张网卡最多绑定 5 个安全组）
  2. 将安全组规则转换为 iptables/nftables 规则
  3. 通过 agent 下发规则到虚拟机所在宿主机
  4. 更新绑定关系记录
- **[State-driven]** 在规则下发期间，绑定状态应展示为"同步中"，完成后展示"已生效"。
- **[Event-driven]** 当用户解绑安全组时，系统应展示确认弹窗"解绑后，该安全组规则将不再对虚拟机 xxx 生效，可能影响网络访问"，确认后清理对应 iptables 规则。
- **[Unwanted]** 如果虚拟机是某安全组的最后一个绑定实例，且该安全组被其他安全组规则引用（作为来源），则系统应警告"安全组 xxx 被安全组 yyy 引用，解绑后 yyy 的规则可能失效"。

#### 4.2.4 规则冲突检测

- **[State-driven]** 在用户添加或编辑规则时，系统应检测以下冲突场景并给出提示：
  - **覆盖冲突**：新规则与已有规则协议+端口+来源完全相同但策略不同 → 警告"此规则将被优先级更高的规则覆盖，当前不会生效"
  - **冗余规则**：新规则是已有规则的子集（如已有 0.0.0.0/0 允许，新增 /24 允许）→ 提示"此规则已被更宽泛的规则覆盖，为冗余规则"
  - **优先级重叠**：新规则优先级与已有规则相同 → 自动调整已有规则优先级
- **[Optional]** 当用户在租户设置中启用"严格模式"时，系统应在检测到覆盖冲突时拒绝保存。

### 4.3 负载均衡

#### 4.3.1 CLB 实例创建

- **[Event-driven]** 当用户点击"创建 CLB"时，系统应展示创建向导：

**步骤 1：基本配置**
| 字段 | 类型 | 说明 |
|------|------|------|
| 实例名称 | 文本 | 2-64 字符，支持字母数字连字符 |
| 网络类型 | 下拉 | 内网（P0）/ 公网（P1） |
| 所属网络 | 下拉 | 选择虚拟网络 |
| 所属子网 | 下拉 | 级联选择（筛选所选网络下的子网） |
| 规格 | 下拉 | 小型（<=1000连接）/ 中型（<=10000连接）/ 大型（<=50000连接） |
| 高可用模式 | 只读 | **主备模式（Keepalived + VIP）**，系统默认创建主备两个实例 |
| 描述 | 文本 | 可选 |

**步骤 2：监听器配置（可选，可后续添加）**
- 协议选择：TCP / UDP / HTTP / HTTPS
- 端口：1-65535
- 调度算法：轮询 / 最小连接 / 源 IP 哈希

**步骤 3：确认创建**

- **[Event-driven]** 当用户确认创建时，系统应：
  1. 从子网中分配一个 VIP（优先分配子网中第一个可用 IP）
  2. 选择两台不同宿主机（通过反亲和调度确保主备分布在不同物理机）
  3. 在主宿主机上部署 Nginx/HAProxy 实例（主 CLB）
  4. 在备宿主机上部署 Nginx/HAProxy 实例（备 CLB），配置与主 CLB 同步
  5. 配置 Keepalived，主 CLB 优先级 100，备 CLB 优先级 90
  6. 绑定 VIP 到主 CLB 的 Keepalived 实例
  7. 创建 CLB 记录（主备两个实例关联同一 CLB 资源）
- **[State-driven]** 在创建过程中，系统应展示进度：分配 VIP → 选择宿主机 → 部署主实例 → 部署备实例 → 配置 Keepalived → 启动服务。
- **[Unwanted]** 如果子网中无可用 IP，则系统应提示"子网 IP 地址已耗尽，请选择其他子网或扩展 CIDR"。
- **[Unwanted]** 如果可用宿主机不足两台（无法满足反亲和需求），则系统应提示"可用宿主机不足，无法创建高可用 CLB 实例"，但仍允许单机创建（降级模式），并在实例列表标注"无高可用保护"。

#### 4.3.1.1 CLB 高可用机制（Keepalived + VIP）

- **[Ubiquitous]** CLB 实例默认采用主备高可用模式。系统自动创建两个实例（主 CLB + 备 CLB），分别部署在不同宿主机上。
- **[Event-driven]** 当主 CLB 所在宿主机发生故障或 Keepalived 心跳超时（>3 秒），VIP 应自动漂移到备 CLB 实例：
  1. 备 CLB 的 Keepalived 检测到主 CLB VRRP 通告超时
  2. 备 CLB 升级为主 CLB，接管 VIP
  3. 发送 GARP 通知物理交换机更新 MAC 地址表
  4. 整个切换过程目标在 3 秒内完成
- **[State-driven]** 在故障切换期间，CLB 状态应展示为"切换中"，切换完成后更新为"运行中（备升主）"，并标注原主 CLB 为"异常"。
- **[Event-driven]** 当原主 CLB 恢复后，系统应自动将其降级为备 CLB（Keepalived 优先级低于当前主 CLB），VIP 不回流（避免二次中断），由运维手动决定是否切回。
- **[Event-driven]** 系统应在 CLB 详情页展示主备状态信息：主 CLB 所在宿主机、备 CLB 所在宿主机、VIP 当前所在宿主机、上次切换时间。
- **[Unwanted]** 如果主备 CLB 均故障（两台宿主机同时故障），则 CLB 状态标记为"不可用"，触发紧急告警。

#### 4.3.2 监听器配置（HTTP/HTTPS）

- **[Event-driven]** 当用户在 CLB 详情页创建 HTTP 监听器时，系统应展示配置表单：

| 字段 | 类型 | 说明 |
|------|------|------|
| 监听协议 | 只读 | HTTP |
| 监听端口 | 数字 | 1-65535，同一 CLB 内端口不可重复 |
| 后端端口 | 数字 | 后端服务器的真实端口 |
| 调度算法 | 下拉 | 轮询 / 最小连接 / 源 IP 哈希 |
| 会话保持 | 开关 | 默认关闭，开启后需设置超时时间（30-3600s） |
| 获取客户端 IP | 开关 | 默认开启（通过 X-Forwarded-For） |

- **[Event-driven]** 当用户配置转发规则时，系统应支持：
  - **域名转发**：`www.example.com` → 指定后端服务器组
  - **URL 路径转发**：`/api/*` → 指定后端服务器组
  - **默认转发**：未匹配的请求 → 默认后端服务器组
- **[Event-driven]** 当用户创建 HTTPS 监听器时，系统应额外要求选择 SSL 证书（从已上传证书列表选择），并支持配置 SSL 协议版本（TLS 1.2 / TLS 1.3）。
- **[Unwanted]** 如果所选端口已被同一 CLB 的其他监听器使用，则系统应提示"端口 xxx 已被监听器 yyy 使用"。

#### 4.3.3 后端服务器管理

- **[Event-driven]** 当用户在监听器详情页点击"添加后端服务器"时，系统应展示选择器：
  - 左侧：可选虚拟机列表（仅展示同一网络且运行中的虚拟机），支持搜索和筛选
  - 右侧：已选虚拟机列表，需填写每台虚拟机的后端端口和权重
- **[Event-driven]** 当用户确认添加时，系统应：
  1. 校验后端服务器可达性（TCP 端口探测）
  2. 更新 Nginx/HAProxy 配置（upstream 组）
  3. 重载 Nginx/HAProxy
  4. 更新后端服务器记录
- **[Unwanted]** 如果后端服务器端口不可达（探测失败），则系统应提示"虚拟机 xxx:yyyy 端口不可达，仍要添加吗？"（允许强制添加，但健康状态标记为"不健康"）。
- **[Event-driven]** 当用户调整后端服务器权重时，系统应实时更新 Nginx/HAProxy 配置并重载。
- **[State-driven]** 在配置重载期间，系统应使用 Nginx/HAProxy 的 graceful reload 机制，确保现有连接不中断。

#### 4.3.4 健康检查

- **[Event-driven]** 当用户配置健康检查时，系统应展示配置表单：

| 字段 | 类型 | 说明 |
|------|------|------|
| 检查协议 | 下拉 | TCP / HTTP / HTTPS |
| 检查端口 | 数字 | 默认使用后端端口，可自定义 |
| 检查路径 | 文本 | HTTP/HTTPS 时必填，如 `/health` |
| 检查域名 | 文本 | HTTP/HTTPS 时可选，用于 Host 头 |
| 响应超时 | 数字 | 2-60 秒，默认 3 秒 |
| 检查间隔 | 数字 | 2-300 秒，默认 5 秒 |
| 健康阈值 | 数字 | 2-10 次，默认 2 次 |
| 不健康阈值 | 数字 | 2-10 次，默认 3 次 |
| HTTP 状态码 | 多选 | 默认 200，可添加 2xx/3xx |

- **[Event-driven]** 当健康检查状态变化时，系统应：
  - **从不健康→健康**：自动将该后端加入流量分发
  - **从健康→不健康**：自动停止向该后端分发流量
- **[State-driven]** 在后端服务器列表的健康状态列，系统应展示：绿色圆点（健康）、红色圆点（不健康）、灰色旋转图标（检查中）。
- **[Unwanted]** 如果监听器下所有后端服务器都不健康，则系统应触发告警"监听器 xxx 所有后端不健康"，并返回 503 状态码。

#### 4.3.5 SSL 证书管理

- **[Event-driven]** 当用户上传 SSL 证书时，系统应展示上传表单：

| 字段 | 类型 | 说明 |
|------|------|------|
| 证书名称 | 文本 | 便于识别的名称 |
| 证书内容 | 文本域 | PEM 格式公钥证书 |
| 私钥内容 | 文本域 | PEM 格式私钥（前端不缓存，传输加密） |
| 描述 | 文本 | 可选 |

- **[Event-driven]** 当用户提交上传时，系统应：
  1. 校验证书格式（PEM）
  2. 校验证书与私钥匹配
  3. 解析证书域名（CN/SAN）
  4. 解析证书到期时间
  5. 加密存储私钥
  6. 创建证书记录
- **[Unwanted]** 如果证书格式不合法，则系统应具体提示"C类证书格式错误：不是有效的 PEM 格式"或"C类私钥与证书不匹配"。
- **[Unwanted]** 如果证书已过期，则系统应允许上传但标记为"已过期"并展示警告。
- **[State-driven]** 在证书到期前 30/15/7/1 天，系统应自动发送告警通知给证书上传者和租户管理员。
- **[Unwanted]** 如果用户尝试删除正在被 HTTPS 监听器使用的证书，则系统应提示"C类证书正被监听器 xxx 使用，请先解绑或更新监听器证书"。

### 4.4 浮动 IP

#### 4.4.1 浮动 IP 申请

- **[Event-driven]** 当用户点击"申请浮动 IP"时，系统应展示申请表单：

| 字段 | 类型 | 说明 |
|------|------|------|
| 所属网络 | 下拉 | 选择虚拟网络，按租户过滤 |
| IP 分配方式 | 单选 | 自动分配（系统从子网 IP 池中选取）/ 手动指定 |
| 手动指定 IP | 文本 | 仅"手动指定"时展示，需校验 IP 在子网 CIDR 内且未被占用 |
| 数量 | 数字 | 默认 1，范围 1-10（仅"自动分配"时可用） |
| 描述 | 文本 | 可选 |

- **[Event-driven]** 当用户提交申请时，系统应：
  1. 校验配额（租户浮动 IP 数量上限，默认 50）
  2. 自动分配模式下：从子网 DHCP 池外选取可用 IP
  3. 手动指定模式下：校验 IP 未被占用且不处于 DHCP 地址池范围
  4. 创建浮动 IP 资源，状态初始为"未绑定"
- **[Unwanted]** 如果子网可用 IP 不足（剩余 < 申请数量），则系统应提示"子网 xxx 可用 IP 不足（剩余 N 个），请选择其他子网或减少申请数量"。
- **[Unwanted]** 如果手动指定的 IP 已被占用（包括存量 VM 的 IP），则系统应提示"IP 地址 xxx 已被占用，请更换"。

#### 4.4.2 浮动 IP 绑定虚拟机

- **[Event-driven]** 当用户对"未绑定"状态的浮动 IP 点击"绑定"时，系统应展示绑定表单：
  - 选择目标虚拟机（下拉搜索，按租户过滤，仅展示与浮动 IP 同一网络的虚拟机）
  - 选择目标网卡（展示虚拟机所有网卡，标注主网卡/辅助网卡）
- **[Event-driven]** 当用户确认绑定时，系统应：
  1. 检查浮动 IP 与目标 VM 是否在同一虚拟网络
  2. **检查网络类型与宿主机限制**：NAT/隔离网络下，目标 VM 必须与浮动 IP 历史绑定记录在同一宿主机（因为 NAT/隔离网络的 IP 仅在宿主机内部可见，不支持跨宿主机漂移）
  3. 通过 agent 在目标虚拟机网卡上添加浮动 IP 作为辅助 IP（`ip addr add`）
  4. 在目标宿主机上配置 ARP 响应（防止 IP 冲突）
  5. 更新浮动 IP 状态为"已绑定"，记录绑定关系
- **[State-driven]** 在浮动 IP 配置期间，绑定状态应展示为"绑定中"，配置完成后展示"已绑定"并显示绑定的 VM 名称和网卡。
- **[Unwanted]** 如果目标虚拟机所在宿主机 agent 不可达，则系统应提示"无法连接到虚拟机所在宿主机，绑定失败"，浮动 IP 保持"未绑定"状态。
- **[Unwanted]** 如果浮动 IP 与目标 VM 不在同一虚拟网络，则系统应提示"浮动 IP 与目标虚拟机必须在同一虚拟网络"。
- **[Unwanted]** 如果浮动 IP 所属网络为 NAT/隔离类型，且目标 VM 与浮动 IP 历史绑定记录不在同一宿主机，则系统应提示"浮动 IP 所属为 NAT/隔离网络，仅支持同宿主机内绑定，不支持跨宿主机漂移"。

**浮动 IP 跨宿主机漂移能力说明**

| 网络类型 | 同宿主机漂移 | 跨宿主机漂移 | 原理 |
|---------|:----------:|:----------:|------|
| 桥接网络 | 支持 | **支持** | 通过 GARP 通知物理交换机更新 MAC 表，跨宿主机 IP 可达 |
| NAT 网络 | 支持 | **不支持** | NAT 网络的 IP 仅在宿主机内部可见，无法跨宿主机路由 |
| 隔离网络 | 支持 | **不支持** | 隔离网络为纯二层隔离域，无法跨宿主机通信 |

#### 4.4.3 浮动 IP 解绑虚拟机

- **[Event-driven]** 当用户对"已绑定"状态的浮动 IP 点击"解绑"时，系统应展示确认弹窗："解绑后，IP 地址 xxx 将从虚拟机 yyy 移除。浮动 IP 资源保留，可重新绑定到其他虚拟机。"
- **[Event-driven]** 当用户确认解绑时，系统应：
  1. 通过 agent 从虚拟机网卡移除浮动 IP（`ip addr del`）
  2. 清理宿主机上的 ARP 配置
  3. 浮动 IP 状态恢复为"未绑定"
  4. 记录解绑历史
- **[Unwanted]** 如果虚拟机已关机或 agent 不可达，则系统应允许强制解绑（仅清理数据库绑定关系，标记需后续手动清理网络配置），并提示"已强制解绑，请在虚拟机恢复后手动检查网络配置"。

#### 4.4.4 浮动 IP 释放

- **[Event-driven]** 当用户对"未绑定"状态的浮动 IP 点击"释放"时，系统应展示确认弹窗（需输入浮动 IP 地址确认）。
- **[Unwanted]** 如果浮动 IP 处于"已绑定"状态，则系统应提示"请先解绑后再释放"，不展示释放按钮。
- **[Event-driven]** 当用户确认释放时，系统应删除浮动 IP 资源，IP 归还子网可用池。
- **[Ubiquitous]** 浮动 IP 释放操作必须记录审计日志（操作人/时间/IP 地址/所属网络）。

#### 4.4.5 浮动 IP 故障漂移（P2）

- **[Optional]** 当启用浮动 IP 漂移功能时，系统应允许用户将已绑定的浮动 IP 从虚拟机 A 直接迁移到虚拟机 B（在同一虚拟网络内），无需先解绑再绑定。
- **[Event-driven]** 当用户触发漂移操作时，系统应首先校验漂移可行性：
  - **桥接网络**：允许跨宿主机漂移，通过 GARP 通知交换机更新 MAC 地址表
  - **NAT 网络**：仅允许同宿主机漂移，拒绝跨宿主机漂移请求
  - **隔离网络**：仅允许同宿主机漂移，拒绝跨宿主机漂移请求
- **[Event-driven]** 当漂移校验通过后，系统应：
  1. 从虚拟机 A 移除浮动 IP（`ip addr del`）
  2. 在旧宿主机清理 ARP 条目（`arp -d`）
  3. 在虚拟机 B 添加浮动 IP（`ip addr add`）
  4. **桥接网络**：发送 GARP（Gratuitous ARP）3 次，间隔 1 秒，通知物理交换机更新 MAC 地址表
  5. 确认新 IP 可达性（从宿主机 ping 浮动 IP）
  6. 更新数据库绑定关系
- **[State-driven]** 整个漂移过程目标在 30 秒内完成（含 3 次 GARP 间隔）。旧宿主机 ARP 清理 + 新宿主机 IP 绑定约 5 秒，GARP 广播确认约 3 秒，可达性验证约 2 秒。
- **[Unwanted]** 如果漂移过程中任一步骤失败，则系统应回滚：将浮动 IP 重新绑定到原虚拟机 A，并提示"漂移失败，已回滚至原虚拟机"。
- **[Unwanted]** 如果漂移目标网络类型为 NAT 或隔离网络且目标 VM 在不同宿主机，则系统应拒绝漂移并提示"浮动 IP 所属为 {网络类型} 网络，仅支持同宿主机漂移，不支持跨宿主机漂移。如需跨宿主机漂移，请使用桥接网络"。---

### 6.3 状态流转

#### 6.3.1 虚拟网络状态流转

```
         ┌──────┐
         │ 创建中 │ ← 创建请求已提交，Bridge/NAT/DHCP 配置中
         └──┬───┘
            │ 成功
            ↓
         ┌──────┐     删除      ┌──────┐
         │ 可用  │ ──────────→  │ 删除中 │
         └──┬───┘              └──┬───┘
            │ 异常                 │ 成功
            ↓                      ↓
         ┌──────┐              ┌──────┐
         │ 异常  │              │ 已删除 │
         └──────┘              └──────┘
```

#### 6.3.2 CLB 实例状态流转

```
         ┌──────┐
         │ 创建中 │ ← 分配 VIP + 部署主备 Nginx/HAProxy + 配置 Keepalived
         └──┬───┘
            │
            ↓
         ┌──────┐     停止     ┌──────┐
         │ 运行中 │ ──────────→ │ 停止中 │
         └──┬───┘              └──┬───┘
            │                     │
            ├── 主故障              ↓
            ↓                  ┌──────┐
         ┌──────┐     启动     │ 已停止 │
         │ 切换中 │ ← ──────────│      │
         └──┬───┘              └──┬───┘
            │                     │ 删除
            ↓                     ↓
         ┌──────┐              ┌──────┐
         │ 异常  │              │ 删除中 │
         └──┬───┘              └──┬───┘
            │                     │
            │ 删除                ↓
            ↓                  ┌──────┐
         ┌──────┐              │ 已删除 │
         │ 删除中 │              └──────┘

            │
            ↓
         ┌──────┐
         │ 已删除 │
         └──────┘
```
**注**："切换中"状态表示主 CLB 故障、VIP 正在向备 CLB 漂移的过程（< 3 秒），切换完成后备 CLB 升级为主，原主 CLB 标记为"异常"。

## 七、边界场景与异常处理

### 7.1 安全组规则冲突

**场景**：用户添加了一条入方向规则"允许 TCP 80 来自 0.0.0.0/0"，又添加了"拒绝 TCP 80 来自 10.0.1.0/24"，两条规则优先级相同。

**处理**：
1. 系统在保存时检测到优先级重叠，自动将后添加的规则优先级+1
2. 如果用户手动设置相同优先级，系统提示"优先级与规则 xxx 冲突，已自动调整为 yyy"
3. 规则匹配按优先级从小到大依次匹配，命中即停止（类似 iptables 链）
4. 前端规则列表支持拖拽排序，直观展示规则生效顺序

### 7.2 删除正在使用的网络

**场景**：用户尝试删除一个仍有 3 台虚拟机在运行的虚拟网络。

**处理**：
1. 系统弹出依赖检查结果：列出所有关联的子网（含子网下虚拟机数量）
2. 禁止删除按钮，按钮置灰并提示"请先删除/迁移所有子网和虚拟机"
3. 提供"查看关联资源"链接，跳转到子网列表和虚拟机列表（按网络筛选）
4. 如果用户强制需要删除，需联系平台管理员进行数据库级操作
5. 删除成功后，VLAN ID 自动回收到可用池，物理交换机上的 VLAN 配置由运维手动清理（或通过平台调用交换机 API，二期）

### 7.3 删除正在使用的安全组

**场景**：用户尝试删除一个已绑定到 10 台虚拟机的安全组。

**处理**：
1. 弹出确认弹窗，列出已绑定的 10 台虚拟机名称和 IP
2. 提供"批量解绑并删除"选项（需额外确认）
3. 如果用户选择"仅解绑不删除"，引导至安全组详情页的绑定实例标签页
4. 删除后，系统异步清理所有虚拟机上的对应 iptables 规则

### 7.4 后端服务器全部不健康

**场景**：CLB 监听器下的 5 台后端服务器全部健康检查失败。

**处理**：
1. CLB 监听器状态变更为"告警"，在列表页以红色高亮展示
2. 系统发送告警通知给 CLB 所属租户的管理员和运维团队
3. CLB 返回 503 Service Unavailable 给客户端（而非连接超时）
4. 用户可通过详情页查看每台后端的具体失败原因（超时/连接拒绝/状态码异常）
5. 提供"暂停流量"按钮，允许管理员手动停止流量分发以便排查

### 7.5 SSL 证书过期

**场景**：绑定了 HTTPS 监听器的 SSL 证书即将过期或已过期。

**处理**：
1. 到期前 30/15/7/1 天：系统自动发送站内信和邮件通知给证书上传者 + 租户管理员
2. 证书列表页：到期时间列颜色变化（>30 天绿色，7-30 天黄色，<7 天红色）
3. 已过期的证书：监听器仍然工作（不中断服务），但列表页展示"已过期"标签
4. 提供"更新证书"入口，上传新证书后自动替换（graceful reload，不中断连接）
5. 如果用户开启了"证书到期自动提醒"功能，可在到期前 7 天每日提醒

### 7.6 LB 实例所在宿主机故障

**场景**：部署了 CLB 主实例的宿主机（Dell R740 物理机）发生故障。

**处理**：
1. 备 CLB 的 Keepalived 检测到主 CLB VRRP 心跳超时（>3 秒），触发 VIP 漂移
2. VIP 自动漂移到备 CLB 所在宿主机，备 CLB 升级为主 CLB
3. 发送 GARP 通知物理交换机更新 MAC 地址表
4. 整个切换过程目标在 3 秒内完成，对客户端透明（短暂丢包 1-2 个请求）
5. 原主 CLB 状态标记为"异常"，触发告警通知运维团队
6. 运维修复原宿主机后，原主 CLB 自动降级为备 CLB（VIP 不回流）
7. 运维可在确认新主 CLB 稳定后，通过平台手动发起 VIP 切回操作

### 7.7 并发修改安全组规则

**场景**：两个管理员同时编辑同一安全组的规则（一个添加规则，一个删除规则）。

**处理**：
1. 安全组规则采用乐观锁机制（版本号）
2. 后提交的用户收到冲突提示"安全组已被其他用户修改，请刷新后重试"
3. 前端展示当前最新规则，保留用户的修改草稿（localStorage）
4. 规则下发采用全量替换策略（非增量），避免并发导致规则状态不一致

### 7.8 网络 IP 地址耗尽

**场景**：子网 `10.0.1.0/24` 中 254 个可用 IP 已全部分配。

**处理**：
1. 子网列表页展示"可用 IP：0/254"，进度条红色
2. 用户尝试创建虚拟机时，该子网在下拉框中置灰并标注"IP 已耗尽"
3. 系统发送告警通知（可用 IP < 10% 时黄色预警，< 5% 时红色告警）
4. 提供解决方案引导：创建新子网（扩展 CIDR）、释放闲置 IP、删除未使用的虚拟机
5. DHCP 地址池耗尽但子网仍有 IP 时，提示用户扩展 DHCP 池范围

### 7.9 安全组引用链断裂

**场景**：安全组 A 的入方向规则引用了安全组 B 作为来源，但安全组 B 被删除。

**处理**：
1. 删除安全组 B 时，系统检测到被安全组 A 引用，弹出警告
2. 用户确认删除后，安全组 A 中引用安全组 B 的规则自动失效（标记为"引用已失效"）
3. 安全组 A 详情页展示失效规则，提供"编辑"和"删除"操作
4. 失效规则不参与 iptables 规则生成（等同于不存在）

### 7.10 虚拟机迁移后安全组同步

**场景**：虚拟机从宿主机 A 迁移到宿主机 B，需要重新应用安全组规则。

**处理**：
1. 虚拟机迁移完成后，计算模块通知网络模块
2. 网络模块查询虚拟机绑定的所有安全组
3. 生成完整 iptables 规则集
4. 通过 agent 下发到新宿主机 B
5. 确认规则生效后，清理旧宿主机 A 上的残留规则
6. 如果新宿主机规则下发失败，虚拟机网络保持隔离状态（默认 DROP）

### 7.11 监听器端口冲突

**场景**：用户尝试在同一 CLB 上创建两个监听器都使用 80 端口。

**处理**：
1. 创建时前端实时校验，提示"端口 80 已被监听器 http-listener-1 使用"
2. 后端 API 也做重复校验（防止绕过前端）
3. 不同 CLB 实例之间端口可以重复（VIP 不同）

### 7.12 DHCP 服务异常

**场景**：负责 DHCP 的 dnsmasq 进程意外退出。

**处理**：
1. 网络模块 agent 定期检测 dnsmasq 进程存活状态（每 10 秒）
2. 检测到进程退出后，自动尝试重启 dnsmasq
3. 连续 3 次重启失败，标记子网 DHCP 状态为"异常"，触发告警
4. 已获取租约的虚拟机不受影响（租约未到期），新虚拟机无法自动获取 IP
5. 提供"手动重启 DHCP"按钮

### 7.13 CLB 配置语法错误

**场景**：生成 Nginx/HAProxy 配置时因数据异常导致语法错误，reload 失败。

**处理**：
1. 系统在 reload 前执行 `nginx -t` / `haproxy -c` 配置语法检查
2. 检查失败则回滚到上一个有效配置，CLB 继续运行
3. 记录错误日志，包含错误配置内容（脱敏后）
4. 触发告警并暂停后续配置变更，等待人工介入
5. CLB 状态标记为"配置异常"

## 附录 B：技术实现要点

| 编号 | 要点 | 说明 |
|------|------|------|
| TI-01 | Linux Bridge 命名规范 | `vn-{network_id前8位}`，避免与现有 Bridge 冲突。NAT/隔离网络 Bridge 绑定到 VLAN 子接口 `eth0.{vlan_id}` |
| TI-02 | iptables chain 命名 | `SG-{sg_id前8位}-{ingress/egress}`，虚拟机网卡对应独立 chain |
| TI-03 | 安全组规则原子下发 | 先写入临时 chain，验证通过后替换原 chain，失败则回滚 |
| TI-04 | Nginx/HAProxy 配置模板 | 使用模板引擎生成配置，语法检查通过后 graceful reload |
| TI-05 | VIP 分配策略 | 从子网 CIDR 的可用 IP 池中顺序分配，支持预留 IP 段 |
| TI-06 | DHCP 静态绑定 | 通过 dnsmasq 的 `--dhcp-host=MAC,IP` 参数实现 |
| TI-07 | 健康检查实现 | HAProxy 模式下使用内置健康检查；Nginx 模式下使用 nginx-upstream-check 模块 |
| TI-08 | SSL 证书存储 | 私钥使用 AES-256-GCM 加密后存储，解密仅在生成 Nginx 配置时进行 |
| TI-09 | 跨宿主机通信 | agent 通过 gRPC 与各宿主机通信，支持 TLS 双向认证。跨宿主机二层互通采用物理交换机 VLAN Trunk（一期）或 VXLAN Overlay（二期） |
| TI-10 | 数据库选型 | PostgreSQL（支持 JSON 字段、事务、乐观锁） |
| TI-11 | VLAN ID 分配 | 全局 VLAN ID 池（2-4094），按需分配，删除网络时回收，利用率超 80% 告警 |
