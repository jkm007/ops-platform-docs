# 运维内网管理系统 — 需求规划文档

> 版本：V2.0 | 日期：2026-07-14 | 状态：需求规划阶段（已与用户对齐）

---

## 一、项目背景

### 1.1 业务背景

企业内部运维团队目前通过命令行、独立开源工具和脚本分散管理物理机、KVM 虚拟机、Ceph 集群、Jenkins、GitLab、Docker、Nginx、MySQL、Redis、Kubernetes 等十余种基础设施组件。随着业务规模扩大，这种"各自为政"的管理方式暴露出以下核心矛盾：

- **入口分散**：管理不同组件需要切换多个工具/终端，运维效率低
- **信息孤岛**：物理机、虚拟机、容器、中间件的资源和状态无法统一呈现
- **操作门槛高**：运维操作依赖命令行经验，新人上手慢、误操作风险大
- **权限失控**：缺乏统一的 RBAC 权限体系，操作不可审计、不可追溯
- **缺乏自助化**：开发/测试团队无法自助创建虚拟机、申请存储/数据库资源，全靠运维人工处理

### 1.2 目标愿景

打造一个**企业级运维内网统一管理平台**，以"类公有云（阿里云/腾讯云）控制台"的交互体验，对内网全部基础设施资源实现**统一纳管、可视化管理、自助化交付、自动化运维**。

### 2.2 核心差距分析

现有开源方案存在一个共同问题：**要么偏向IaaS虚拟化（CloudStack/Proxmox），要么偏向监控运维（OMP/OPSONE），没有方案同时覆盖"资源创建交付"和"运维管理"两个核心场景**。

本项目需要填补的空白：
- 像阿里云ECS一样创建管理KVM虚拟机（而不仅仅是纳管已有VM）
- 像阿里云OSS一样管理Ceph对象存储（而不仅仅是监控Ceph状态）
- 像阿里云CLB一样管理Nginx/HAProxy负载均衡
- 像阿里云VPC一样管理网络与安全组
- 同时还要纳管物理机、Docker、K8s、Jenkins、GitLab、MySQL、Redis 等 DevOps 基础设施

## 三、用户画像与核心场景

### 3.1 用户角色

| 角色 | 核心诉求 | 使用频率 |
|------|---------|---------|
| **运维工程师** | 资源统一管理、快速定位问题、批量操作、自动化运维 | 每天 |
| **开发工程师** | 自助创建虚拟机、申请数据库/存储、查看服务状态 | 按需 |
| **测试工程师** | 快速创建测试环境、克隆虚拟机、申请中间件实例 | 按需 |
| **运维管理者** | 资源概览、成本分析、权限审批、操作审计 | 每周 |
| **安全审计员** | 查看操作日志、审计合规 | 按需 |

## 五、技术架构建议

### 5.1 技术选型建议

| 层级 | 推荐方案 | 备选方案 | 理由 |
|------|---------|---------|------|

## 八、扩展性分析与架构保障

### 8.1 核心原则：只读纳管 + 不影响存量

基于"不可影响现有 KVM/Ceph/K8s"这一硬约束，本平台设计遵循以下原则：

> **"只读纳管优先、写操作需主动触发、零侵入式接入"**

具体保障措施：

| 存量系统 | 接入方式 | 风险等级 | 保障策略 |
|---------|---------|---------|---------|
| **KVM (virsh/libvirt)** | 通过 libvirt API **只读连接**现有 qemu:///system | 🟢 低 | 不修改任何 VM 定义，不创建新的 virsh 连接池，仅读取状态信息 |
| **Ceph 集群** | 通过 Ceph RESTful API (mgr/dashboard) 或 `ceph` 命令行 **只读查询** | 🟢 低 | 不修改 ceph.conf，不操作 OSD/PG，仅读取集群状态和性能指标 |
| **K8s 集群** | 通过 kubeconfig **只读 ServiceAccount** 接入 | 🟢 低 | 使用只读 RBAC 角色，仅 list/get/watch，不执行 create/update/delete |

> ⚠️ **关键约束**：新平台创建的资源（如通过平台新建的 KVM 虚拟机、Ceph RBD 云盘等）可以与存量资源共存，但平台**不会**自动接管或修改已有资源的配置。

### 8.2 架构可扩展性设计

#### 8.2.1 插件化资源驱动架构

平台采用 **"内核 + 资源驱动插件"** 的架构模式，每种资源类型（KVM、Ceph、K8s、MySQL 等）作为一个独立的驱动插件接入：

```
┌─────────────────────────────────────────────┐
│              统一 Portal (React)              │
├─────────────────────────────────────────────┤
│              统一 API Gateway                 │
├─────────────────────────────────────────────┤
│          资源管理内核 (Resource Kernel)         │
│  ┌─────────────────────────────────────┐    │
│  │  资源注册  │ 状态机  │ 权限  │ 审计  │    │
│  └─────────────────────────────────────┘    │
├─────────────────────────────────────────────┤
│           资源驱动插件层 (Driver Plugin)        │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐       │
│  │ KVM  │ │ Ceph │ │ K8s  │ │MySQL │  ...  │
│  │Driver│ │Driver│ │Driver│ │Driver│       │
│  └──────┘ └──────┘ └──────┘ └──────┘       │
└─────────────────────────────────────────────┘
```

#### 8.2.2 驱动插件接口规范

每个资源驱动需要实现统一接口（Go interface）：

```go
// 资源驱动标准接口
type ResourceDriver interface {
    // 元信息
    ResourceType() string           // 资源类型: "kvm", "ceph", "k8s", "mysql"...
    Version() string                // 驱动版本
    
    // 生命周期
    Connect(config DriverConfig) error  // 连接资源后端
    Disconnect() error                  // 断开连接
    HealthCheck() error                 // 健康检查
    
    // CRUD 操作（按需实现）
    List(ctx context.Context, query Query) ([]Resource, error)
    Get(ctx context.Context, id string) (Resource, error)
    Create(ctx context.Context, spec ResourceSpec) (Resource, error)  // 可选
    Update(ctx context.Context, id string, spec ResourceSpec) error   // 可选
    Delete(ctx context.Context, id string) error                      // 可选
    
    // 操作
    SupportedActions() []string      // 支持的操作: ["start","stop","restart"...]
    ExecuteAction(ctx context.Context, id string, action string, params map[string]interface{}) error
    
    // 监控
    CollectMetrics(ctx context.Context, id string) ([]Metric, error)
}
```

#### 8.2.3 未来扩展场景

| 扩展场景 | 实现方式 | 工作量估算 |
|---------|---------|-----------|
| 新增 VMware 虚拟机管理 | 开发 `vmware-driver` 插件，对接 vCenter API | 3-4周 |
| 新增 PostgreSQL/ElasticSearch 管理 | 开发 `postgres-driver` / `es-driver`，复用中间件管理框架 | 1-2周/个 |

### 8.4 技术架构扩展性保障

| 维度 | 设计决策 | 扩展性收益 |
|------|---------|-----------|
| **后端微服务化** | Go 服务按领域拆分（compute-service/storage-service/network-service...），gRPC通信 | 可独立部署、独立扩缩、独立升级 |
| **前端微前端** | 使用 Module Federation 或 qiankun，每个模块独立打包部署 | 模块独立开发、独立发布 |
| **消息驱动** | 异步任务通过消息队列解耦（NATS/RabbitMQ） | 新增消费者无需修改生产者 |
| **配置中心** | 驱动配置、租户配置统一管理 | 新增驱动只需配置，无需改代码 |
| **API版本化** | `/api/v1/...`、`/api/v2/...` | 平滑升级，向后兼容 |

### 8.5 与现有开源方案的关系

| 方案 | 与本项目关系 | 策略 |
|------|------------|------|
| **Cloudpods** | 参考其多云纳管架构和驱动设计 | 借鉴其资源抽象层设计思路，但本项目聚焦内网运维全栈管理 |
| **OMP (云智慧)** | 参考其应用商店和监控集成 | 借鉴组件管理思想 |
| **OPSONE** | 参考其轻量化Agent和巡检机制 | 借鉴 Agent 通信设计 |
| **Ceph Dashboard** | Ceph管理可嵌入或链接 | 一期可直接嵌入Dashboard iframe，二期自研管理界面 |
| **K8s Dashboard / Lens** | K8s管理参考 | 一期嵌入，二期自研 |

> 💡 **关键决策**：本项目不走"基于CloudStack/OpenStack二次开发"的路线，而是**自研轻量内核 + 插件化驱动**。原因：
> 1. CloudStack/OpenStack 太重，部署和维护成本高
> 2. 无法灵活纳管 DevOps 工具链（Jenkins/GitLab/Nginx）
> 3. 与存量 KVM/Ceph/K8s 共存困难，侵入性大
> 4. 自研架构更容易做定制化扩展
