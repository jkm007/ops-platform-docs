## 一、模块概述

### 1.1 模块定位

计算模块是运维内网统一管理平台的核心基础设施管理模块，负责对企业内部物理服务器和 KVM 虚拟化资源进行统一纳管、监控与生命周期管理。模块对上承接业务系统的算力需求，对下驱动物理机与虚拟化层的资源调度，是整个平台"基础设施即代码（IaC）"能力的关键组成部分。

### 1.2 与对标产品的关系

对标阿里云 ECS（Elastic Compute Service），本模块提供以下对标能力：

| 阿里云 ECS 能力 | 本模块对应能力 | 差异说明 |
|----------------|--------------|---------|
| 实例管理 | KVM 虚拟机管理 | 仅支持 KVM/Libvirt，不涉及公有云弹性伸缩 |
| 镜像管理 | 镜像管理 | 支持 ISO 上传、IMG 导入、实例制作为镜像 |
| 实例规格 | 规格管理 | 预定义规格族，不涉及突发性能实例等云原生规格 |
| VNC 管理 | Web VNC 远程连接 | 基于 noVNC 实现，不依赖第三方 VNC 代理 |
| 安全组 | 不在此模块 | 由网络模块统一管理 |
| 弹性 IP | 不在此模块 | 由网络模块统一管理 |
| 快照 | 快照管理 | 基于 Libvirt snapshot API，不支持自动快照策略 |
| 云助手 | QEMU Guest Agent | 通过 QGA 实现密码重置、文件注入等 |

本模块不追求完全对标公有云的全部功能，而是在满足企业内部运维需求的前提下，提供一致的交互体验和 API 设计风格。

## 四、详细需求描述（EARS 原则）

### 4.1 物理机纳管与监控

**F-PM-01 — 物理机列表**

- `[Ubiquitous]` 系统应始终以表格形式展示所有已纳管的物理机，每行包含：物理机名称、管理 IP、BMC IP、型号、CPU 型号/核数、总内存、运行状态、健康状态、所属机房/机架位、最后检测时间。
- `[Event-driven]` 当用户输入搜索关键词（名称、IP、型号）时，系统应在 500ms 内返回匹配的物理机列表。
- `[Event-driven]` 当用户选择筛选条件（运行状态、健康状态、机房位置）时，系统应实时过滤列表结果。
- `[State-driven]` 在物理机列表加载期间，系统应展示骨架屏占位，而非空白页面。
- `[Unwanted]` 如果后端接口超时（超过 10s 无响应），则系统应提示"加载超时，请刷新重试"并提供手动刷新按钮。

**F-PM-02 — 物理机详情**

- `[Ubiquitous]` 系统应始终将物理机详情页分为以下信息区域：基本信息（名称、管理 IP、BMC IP、MAC 地址、型号、序列号、固件版本、机房/机架位/U位）、硬件配置（CPU 型号及核数/线程数、内存总量及插槽分布、磁盘列表及容量、网卡列表及速率）、传感器数据（CPU 温度、进风口温度、风扇转速、电源状态、电压读数）、资源监控图表（CPU 使用率、内存使用率、磁盘 IOPS、网络吞吐量）。
- `[Event-driven]` 当用户进入物理机详情页时，系统应每 30 秒自动刷新传感器数据和监控图表。
- `[Event-driven]` 当用户切换监控图表时间范围（1h/6h/24h/7d）时，系统应重新请求对应粒度的历史数据。
- `[Unwanted]` 如果 BMC 不可达导致传感器数据为空，则系统应在传感器区域显示"BMC 连接异常，传感器数据不可用"并展示最后成功采集的时间戳。

**F-PM-03 — 物理机纳管**

- `[Ubiquitous]` 系统应始终要求纳管物理机时提供以下必填信息：物理机名称（唯一）、管理 IP、BMC IP、BMC 用户名、BMC 密码（加密传输）、机房位置、机架位。
- `[Event-driven]` 当用户提交纳管表单时，系统应先验证 BMC 连通性和凭据有效性，验证通过后方可保存纳管记录。
- `[Event-driven]` 当 BMC 凭据验证失败时，系统应在表单对应字段下方显示红色错误提示"BMC 连接失败：[具体错误原因]"，并阻止提交。
- `[Unwanted]` 如果纳管的物理机管理 IP 或 BMC IP 已存在于系统中，则系统应提示"该 IP 地址已纳管，请勿重复添加"并阻止提交。
- `[Event-driven]` 当纳管成功后，系统应在 3 秒后自动跳转至物理机列表页，并在列表顶部展示绿色成功提示"物理机 [名称] 纳管成功"。

**F-PM-05 — 资源监控**

- `[Ubiquitous]` 系统应始终以折线图形式展示 CPU 使用率、内存使用率、磁盘 IOPS、网络吞吐量的历史趋势，默认展示最近 1 小时数据。
- `[Ubiquitous]` 系统应始终在图表上标注告警阈值线（红色虚线），当指标超过阈值时对应区域高亮显示。
- `[State-driven]` 在监控数据加载期间，系统应在图表区域展示加载动画。
- `[Unwanted]` 如果监控数据采集服务中断导致最近 5 分钟无新数据，则系统应在图表上方展示黄色警告"监控数据可能延迟，正在重试采集"。

### 4.2 物理机远程控制

**F-PM-07/08/09 — 远程开机/关机/重启**

- `[Ubiquitous]` 系统应始终在物理机详情页的操作栏提供"开机""关机""重启"按钮，按钮状态与物理机当前状态联动。
- `[Ubiquitous]` 系统应始终在执行远程控制操作前弹出二次确认对话框，明确告知操作对象和影响范围。
- `[Event-driven]` 当用户点击"关机"按钮时，系统应提供两个选项："优雅关机（推荐）"和"强制关机"。选择优雅关机时，系统应等待操作系统正常关闭，超时 120 秒后自动转为强制关机。
- `[Event-driven]` 当远程控制指令通过 IPMI 发送成功后，系统应在页面顶部展示操作进度提示"正在 [开机/关机/重启] 物理机 [名称]..."，并每 5 秒轮询物理机状态直至操作完成或超时。
- `[Unwanted]` 如果 IPMI 指令执行失败（返回非零状态码或超时），则系统应展示红色错误提示"[开机/关机/重启] 失败：[IPMI 返回的错误信息]"，并提供"重试"按钮。
- `[Unwanted]` 如果物理机当前状态与目标操作冲突（如已开机状态下点击开机），则系统应禁用对应按钮并提示"当前状态不允许此操作"。
- `[State-driven]` 在远程控制操作进行期间（轮询状态中），系统应禁用所有远程控制按钮，防止重复操作。

**F-PM-10 — KVM Console**

- `[Event-driven]` 当用户点击"KVM Console"按钮时，系统应通过 IPMI/Redfish 协议获取 KVM 会话 URL，并在新浏览器标签页中打开 noVNC 连接。
- `[Ubiquitous]` 系统应始终在 KVM Console 页面提供以下功能按钮：发送 Ctrl+Alt+Del、全屏切换、剪贴板同步（可选）。
- `[Unwanted]` 如果 KVM 会话建立失败（BMC 不支持或网络不通），则系统应提示"KVM Console 连接失败：[原因]"并建议检查 BMC 配置。
- `[Unwanted]` 如果 KVM 会话在 5 分钟内无任何交互操作，则系统应自动断开连接并在页面显示"会话已超时，请重新连接"。

### 4.3 虚拟机创建全流程

**F-VM-02 — 创建虚拟机（向导式流程）**

创建流程为 **7 步向导**：选择宿主机 → 选择镜像 → cloud-init 配置 → 选择规格 → **选择存储类型** → 选择网络 → 确认创建。

**步骤一：选择宿主机**

- `[Ubiquitous]` 系统应始终以列表形式展示所有可用的宿主机节点，每行包含：宿主机名称、CPU 总量/已分配/可用、内存总量/已分配/可用、本地磁盘总量/已用/可用、运行状态、所属资源池。
- `[Event-driven]` 当用户在创建向导中选择了目标租户后，系统应仅展示该租户有权限使用的宿主机资源池中的节点。
- `[Unwanted]` 如果所有宿主机资源均不满足最低规格要求（1vCPU + 1GB 内存），则系统应提示"当前资源池无可用宿主机，请联系平台管理员扩容"并阻止进入下一步。
- `[Event-driven]` 当用户选中某台宿主机后，系统应在右侧面板展示该宿主机的资源分配详情和预计创建后的资源变化预览。

**步骤二：选择镜像**

- `[Ubiquitous]` 系统应始终以卡片形式展示可用镜像，每张卡片包含：镜像名称、OS 图标、版本号、镜像类型（ISO/cloud-init模板/qcow2模板）、镜像大小、更新时间。
- `[Ubiquitous]` 系统应始终在镜像列表顶部提供筛选条件：OS 类型（Linux/Windows）、镜像类型、关键词搜索。
- `[Event-driven]` 当用户选择 ISO 类型镜像时，系统应提示"使用 ISO 镜像创建将需要手动完成操作系统安装，不支持 cloud-init"。
- `[Event-driven]` 当用户选择 cloud-init 模板镜像时，系统应在下一步显示 cloud-init 配置步骤。
- `[Event-driven]` 当用户选择 qcow2 模板镜像（无 cloud-init）时，系统应跳过 cloud-init 配置步骤，提示"该镜像不支持 cloud-init 自动初始化，创建后需手动配置"。

**步骤三：cloud-init 配置（仅 cloud-init 镜像显示）**

- `[Event-driven]` 当用户选择了 cloud-init 模板镜像时，系统应展示 cloud-init 配置表单：

| 字段 | 类型 | 说明 |
|------|------|------|
| 主机名 | 文本 | 默认自动生成 `{tenant}-{timestamp}`，支持自定义 |
| Root 密码 | 密码 | 可选；为空则仅使用 SSH Key 登录 |
| SSH 公钥 | 文本域 | 可选；支持粘贴公钥或从已保存的 Key 列表选择 |
| 自定义脚本 | 文本域 | 可选；cloud-init user-data 脚本，首次启动执行 |

- `[Event-driven]` 当用户提交创建时，系统应：
  1. 根据表单生成 cloud-init NoCloud ISO（含 meta-data + user-data + vendor-data）
  2. 将 ISO 作为虚拟光驱挂载到虚拟机
  3. 首次启动后 cloud-init 自动执行初始化，完成后弹出光驱
- `[Unwanted]` 如果用户既未设置密码也未配置 SSH Key，则系统应提示"至少需要设置 Root 密码或配置 SSH 公钥，否则创建后将无法登录"。
- `[Ubiquitous]` 系统应始终支持用户保存常用的 cloud-init 配置为模板（如"Web服务器模板"），下次创建时一键加载。

**步骤四：选择规格**

- `[Ubiquitous]` 系统应始终以表格形式展示可用规格，每行包含：规格名称、规格族（通用型/计算型/内存型）、vCPU 数量、内存大小、适用场景描述。
- `[Event-driven]` 当用户在规格列表中切换筛选规格族时，系统应实时过滤展示对应规格族的规格。
- `[Unwanted]` 如果所选规格的资源需求超过当前选中宿主机的可用资源（考虑超分比），则系统应在规格行显示红色标记"资源不足"并禁用该规格的选中操作。
- `[Unwanted]` 资源超分策略：CPU 默认 2:1 超分（混部环境，需预留 Ceph/K8s 基线），内存不超分（1:1）。平台管理员可在资源池设置中调整超分比。

**步骤五：选择存储类型（新增步骤）**

- `[Ubiquitous]` 系统应始终在存储配置步骤展示以下选项：

| 存储类型 | 底层实现 | 性能特点 | 迁移支持 | 适用场景 |
|---------|---------|---------|---------|---------|
| **本地存储（qcow2）** | `/home/kvm/images/` 本地 qcow2 文件 | 高性能、低延迟 | 不支持在线迁移 | 数据库、高性能计算 |
| **分布式存储（Ceph RBD）** | Ceph RBD 块设备 | 高可用、可迁移 | 支持在线/离线迁移 | 普通应用、需高可用的服务 |

- `[Event-driven]` 当用户选择"本地存储"时，系统应：
  1. 在目标宿主机 `/home/kvm/images/` 目录下，通过 `qemu-img create -f qcow2 -b {镜像路径} {vm磁盘路径}` 创建 backing file
  2. 展示系统盘大小配置（默认 40GB，范围 20GB-2TB）
  3. 展示当前宿主机 `/home/kvm/images/` 可用空间
- `[Event-driven]` 当用户选择"分布式存储（Ceph RBD）"时，系统应：
  1. 从 Ceph 存储池创建 RBD 镜像
  2. 展示可用的 Ceph 存储池列表和容量
  3. 展示系统盘大小配置（默认 40GB，范围 20GB-2TB）
- `[Unwanted]` 如果选择本地存储但宿主机 `/home/kvm/images/` 可用空间不足（< 所需磁盘大小 × 1.2），则系统应提示"宿主机本地存储空间不足（可用 X GB，需要 Y GB），请选择其他宿主机或切换为 Ceph RBD 存储"。
- `[Unwanted]` 如果选择 Ceph RBD 但存储池可用空间不足，则系统应提示"Ceph 存储池 [名称] 空间不足，请联系存储管理员扩容"。
- `[Event-driven]` 当用户需要额外数据盘时，系统应支持添加 1-10 块数据盘，每块可独立选择存储类型和大小。

**步骤六：选择网络**

- `[Ubiquitous]` 系统应始终以表格形式展示可用网络，每行包含：网络名称、VLAN ID、子网 CIDR、网关、可用 IP 数量、网络类型（NAT/隔离/桥接）。
- `[Event-driven]` 当用户选择网络后，系统应展示该网络中可用 IP 范围预览。
- `[Ubiquitous]` 系统应始终支持以下 IP 分配方式：DHCP 自动分配（默认）、手动指定 IP 地址。
- `[Unwanted]` 如果用户选择手动指定 IP 地址，但输入的 IP 不在子网范围内或已被占用，则系统应在输入框下方显示红色错误提示"[IP 地址] 不可用：[原因]"。
- `[Event-driven]` 当用户选择多块网卡时（最多 4 块），系统应自动为每块网卡分配序号（eth0/eth1...）并支持选择不同网络。

**步骤七：确认创建**

- `[Ubiquitous]` 系统应始终在确认页以汇总卡片形式展示所有配置信息：虚拟机名称、所属租户、宿主机、镜像、cloud-init 配置、规格（vCPU/内存）、**存储类型及磁盘配置**、网络配置。
- `[Ubiquitous]` 系统应始终在确认页提供虚拟机名称输入框，支持用户自定义名称（2-64 字符，字母数字及-_.），默认生成格式为 `{tenant}-{timestamp}`。
- `[Event-driven]` 当用户点击"确认创建"按钮时，系统应：
  1. 前端校验所有配置项完整性
  2. 后端再次校验资源可用性（防止并发创建导致超卖）
  3. 如果使用 cloud-init 镜像：生成 NoCloud ISO → 创建磁盘（qcow2 或 RBD）→ 挂载 ISO → 启动 VM → cloud-init 执行 → 卸载 ISO
  4. 如果使用非 cloud-init 镜像：创建磁盘 → 启动 VM
  5. 校验通过后提交创建任务，在列表页顶部展示"虚拟机创建任务已提交，预计 3-5 分钟完成"
  6. 自动跳转至虚拟机列表页，新创建的虚拟机显示"创建中"状态
- `[Unwanted]` 如果后端资源校验失败（并发创建导致资源不足），则系统应回滚创建流程并提示"创建失败：宿主机 [名称] 资源不足，请重新选择宿主机"。
- `[Unwanted]` 如果虚拟机创建任务在 15 分钟后仍未完成，则系统应将虚拟机状态标记为"创建失败"并展示失败原因（含详细日志）。
- `[Unwanted]` 如果用户在确认页面停留超过 30 分钟未操作，则系统应提示"会话即将过期，请尽快确认或刷新重试"，并在 5 分钟后自动跳转至列表页。

### 4.4 虚拟机生命周期操作

**F-VM-04/05/06/07 — 启动/关机/强制关机/重启**

- `[Ubiquitous]` 系统应始终在虚拟机列表每行操作列和详情页操作栏提供状态关联的操作按钮。
- `[Ubiquitous]` 系统应始终在以下操作前弹出二次确认对话框：强制关机、强制重启、删除。
- `[Event-driven]` 当用户点击"关机"按钮（运行中状态下的虚拟机）时，系统应通过 libvirt API 发送 ACPI shutdown 信号，等待 60 秒，若虚拟机未响应则提示用户可选择强制关机。
- `[Event-driven]` 当用户点击"强制关机"按钮时，系统应通过 libvirt API 执行 `virDomainDestroy`（等价于拔电源），并弹出警告"强制关机可能导致数据丢失，确定继续？"。
- `[Event-driven]` 当用户点击"重启"按钮（运行中状态下的虚拟机）时，系统应提供两个选项："优雅重启（推荐）"和"强制重启"。优雅重启通过 ACPI 实现，超时 90 秒后提示可转为强制重启。
- `[Unwanted]` 如果虚拟机在关机/重启操作后 120 秒内未达到目标状态，则系统应标记操作为"超时"并提示用户手动检查虚拟机状态。
- `[State-driven]` 在生命周期操作执行期间，系统应禁用该虚拟机的所有操作按钮并在状态列显示转圈动画和操作进度文字（如"关机中..."）。

**F-VM-08/09 — 暂停/恢复**

- `[Event-driven]` 当用户点击"暂停"按钮（运行中状态下的虚拟机）时，系统应通过 libvirt API 执行 `virDomainSuspend`，将虚拟机内存状态写入磁盘并停止 CPU 调度。
- `[Event-driven]` 当用户点击"恢复"按钮（暂停状态下的虚拟机）时，系统应通过 libvirt API 执行 `virDomainResume`，从磁盘恢复内存状态并恢复 CPU 调度。
- `[Ubiquitous]` 系统应始终在虚拟机暂停状态持续超过 24 小时后，在虚拟机列表行展示黄色标记"长时间暂停"，提醒用户关注。

**F-VM-10 — 删除**

- `[Ubiquitous]` 系统应始终在删除操作前弹出确认对话框，包含以下内容：
  - 提示文字"确定要删除虚拟机 [名称] 吗？此操作不可恢复。"
  - 复选框"同时删除关联的磁盘（系统盘 + 数据盘）"（默认不勾选）
  - 复选框"同时删除关联的快照"（默认不勾选）
  - 输入框"请输入虚拟机名称以确认删除"
- `[Event-driven]` 当用户输入虚拟机名称与待删除虚拟机名称匹配并点击确认后，系统应执行删除操作。
- `[Unwanted]` 如果虚拟机当前处于"运行中"状态，则系统应在确认对话框中额外显示红色警告"虚拟机正在运行中，删除前将自动强制关机，可能导致数据丢失"。
- `[Event-driven]` 当删除操作完成后，系统应在列表页展示绿色成功提示"虚拟机 [名称] 已删除"，同时从列表中移除该行（或标记为已删除状态保留 24 小时）。

### 4.5 Web VNC 远程连接

**F-VM-11 — Web VNC 连接**

- `[Event-driven]` 当用户点击"VNC 连接"按钮（运行中或暂停状态下的虚拟机）时，系统应通过 libvirt API 获取 VNC 端口号和密码（一次性密码），并在新浏览器标签页中打开 noVNC 客户端。
- `[Ubiquitous]` 系统应始终为每次 VNC 连接生成一次性随机密码，连接关闭后密码立即失效。
- `[Ubiquitous]` 系统应始终在 noVNC 页面提供以下功能：发送 Ctrl+Alt+Del、全屏切换、分辨率自适应。
- `[Unwanted]` 如果 VNC 连接建立后 30 分钟内无任何用户输入（键盘/鼠标事件），则系统应自动断开连接并在页面显示"连接已因超时关闭，请重新连接"。
- `[Unwanted]` 如果 noVNC WebSocket 连接断开（非用户主动关闭），则系统应在页面中央显示"连接已断开，正在重试..."并自动重试最多 3 次，每次间隔 2 秒，3 次失败后提示"VNC 连接失败，请检查虚拟机状态后重试"。
- `[Unwanted]` 如果虚拟机未安装 VNC 显卡驱动或 VNC 服务未启用，则系统应提示"该虚拟机不支持 VNC 连接，请检查虚拟机配置"。

### 4.6 重置密码

**F-VM-12 — 重置密码**

- `[Ubiquitous]` 系统应始终在虚拟机详情页提供"重置密码"功能入口，入口仅在虚拟机运行中且安装了 QEMU Guest Agent 时可用。
- `[Event-driven]` 当用户点击"重置密码"时，系统应弹出对话框包含以下内容：
  - 说明文字"将通过 QEMU Guest Agent 重置操作系统密码，此操作需要虚拟机内已安装并运行 QEMU Guest Agent。"
  - 目标用户选择（Linux: root / Windows: Administrator / 自定义用户名）
  - 新密码输入框（至少 8 位，含大小写字母、数字、特殊字符中至少三种）
  - 确认密码输入框
- `[Event-driven]` 当用户提交重置密码请求后，系统应通过 libvirt API 调用 QEMU Guest Agent 的 `guest-set-user-password` 命令。
- `[Unwanted]` 如果 QEMU Guest Agent 未响应（30 秒超时），则系统应提示"重置密码失败：QEMU Guest Agent 未响应。请确认虚拟机操作系统内已安装并运行 qemu-guest-agent 服务"。
- `[Unwanted]` 如果新密码不满足复杂度要求，则系统应在输入框下方显示"密码需至少 8 位，且包含大小写字母、数字、特殊字符中至少三种"。
- `[Event-driven]` 当密码重置成功后，系统应展示绿色成功提示"密码重置成功，请妥善保管新密码"，并在 10 秒后自动关闭对话框，不展示明文密码。

### 4.7 快照创建与恢复

**F-VM-13 — 创建快照**

- `[Ubiquitous]` 系统应始终在虚拟机详情页的"快照"标签页提供"创建快照"按钮。
- `[Event-driven]` 当用户点击"创建快照"时，系统应弹出对话框包含：
  - 快照名称输入框（2-64 字符，默认格式：`snap-{YYYYMMDD-HHmmss}`）
  - 快照描述输入框（选填，最多 200 字符）
  - 提示"快照创建期间虚拟机将短暂暂停 IO，建议在业务低峰期操作"
- `[Event-driven]` 当用户确认创建后，系统应根据存储类型执行不同快照策略：
  - **本地 qcow2**：通过 `virDomainSnapshotCreateXML` 创建 libvirt 快照，快照文件存储在 `/home/kvm/kvm_snap/{vm-name}/` 目录下
  - **Ceph RBD**：通过 `rbd snap create` 创建 RBD 快照
- `[Unwanted]` 如果本地 qcow2 快照创建时宿主机 `/home/kvm/kvm_snap/` 空间不足，则系统应提示"快照失败：宿主机快照存储空间不足，请清理旧快照后重试"。
- `[Unwanted]` 如果虚拟机无关联磁盘（仅 ISO 启动），则系统应提示"无法创建快照：虚拟机无关联磁盘"。
- `[Ubiquitous]` 系统应始终在快照列表中展示每份快照的存储类型标识（本地/Ceph）。

**F-VM-14 — 快照恢复**

- `[Ubiquitous]` 系统应始终在快照恢复前要求虚拟机处于"已停止"状态。
- `[Event-driven]` 当用户对运行中的虚拟机执行快照恢复时，系统应提示"恢复快照需要先关闭虚拟机，是否立即关机并恢复？"。
- `[Ubiquitous]` 系统应始终在快照恢复前弹出二次确认对话框。
- `[Event-driven]` 当用户确认恢复后，系统应根据存储类型执行：
  - **本地 qcow2**：`virDomainSnapshotRevert`
  - **Ceph RBD**：`rbd snap rollback`
- `[Unwanted]` 如果快照文件损坏或丢失，则系统应提示"恢复快照失败：快照文件不可用"。
- `[Unwanted]` 如果 Ceph RBD 快照被其他客户端锁定，则系统应提示"Ceph RBD 快照正被其他操作占用，请稍后重试"。

**F-VM-15 — 快照删除**

- `[Event-driven]` 当用户点击"删除快照"时，系统应弹出确认对话框。
- `[Event-driven]` 当用户确认后，系统应根据存储类型删除快照。
- `[Unwanted]` 如果本地 qcow2 快照存在子快照（backing chain 中间节点），则系统应提示"该快照存在依赖的子快照，删除后需合并数据（blockcommit），可能耗时较长，确定继续？"。

### 4.8 实例克隆

**F-VM-16 — 实例克隆**

- `[Ubiquitous]` 系统应始终要求被克隆的虚拟机处于"已停止"状态。
- `[Event-driven]` 当用户点击"克隆"按钮时，系统应弹出克隆配置对话框包含：
  - 新虚拟机名称（必填，2-64 字符）
  - 目标宿主机选择（默认与源虚拟机相同）
  - 目标存储类型（本地 qcow2 / Ceph RBD，默认与源 VM 相同）
  - 网络配置（默认与源虚拟机相同，支持修改）
- `[Event-driven]` 当用户确认克隆后，系统应根据存储类型执行：
  - **本地 qcow2 → 本地 qcow2**：`qemu-img create -f qcow2 -b {源磁盘} {新磁盘}` 创建 backing file，然后 `virsh define` 新 domain
  - **本地 qcow2 → Ceph RBD**：`qemu-img convert {源qcow2} rbd:{pool}/{image}` 转换格式后 define
  - **Ceph RBD → Ceph RBD**：`rbd cp {源image} {新image}` 然后 define
  - **Ceph RBD → 本地 qcow2**：`qemu-img convert rbd:{pool}/{image} {目标qcow2}` 转换后 define
  1. 在目标宿主机上复制源虚拟机磁盘镜像
  2. 基于源虚拟机 XML 定义创建新虚拟机 XML（替换 UUID、MAC 地址、名称）
  3. 注册新虚拟机到 libvirt
- `[Unwanted]` 如果目标宿主机可用磁盘空间小于源虚拟机磁盘总大小，则系统应提示"克隆失败：目标宿主机磁盘空间不足，需要 [X] GiB，可用 [Y] GiB"。
- `[Unwanted]` 如果克隆过程中磁盘复制失败，则系统应清理已复制的部分文件并提示"克隆失败：[错误原因]，已清理残留文件"。
- `[Event-driven]` 当克隆完成后，系统应在虚拟机列表页展示新创建的虚拟机（状态为"已停止"），并在页面顶部展示成功提示。

### 4.9 镜像管理

**F-IMG-02 — 上传 ISO 镜像**

- `[Ubiquitous]` 系统应始终支持以下 ISO 上传方式：本地文件上传（浏览器）、从远端 URL 拉取（服务器端下载）。
- `[Ubiquitous]` 系统应始终在上传前校验文件扩展名为 `.iso`，单文件大小不超过 10GB。
- `[Event-driven]` 当用户通过浏览器上传时，系统应展示上传进度条（百分比和速度），并支持断点续传。
- `[Unwanted]` 如果上传过程中网络中断，则系统应保留已上传的分片并在网络恢复后自动续传。
- `[Unwanted]` 如果上传的文件格式不是 ISO（通过文件头魔数校验），则系统应提示"文件格式校验失败：仅支持 ISO 9660 格式的镜像文件"。
- `[Event-driven]` 当上传完成后，系统应将镜像文件存储到 CephFS 镜像存储池，并在镜像列表中添加新记录。

**F-IMG-03 — 导入 IMG 镜像**

- `[Ubiquitous]` 系统应始终支持以下 IMG 格式：qcow2、raw。
- `[Event-driven]` 当用户导入 qcow2/raw 镜像时，系统应提供"导入为系统盘模板"和"导入为数据盘模板"两种选项。
- `[Event-driven]` 当导入完成后，系统应自动检测镜像中的操作系统信息（通过 libguestfs），并填充 OS 类型和版本字段。
- `[Unwanted]` 如果操作系统检测失败，则系统应将 OS 类型标记为"未知"，并提示用户手动选择。

**F-IMG-05 — 删除镜像**

- `[Unwanted]` 如果镜像正在被虚拟机使用（系统盘或数据盘引用），则系统应提示"无法删除：该镜像正被 [N] 台虚拟机使用，请先删除相关虚拟机或更换镜像"并阻止删除操作。
- `[Event-driven]` 当用户确认删除未使用的镜像时，系统应从 CephFS 存储中删除镜像文件并从数据库中删除镜像记录。

**F-IMG-06 — 实例制镜像**

- `[Ubiquitous]` 系统应始终要求被制镜像的虚拟机处于"已停止"状态。
- `[Event-driven]` 当用户点击"制作为镜像"时，系统应弹出对话框包含：
  - 镜像名称（必填）
  - 镜像描述（选填）
  - 提示"将虚拟机系统盘制作为镜像模板，制作期间虚拟机不可用"
- `[Event-driven]` 当用户确认后，系统应复制虚拟机系统盘到镜像存储池，完成后自动将 OS 信息和已安装软件信息写入镜像元数据。
- `[Unwanted]` 如果制作过程中磁盘空间不足，则系统应提示"制作镜像失败：镜像存储池空间不足，需要 [X] GiB，可用 [Y] GiB"。

### 4.10 规格管理

**F-SPEC-01/02 — 规格列表与创建**

- `[Ubiquitous]` 系统应始终预置以下默认规格族：
  - 通用型（general）：vCPU:内存 = 1:4，适用于 Web 服务、开发测试
  - 计算型（compute）：vCPU:内存 = 1:2，适用于批处理、科学计算
  - 内存型（memory）：vCPU:内存 = 1:8，适用于数据库、缓存服务
- `[Event-driven]` 当平台管理员创建新规格时，系统应校验 vCPU 和内存值为正整数，且内存大小（GB）为 2 的整数倍。
- `[Unwanted]` 如果创建规格时 vCPU 或内存值超出宿主机物理上限，则系统应提示"规格超出宿主机最大配置 [X vCPU / Y GB]，请调整"。

**F-SPEC-04 — 删除规格**

- `[Unwanted]` 如果规格正被虚拟机使用，则系统应提示"无法删除：该规格正被 [N] 台虚拟机使用，请先变更虚拟机规格或删除虚拟机"并阻止删除操作。

### 4.11 宿主机资源池管理

**F-POOL-01 — 资源池列表**

- `[Ubiquitous]` 系统应始终以汇总视图展示所有宿主机资源池，包含：宿主机数量、CPU 总核数/已分配/可用、内存总量/已分配/可用、磁盘总量/已用/可用、运行中 VM 数量。
- `[Event-driven]` 当用户选择按租户筛选时，系统应展示该租户在各宿主机的资源占用明细。

**F-POOL-03 — 超分比设置**

- `[Ubiquitous]` 系统应始终默认 CPU 超分比为 4:1，内存超分比为 1:1（不超分内存）。
- `[Event-driven]` 当平台管理员修改超分比时，系统应重新计算可用资源并更新展示。
- `[Unwanted]` 如果超分比设置导致已分配的虚拟资源总量超过（物理资源 × 超分比），则系统应提示警告但不阻止设置，仅在资源池视图中将超分配置标记为红色。

**F-POOL-04 — 资源预警**

- `[Event-driven]` 当宿主机 CPU 或内存可用率低于 20% 时，系统应自动在资源池视图将该宿主机行标记为黄色"资源紧张"。
- `[Event-driven]` 当宿主机 CPU 或内存可用率低于 5% 时，系统应自动在资源池视图将该宿主机行标记为红色"资源不足"，并向平台管理员发送告警通知。
- `[Unwanted]` 如果宿主机离线（agent 心跳超过 60 秒未上报），则系统应将该宿主机标记为灰色"离线"状态，并暂停在该宿主机上创建新虚拟机。

### 4.12 存量 VM 纳管

**F-VM-18 — 存量纳管**

- `[Ubiquitous]` 系统应始终以只读模式纳管存量 KVM 虚拟机，不修改现有 libvirt domain XML 定义，不触发任何磁盘写入操作。
- `[Event-driven]` 当平台管理员在宿主机详情页点击"同步存量虚拟机"时，系统应通过 libvirt API 执行 `virConnectListAllDomains` 列出该宿主机上所有已定义的虚拟机，与平台数据库比对后，将未纳管的虚拟机以"只读纳管"标记加入平台。
- `[Ubiquitous]` 系统应始终在只读纳管的虚拟机列表行展示"只读"标签，并限制以下操作：不可修改配置、不可删除、不可克隆、不可重置密码、不可变更规格。
- `[Ubiquitous]` 系统应始终允许对只读纳管的虚拟机执行以下只读操作：查看详情、查看监控、Web VNC 连接（需虚拟机支持）、查看快照列表。
- `[Ubiquitous]` 系统应始终允许对只读纳管的虚拟机执行以下生命周期操作：开机、关机、重启、暂停、恢复（与普通 VM 一致）。
- `[Event-driven]` 当用户尝试对只读纳管虚拟机执行禁止的操作时，系统应提示"该虚拟机为存量纳管（只读），不支持此操作。如需完整管理能力，请通过平台重新创建虚拟机"。
- `[Event-driven]` 当平台管理员将某台只读纳管的虚拟机转为"平台管理"模式时，系统应记录该操作日志并发送通知给相关租户管理员，转换后不再限制任何操作。

## 五、流程说明

### 5.1 创建虚拟机完整流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                        虚拟机创建向导流程                              │
├───────────┬───────────┬───────────┬───────────┬───────────────────────┤
│  选择宿主机 │  选择镜像  │  选择规格  │  选择网络  │      确认创建          │
│  (Step 1) │  (Step 2) │  (Step 3) │  (Step 4) │      (Step 5)         │
└───────────┴───────────┴───────────┴───────────┴───────────────────────┘
```

**关键步骤说明：**

1. **选择宿主机**：系统根据租户权限展示可用资源池。用户选择目标宿主机后，右侧实时展示资源预览（已用/总量/创建后预计）。系统自动排除资源不足、离线或维护中的宿主机。

2. **选择镜像**：展示租户可见的镜像列表。ISO 镜像和模板镜像以不同标签区分。选择 ISO 时提示需手动安装系统，选择模板时提示自动初始化。

3. **选择规格**：根据所选宿主机的剩余资源，自动过滤不可用的规格。规格行展示 vCPU、内存及与宿主机剩余资源的匹配度。

4. **选择网络**：展示租户可见的网络列表。支持单网卡或多网卡配置（最多 4 块）。IP 分配支持 DHCP 或手动指定。手动指定时前端校验 IP 格式和子网归属，后端校验 IP 是否已被占用。

5. **确认创建**：汇总展示全部配置。用户输入虚拟机名称（有默认值）。点击确认后：
   - 后端二次校验资源可用性（防并发超卖）
   - 从镜像模板克隆系统盘
   - 通过 libvirt API 创建 domain XML
   - 分配并挂载网络接口
   - 注入初始化配置（cloud-init 或 QGA）
   - 启动虚拟机
   - 更新数据库记录

**创建后自动操作（模板镜像）：**
- 设置主机名
- 配置网络 IP
- 注入 SSH 公钥（如有配置）
- 安装 QEMU Guest Agent（如镜像未预装）
- 执行用户自定义初始化脚本（如有）

### 5.2 存量 VM 纳管流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  选择宿主机    │───>│  发现存量 VM  │───>│  比对与标记   │───>│  只读纳管     │
│  (资源池页面)  │    │  (libvirt API)│    │  (自动匹配)   │    │  (入库展示)   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

**关键步骤说明：**

1. 平台管理员进入宿主机详情页，点击"同步存量虚拟机"。
2. 系统通过 libvirt API 获取该宿主机所有 domain 列表。
3. 与平台数据库中的虚拟机记录进行比对（按 UUID 匹配）：
   - 已存在于平台的：跳过
   - 不存在于平台的：标记为"存量纳管"候选
4. 系统展示比对结果，管理员确认后执行纳管：
   - 读取 domain XML 获取配置信息（vCPU、内存、磁盘、网络）
   - 写入平台数据库，标记 `managed=false`（只读纳管）
   - 关联到当前宿主机和默认租户
   - 不修改任何 libvirt domain XML
5. 纳管完成后，存量虚拟机出现在虚拟机列表中，带"只读"标签。

### 5.3 快照恢复流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  选择快照     │───>│  检查 VM 状态  │───>│  执行恢复     │───>│  恢复后处理   │
│  (快照列表)    │    │  (必须已停止)  │    │  (libvirt API)│    │  (重启/通知)  │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

**关键步骤说明：**

1. 用户进入虚拟机详情页的"快照"标签页，选择目标快照，点击"恢复"。
2. 系统检查虚拟机状态：
   - 已停止：直接进入确认步骤
   - 运行中/暂停：弹出提示"恢复快照需要关机，是否继续？"
3. 用户确认后（二次确认弹窗），系统记录恢复前状态（运行中/已停止）。
4. 系统通过 libvirt API 执行 `virDomainSnapshotRevert`。
5. 恢复完成后：
   - 如果恢复前为"运行中"：系统自动启动虚拟机
   - 如果恢复前为"已停止"：虚拟机保持停止状态
   - 系统在操作日志中记录快照恢复事件
6. 系统展示恢复成功提示。

### 6.2 关键交互规则

#### 6.2.1 状态联动规则

| 虚拟机状态 | 可用操作 | 禁用操作（灰显） |
|-----------|---------|----------------|
| 运行中 | 关机、强制关机、重启、暂停、VNC连接、重置密码、创建快照 | 开机 |
| 已停止 | 开机、删除、克隆、配置变更、制作镜像、快照恢复 | 关机、重启、暂停、VNC连接、重置密码 |
| 暂停 | 恢复、强制关机、VNC连接 | 开机、关机、重启、删除、克隆 |
| 创建中 | 无（所有操作禁用） | 全部 |
| 创建失败 | 删除 | 全部其他操作 |
| 异常 | 强制关机、强制重启、删除 | 常规操作 |
| 删除中 | 无（所有操作禁用） | 全部 |

#### 6.2.2 确认弹窗规则

| 操作 | 确认类型 | 额外条件 |
|------|---------|---------|
| 强制关机 | 二次确认 + 警告 | 提示"可能导致数据丢失" |
| 强制重启 | 二次确认 + 警告 | 提示"可能导致数据丢失" |
| 删除 | 输入名称确认 + 复选框 | 选择是否同时删除磁盘和快照 |
| 快照恢复 | 二次确认 + 警告 | 提示"当前数据将被覆盖，不可撤销" |
| 制作镜像 | 二次确认 | 提示"制作期间虚拟机不可用" |

#### 6.2.3 批量操作规则

- 批量操作仅支持同状态虚拟机（如：批量开机仅对"已停止"状态的虚拟机有效）
- 批量操作前系统自动过滤不符合条件的虚拟机，并在确认弹窗中展示"已选择 N 台，其中 M 台可执行此操作，K 台不符合条件将被跳过"
- 批量删除最大支持 50 台，超出时提示"单次最多删除 50 台，请分批操作"

#### 6.2.4 实时刷新规则

- 虚拟机列表页每 15 秒自动刷新状态列
- 虚拟机详情监控页每 30 秒自动刷新图表
- 物理机详情传感器数据每 30 秒自动刷新
- 用户手动操作后（开机/关机等），对应行的状态立即更新并开始轮询直至操作完成

### 6.3 虚拟机状态机

```
                    ┌─────────────────────────────────┐
                    │                                 │
                    ▼                                 │
              ┌──────────┐    创建成功    ┌──────────┐ │
    创建请求──>│  创建中   │────────────>│  已停止   │ │
              └────┬─────┘              └────┬─────┘ │

                   ▼                          ▼       │
              ┌──────────┐              ┌──────────┐ │
              │ 创建失败  │              │  运行中   │─┤ 重启
              └──────────┘              └────┬─────┘ │
                                            │        │
                                   暂停      │关机    │
                                   ┌────────┐        │
                                   ▼        ▼        │
                              ┌────────┐ ┌──────────┐│
                              │  暂停   │ │  已停止   ││
                              └───┬────┘ └────┬─────┘│
                                  │恢复        │删除   │
                                  ▼            ▼      │
                              ┌────────┐ ┌──────────┐ │
                              │  运行中  │ │  删除中   │ │
                              └────────┘ └────┬─────┘ │
                                             │       │
                                             ▼       │
                                        ┌──────────┐ │
                                        │  已删除   │ │

                    ┌──────────┐                     │
                    │  异常    │<─── 宿主机故障/存储异常│
```

**状态说明：**

| 状态 | 含义 | 是否稳定态 |
|------|------|----------|
| 创建中 | 虚拟机正在创建，资源已分配，系统初始化中 | 否（过渡态） |
| 运行中 | 虚拟机正常运行，CPU 调度中 | 是 |
| 已停止 | 虚拟机已关机，不占用 CPU 和内存 | 是 |
| 暂停 | 虚拟机暂停，内存已写入磁盘，CPU 停止调度 | 是 |

| 已删除 | 虚拟机已删除（软删除保留 24 小时后物理清除） | 是（终态） |
| 创建失败 | 创建流程异常终止 | 是（可重试或删除） |
| 异常 | 宿主机故障或存储异常导致状态未知 | 是（需人工介入） |

## 七、边界场景与异常处理

### 场景 1：宿主机资源不足时创建虚拟机

**触发条件**：用户在创建向导中选择了某台宿主机，但在确认创建时，该宿主机的可用资源已被其他并发创建请求消耗殆尽。

**处理方式**：
- 后端在创建前进行二次资源校验（原子操作：检查 + 预留）
- 如果资源不足，返回错误码 `RESOURCE_INSUFFICIENT`，前端展示"创建失败：宿主机 [名称] 资源不足，请重新选择宿主机"
- 已预留的资源立即释放
- 建议用户选择其他宿主机或联系管理员扩容

### 场景 2：虚拟机操作系统无 QEMU Guest Agent 时重置密码

**触发条件**：用户对一台运行中但未安装 QEMU Guest Agent 的虚拟机执行重置密码操作。

**处理方式**：
- 前端通过 libvirt API 查询 QGA 连接状态，无 QGA 时"重置密码"按钮灰显，hover 提示"需安装 QEMU Guest Agent"
- 如果 QGA 在前端查询时可用但执行时超时（30s），返回错误 `QGA_TIMEOUT`
- 前端展示"重置密码失败：QEMU Guest Agent 未响应。请确认虚拟机操作系统内已安装并运行 qemu-guest-agent 服务"
- 提供"如何安装 QEMU Guest Agent"的帮助链接

### 场景 3：删除正在运行中的虚拟机

**触发条件**：用户对一台"运行中"状态的虚拟机执行删除操作。

**处理方式**：
- 弹出确认对话框时额外展示红色警告"虚拟机正在运行中，删除前将自动强制关机，可能导致数据丢失"
- 用户确认后，系统先执行强制关机，等待 60 秒
- 关机成功（或超时）后执行删除流程
- 如果关机超时且虚拟机仍运行，仍执行删除（先 destroy domain，再删除磁盘）
- 记录操作日志，标注"删除时虚拟机仍在运行"

### 场景 4：镜像上传失败

**触发条件**：用户通过浏览器上传 ISO 镜像时网络中断或服务端磁盘空间不足。

**处理方式**：
- 网络中断：前端检测到上传中断后，保留已上传的分片信息，展示"上传中断，正在尝试恢复..."。网络恢复后从断点续传。如果 30 分钟内未恢复，标记上传任务为"失败"，提示用户重新上传。
- 磁盘空间不足：服务端在接收分片时检查存储池剩余空间，不足时返回错误 `STORAGE_FULL`。前端展示"上传失败：存储空间不足，请联系平台管理员扩容"，已上传的分片文件自动清理。
- 文件格式校验失败（非 ISO 文件头）：服务端在接收第一个分片后校验魔数，不匹配则拒绝上传并返回 `INVALID_FORMAT`。

### 场景 5：VNC 连接超时

**触发条件**：用户点击 VNC 连接后，WebSocket 连接在 10 秒内未建立。

**处理方式**：
- 前端展示"正在建立 VNC 连接..."的加载状态
- 10 秒超时后展示"VNC 连接超时，可能原因：1) 虚拟机 VNC 服务未启动 2) 网络不通 3) 防火墙拦截"
- 提供"重试"按钮
- 后台记录连接失败日志，包含时间戳、虚拟机 UUID、宿主机 IP

### 场景 6：宿主机离线

**触发条件**：宿主机上的 agent 心跳超过 60 秒未上报。

**处理方式**：
- 系统自动将该宿主机标记为"离线"状态
- 该宿主机上的所有虚拟机状态标记为"未知"（不可操作）
- 暂停在该宿主机上创建新虚拟机（创建向导中该宿主机灰显，标注"离线"）
- 向平台管理员发送告警通知（站内信 + 邮件）
- 宿主机恢复后，agent 重新上报心跳，系统自动恢复宿主机和虚拟机的正常状态

### 场景 7：并发创建虚拟机导致超卖

**触发条件**：多个用户同时在同一台宿主机上创建虚拟机，总计资源需求超过宿主机可用资源。

**处理方式**：
- 后端创建接口使用数据库行级锁或乐观锁机制
- 在确认创建时，执行原子操作：`SELECT ... FOR UPDATE` 锁定宿主机资源记录，检查可用资源，扣减资源，释放锁
- 如果资源不足，返回错误并回滚事务
- 前端收到错误后展示"创建失败：资源不足，请重新选择宿主机"

### 场景 8：快照链过长导致恢复性能下降

**触发条件**：虚拟机存在 10 个以上快照形成长链。

**处理方式**：
- 系统在快照数量超过 10 时在快照列表顶部展示黄色提示"快照数量较多（当前 N 个），建议清理不需要的快照以提升恢复性能"
- 创建新快照时，如果快照数量超过 20，弹出额外确认"快照数量已达上限（20），创建新快照可能影响性能，确定继续？"

### 场景 9：物理机 IPMI/BMC 凭据过期

**触发条件**：已纳管的物理机 BMC 用户名或密码被线下修改，导致平台无法通过 IPMI 控制物理机。

**处理方式**：
- 物理机详情页的传感器和监控数据区域展示"BMC 连接异常，请检查凭据是否有效"
- 远程控制按钮灰显，hover 提示"BMC 连接异常，请更新凭据"
- 在物理机列表该行展示黄色警告图标
- 提供"更新 BMC 凭据"入口，允许平台管理员重新输入用户名密码

### 场景 10：虚拟机磁盘空间写满导致运行异常

**触发条件**：虚拟机系统盘或数据盘使用率达到 100%。

**处理方式**：
- 监控系统检测到磁盘使用率超过 95% 时发出告警
- 虚拟机详情页监控图表磁盘区域高亮显示红色
- 虚拟机列表状态保持"运行中"但增加黄色警告图标
- 如果磁盘写满导致虚拟机 IO 错误，虚拟机可能进入"异常"状态，需人工介入

### 场景 11：存量纳管时发现虚拟机 XML 定义异常

**触发条件**：存量虚拟机的 libvirt domain XML 存在格式错误或缺少必要字段。

**处理方式**：
- 纳管时逐台解析 domain XML
- 对于 XML 解析失败的虚拟机，跳过纳管并在结果报告中展示"解析失败：[虚拟机名称] — [错误原因]"
- 不中断其他正常虚拟机的纳管流程
- 纳管完成后展示汇总：成功 N 台，跳过 M 台，失败 K 台

### 场景 12：克隆时目标宿主机与源宿主机 CPU 型号不兼容

**触发条件**：用户将虚拟机克隆到不同型号 CPU 的宿主机。

**处理方式**：
- 克隆时检查源和目标宿主机的 CPU model
- 如果不一致，系统自动将新虚拟机 XML 中的 CPU model 设置为 `host-model`（libvirt 会自动适配）
- 在克隆确认页面展示提示"目标宿主机 CPU 型号与源宿主机不同，将自动适配 CPU 配置"
