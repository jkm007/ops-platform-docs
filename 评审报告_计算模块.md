## 一、总体评价

**结论：需大改**

PRD 整体结构完整，对标阿里云 ECS 的交互设计思路清晰，EARS 原则的需求描述较为规范。但存在一个**架构级根本性偏差**：PRD 全文假设虚拟机磁盘使用 Ceph RBD（分布式块存储），而实际环境使用**本地 qcow2 文件**（`/home/kvm/images/`）。这一偏差导致创建流程、迁移、快照、备份、克隆、存量纳管等核心功能的实现方案全部需要重写。此外，cloud-init 集成细节、存量兼容的可行性、混部环境的资源超分策略等也需要深入修正。建议在修正存储方案假设后再进行下一轮评审。

## 二、逐条评审结论

### 2.1 磁盘方案：PRD 假设 Ceph RBD vs 实际本地 qcow2

**结论：PRD 需大改，这是本次评审最严重的架构偏差。**

PRD 中镜像存储指向 CephFS（F-IMG-02）、虚拟机磁盘隐含依赖 Ceph RBD（附录术语表明确列出），但实际环境是 `/home/kvm/images/` 本地 qcow2 文件。这导致以下影响：

- **虚拟机创建流程**：PRD 的"镜像模板克隆系统盘"逻辑在 Ceph RBD 下是秒级 COW 克隆，在本地 qcow2 下需要物理复制 qcow2 backing file + 创建 overlay，耗时与磁盘大小成正比，3-5 分钟的创建时间承诺无法保证。
- **迁移**：PRD F-VM-19 标记为 P2 且需关机，这在本地 qcow2 下是正确的（需要复制磁盘文件到目标宿主机），但 PRD 未描述迁移时磁盘文件传输机制、增量迁移策略、以及 `/home/kvm/` 路径在跨节点时的一致性问题。
- **快照**：PRD 的快照方案基于 libvirt snapshot API（F-VM-13），这在本地 qcow2 下可行（snapshot 创建为 qcow2 内部快照或外部快照文件），但需明确：外部快照会产生快照链文件，PRD 第 308 行提到"合并数据到子快照"是正确的，但需补充 blockcommit 操作的详细说明。
- **克隆**：PRD F-VM-16 写"复制源虚拟机磁盘镜像"，在本地 qcow2 下是正确的，但未考虑 qcow2 backing chain 的处理——克隆时需要 flatten 整个 backing chain 还是保留 backing file 引用？

**必须修改**：全文将所有 Ceph RBD/CephFS 的存储假设替换为本地 qcow2 文件方案，明确各宿主机上 `/home/kvm/images/`、`/home/kvm/cloud-init/images/`、`/home/kvm/kvm_snap/`、`/home/kvm/kvm_backup/` 四个目录的用途和生命周期管理。

### 2.2 cloud-init 集成：PRD 覆盖不足

**结论：PRD 有提及但细节严重缺失，需补充完整的 cloud-init 创建流程设计。**

- **已有覆盖**：第 437 行提到"注入初始化配置（cloud-init 或 QGA）"，第 441-446 行列出创建后自动操作（设置主机名、配置网络 IP、注入 SSH 公钥、安装 QGA），方向正确。
- **缺失内容**：
  1. **镜像准备流程**：PRD 未说明 cloud-init 镜像如何准备——是预先在 `/home/kvm/cloud-init/images/` 中存放已安装 cloud-init 的 qcow2 模板镜像，还是上传 ISO 手动安装后再制作为模板？这与 F-IMG-02（上传 ISO）和 F-IMG-06（实例制镜像）的流程衔接不清晰。
  2. **user-data 注入机制**：第 437 行写"cloud-init 或 QGA"，但这两者是完全不同的机制。cloud-init 通过 ISO 挂载（cloud-init NoCloud datasource）或 nocloud 网络源注入，QGA 是在 VM 启动后通过 QMP 命令注入。PRD 应明确采用 **NoCloud ISO 方式**（生成含 meta-data/user-data 的 ISO 并挂载为 cdrom），这是本地 qcow2 环境下最可靠的做法。
  3. **SSH key 配置**：PRD 第 444 行仅写"注入 SSH 公钥（如有配置）"，未说明 SSH key 的来源——是用户创建时输入、从平台密钥对管理模块获取、还是使用租户默认密钥？user-data 中 `ssh_authorized_keys` 的写入格式需要明确。
  4. **网络配置**：第 443 行"配置网络 IP"过于笼统。cloud-init 下网络配置通过 user-data 中的 `network-config` 或 `write_files` + `runcmd` 实现，需明确是使用 cloud-init 原生网络配置 v1/v2 还是通过自定义脚本。
  5. **主机名规范**：第 442 行"设置主机名"，需明确主机名来源（用户输入 vs 自动生成），以及 cloud-init 中 `hostname` / `fqdn` 字段的对应关系。
  6. **错误处理**：cloud-init 初始化失败时（如网络不通导致 apt/yum 源不可达），VM 将启动但未完成初始化，PRD 未定义这种中间状态的检测和处理方式。

**必须修改**：新增"cloud-init 集成设计"专节，详细描述 NoCloud ISO 生成流程、user-data 模板结构、SSH key 注入、网络配置方式、初始化状态检测机制。

### 2.3 存量兼容：只读纳管 32 台存量 VM

**结论：方案基本可行，但存在 3 个需要修正的细节问题。**

PRD 第 397-406 行的存量纳管设计思路正确——通过 libvirt API 读取 domain XML、只读标记、限制修改操作。但结合实际环境需要修正：

- **UUID 匹配风险**：第 463 行"按 UUID 匹配"是正确做法，但需注意：如果存量 VM 是通过 `virt-clone` 或手动复制 qcow2 文件 + 修改 XML 创建的，可能出现 UUID 冲突。建议增加 **MAC 地址作为辅助匹配键**。
- **磁盘路径发现**：存量 VM 的 domain XML 中磁盘 source file 指向 `/home/kvm/images/xxx.qcow2`，纳管时需要解析此路径、校验文件是否存在、读取 qcow2 文件信息（虚拟大小、实际占用、backing file），并将这些信息写入平台数据库。PRD 第 466 行只写"读取 domain XML 获取配置信息（vCPU、内存、磁盘、网络）"，未细化磁盘发现步骤。
- **存量 VM 的 cloud-init 状态未知**：32 台存量 VM 的初始化方式可能不一致（部分手动安装、部分 cloud-init），纳管后平台无法保证这些 VM 与平台创建的 VM 在配置管理上的一致性。建议在纳管报告中标注"初始化方式未知"。
- **只读转平台管理的风险**：第 405-406 行允许将只读纳管转为平台管理模式。转换后平台将接管生命周期管理，但需注意：转换前平台必须对存量 VM 的磁盘文件、快照链、备份文件进行全面健康检查，防止接管后因磁盘文件状态异常导致操作失败。

**建议优化**：补充磁盘路径解析和 qcow2 文件信息采集的具体步骤，增加 MAC 地址辅助匹配，定义"只读转平台管理"的前置条件检查清单。

### 2.4 资源超分：CPU 4:1、内存不超分

**结论：方案在混部场景下风险较高，建议调整为更保守的比例并增加动态监控。**

PRD 第 385 行"CPU 超分比 4:1，内存 1:1（不超分）"：

- **CPU 4:1 超分评估**：6 台 R740 同时承载 KVM、Ceph OSD、K8s worker 三种负载。Ceph OSD 对 CPU 消耗不小（尤其在 recovery/backfill 时），K8s 容器化业务也有持续 CPU 消耗。实际可用于 KVM 的 CPU 资源需要扣除 Ceph + K8s 的基线消耗后才能计算超分比。建议先采集各节点 CPU 使用率基线数据（至少 7 天），然后按"（物理核数 - Ceph基线 - K8s基线）× 超分比"计算可用 vCPU。盲目 4:1 可能导致 Ceph 性能下降或 VM CPU steal time 飙升。
- **内存不超分（1:1）**：方向正确，但同样需要扣除 Ceph OSD 内存占用（通常建议每 TB OSD 预留 1GB 内存）、K8s 节点预留、以及系统自身开销。实际可分配内存 = 物理内存 - Ceph预留 - K8s预留 - 系统预留（建议至少 8GB/节点）。
- **缺乏超分安全边际**：PRD 第 387-388 行"超分比设置导致已分配超过物理×超分比时仅警告不阻止"，这在混部环境下过于宽松。建议增加**硬限制**：当 Ceph 集群处于 recovery 状态或 K8s 节点负载超过阈值时，自动暂停新 VM 创建，防止资源争抢导致所有服务雪崩。
- **规格过滤逻辑不完整**：第 202 行"规格资源需求超过宿主机可用资源时禁用"，但"可用资源"的计算公式未定义——是否已扣除混部负载基线？是否按超分比换算？这直接影响创建向导的资源展示准确性。

**必须修改**：明确混部环境下的可用资源计算公式（扣除 Ceph/K8s 基线），CPU 超分比建议下调至 2:1~3:1（待基线数据确认），增加混部负载感知的动态创建控制。

### 2.5 快照和备份：与现有目录的整合

**结论：PRD 快照方案与现有目录结构存在冲突，需重新设计整合方案。**

现有环境有 `/home/kvm/kvm_snap/` 和 `/home/kvm/kvm_backup/` 两个独立目录，而 PRD 仅基于 libvirt snapshot API 设计快照功能：

- **libvirt snapshot 的文件位置**：libvirt 外部快照（`--disk-only`）创建的 qcow2 快照文件默认位于虚拟机磁盘同目录（即 `/home/kvm/images/`），或由 XML 中指定的路径决定。现有 `/home/kvm/kvm_snap/` 目录说明团队已有手动快照管理实践，需要确认：存量快照是如何创建的（`virsh snapshot-create` vs 手动 `qemu-img create`）？是内部快照还是外部快照？
- **备份目录未纳入 PRD**：`/home/kvm/kvm_backup/` 目录在 PRD 中完全没有提及。PRD 功能清单中无"备份管理"功能。建议至少增加 **P1 级别的备份查看功能**（列出存量备份文件、关联到 VM、支持恢复操作），否则纳管后存量备份文件将成为"盲区"。
- **快照存储路径统一**：PRD 若继续使用 libvirt snapshot API，建议统一将快照文件写入 `/home/kvm/kvm_snap/`，与现有实践保持一致。需要设计存量快照的发现和导入机制——通过解析 snapshot XML 或扫描 qcow2 backing file chain 来识别已存在的快照关系。
- **备份方案缺失**：现有 `/home/kvm/kvm_backup/` 目录暗示团队已有备份实践（可能是 `qemu-img convert` 或 `cp` 产生的完整磁盘副本）。PRD 应补充备份管理功能（备份列表、从备份恢复、备份清理策略），否则平台上线后备份操作将退化为 SSH 手动操作。

**必须修改**：增加"备份管理"功能模块（至少 P1 查看+恢复），明确快照文件存储路径统一到 `/home/kvm/kvm_snap/`，设计存量快照和备份文件的发现与导入机制。

### 2.6 创建流程：5 步向导完整性

**结论：流程框架完整，但缺少 cloud-init 专属配置步骤，且磁盘配置缺失。**

PRD 第 180-225 行的 5 步创建流程：

- **缺失步骤：cloud-init 配置**。当前 5 步为"选宿主机→选镜像→选规格→选网络→确认"。在使用 cloud-init 模板镜像时，缺少关键的配置步骤：
  - SSH 公钥注入（从密钥对列表选择或粘贴）
  - 主机名设置
  - 用户自定义初始化脚本（可选，PRD 第 446 行提到但无对应 UI）
  - 建议将 Step 3（选规格）与 cloud-init 配置合并为"选规格与初始化配置"，或增加 Step 2.5 "cloud-init 配置"子步骤。
- **系统盘大小不可配**：第 215 行汇总信息包含"系统盘大小"，但前序步骤中没有任何地方让用户指定系统盘大小。实际需求中，用户可能需要选择大于镜像模板的系统盘（如模板 20GB，实际需要 100GB）。PRD 需在 Step 3 或 Step 5 中增加系统盘大小配置。
- **数据盘配置缺失**：第 215 行提到"数据盘配置（如有）"，但整个 5 步向导中没有数据盘创建步骤。需增加"添加数据盘"的可选步骤或子步骤（选择容量、存储池位置）。
- **镜像与 cloud-init 兼容性校验缺失**：第 195 行"选择 ISO 时提示手动安装"、第 196 行"选择模板时提示自动初始化"，但未校验模板镜像是否已安装 cloud-init。如果用户选择了一个未预装 cloud-init 的 qcow2 模板，创建后初始化将静默失败。

**必须修改**：增加 cloud-init 配置步骤（SSH key、主机名、自定义脚本），增加系统盘大小配置，增加可选的数据盘创建步骤，增加镜像 cloud-init 兼容性校验。

### 2.7 技术风险识别

**结论：识别以下 5 个关键技术风险。**

| 编号 | 风险描述 | 影响程度 | 建议措施 |
|------|---------|---------|---------|
| RISK-01 | **本地 qcow2 单点故障**：虚拟机磁盘文件存储在单台宿主机本地，宿主机故障将导致该节点所有 VM 不可用，且无法像 Ceph RBD 那样在其他节点快速拉起。6 台 R740 任一台宕机，影响面为 ~1/6 的 VM 容量。 | **高** | 短期：加强宿主机硬件监控和备件储备；中期：实现 qcow2 文件定时增量备份到其他节点或 CephFS；长期：评估是否迁移到 Ceph RBD。 |
| RISK-02 | **Ceph + KVM + K8s 混部资源争抢**：Ceph OSD recovery/backfill、K8s 容器突发流量、KVM 虚拟机负载三者叠加时，可能导致磁盘 IO 饥饿（Ceph 和 KVM 共享磁盘控制器）或 CPU 饥饿（所有服务争抢同一物理核）。 | **高** | 实施 cgroup 资源隔离（CPU set、内存 limit、IO 权重），配置 Ceph OSD 优先级和恢复限速，建立资源争抢的实时检测和告警机制。 |
| RISK-03 | **qcow2 backing chain 过长导致性能衰减**：cloud-init 模板镜像 → VM overlay → 快照 overlay → 快照 overlay ... 链过长后，每次 IO 需要穿透多层 qcow2，磁盘性能线性下降。PRD 第 308 行提到快照链管理但仅限制数量（20），未考虑合并策略。 | **中** | 实现自动 blockcommit 策略：快照链深度超过 5 层时自动合并（业务低峰期），或定期将运行时间超过 N 天的 VM 进行 backing file flatten。 |
| RISK-04 | **存量 VM 纳管的配置漂移**：32 台存量 VM 的 domain XML 可能包含非标准配置（手动修改的 CPU pinning、自定义设备、非标准网络桥接），纳管后平台对这些配置无感知，后续操作（如配置变更）可能破坏这些手动配置。 | **中** | 纳管时对 domain XML 进行完整性扫描和报告，标记非标准配置项，限制对此类 VM 的配置变更操作。 |
| RISK-05 | **cloud-init 初始化失败无感知**：VM 启动后，cloud-init 在 guest 内部执行初始化（网络配置、软件安装、脚本执行），如果初始化过程中某个步骤失败（如 yum 源不可达），libvirt 层面无法感知——VM 状态显示"运行中"但实际未完成初始化。 | **中** | 实现 cloud-init 状态检测机制：在 VM 启动后通过 QGA 或轮询 SSH 端口检测 cloud-init 是否完成（`cloud-init status`），超时未完成则标记为"初始化异常"状态。 |

## 附录：评审方法说明

本评审采用以下方法：
1. **架构对齐分析**：将 PRD 技术假设与实际环境逐项对比，识别架构级偏差。
2. **功能完整性检查**：基于实际环境验证创建、迁移、快照、备份、克隆等功能的实现可行性。
3. **EARS 需求覆盖度**：检查详细需求是否覆盖了实际环境的关键操作场景。
4. **风险评估矩阵**：从"影响范围 × 发生概率"两个维度评估技术风险等级。
