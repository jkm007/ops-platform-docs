## 一、模块概述

### 1.1 模块定位

权限与审计模块是运维内网管理平台的**安全基石**，负责统一管控"谁能做什么、能看到什么、做了什么"，实现基于多租户 RBAC 的细粒度访问控制和全链路操作审计。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| 多租户隔离 | 不同部门/团队只能看到和操作自己租户下的资源，不可越权访问 |
| 最小权限原则 | 默认无权限，按需分配角色，角色内按资源粒度授权 |
| 全量操作审计 | 所有 CUD（创建/修改/删除）操作 100% 记录，不可篡改、不可删除 |
| 安全合规 | 满足等保三级对身份鉴别、访问控制、安全审计的要求 |

### 1.3 子模块组成

```
权限与审计模块
├── 用户管理（含密码策略、登录安全）
├── 角色管理（预定义 + 自定义）
├── 多租户管理（租户创建/配额/隔离）
├── 权限策略（资源级 + 操作级 RBAC）
└── 操作审计（日志记录/搜索/导出）
```

### 1.4 与其他模块的关系

- **基础设施模块**：租户配额需校验物理资源总量
- **KVM 管理模块**：虚拟机 CRUD 操作需进行权限校验并记录审计日志
- **Ceph 管理模块**：RBD/RGW/CephFS 操作需按租户隔离存储池
- **K8s 管理模块**：集群资源需归属到租户 namespace，权限校验到 namespace 级别
- **日志系统**：审计日志写入独立日志存储，仅平台管理员可查询

## 二、用户故事

### US-01：平台管理员创建新租户并初始化

> **作为** 平台管理员，
> **我希望** 能够创建一个新租户（如"数据部"），设置其资源配额（VM 数量/CPU/内存/存储），并指定租户管理员，
> **以便** 新部门可以自主管理自己的云资源，且不超过配额限制。

**验收条件：**
- 平台管理员可在租户管理页点击"创建租户"
- 填写租户名称、描述、配额（VM 上限、CPU 核数、内存 GB、存储 GB）
- 指定至少 1 名租户管理员
- 创建后自动初始化租户默认角色（租户管理员/运维/开发者/只读）
- 租户管理员可登录并开始管理本租户资源

---

### US-02：租户管理员为团队成员分配角色

> **作为** 租户管理员（研发部），
> **我希望** 能够将"研发部"的同事添加为租户用户，并分配"租户开发者"角色，
> **以便** 开发人员可以自助创建开发和测试用的虚拟机。

**验收条件：**
- 租户管理员可在用户管理页看到本租户所有用户
- 可创建新用户或邀请已有平台用户加入本租户
- 可为用户分配本租户内的角色（租户运维/开发者/只读，或自定义角色）
- 不可分配"平台管理员"角色或跨租户角色
- 角色变更后即时生效，无需重新登录

---

### US-03：开发者自助创建虚拟机

> **作为** 研发部的开发者，
> **我希望** 能够在租户配额内自助创建虚拟机，
> **以便** 快速搭建开发测试环境，无需等待运维人员手动操作。

**验收条件：**
- 开发者登录后仅看到"研发部"租户的资源
- 创建 VM 时可选择规格，系统自动校验配额余量
- 超出配额时提示"租户配额不足，请联系租户管理员"
- 创建成功后，审计日志记录操作人、时间、VM 规格、IP
- 操作记录在审计日志中可搜索

---

### US-04：安全管理员排查异常登录

> **作为** 安全管理员（平台管理员角色），
> **我希望** 能够查看所有用户的登录日志，筛选异地登录和频繁失败记录，
> **以便** 及时发现并处理潜在的安全威胁。

**验收条件：**
- 审计日志页可按"登录"事件类型筛选
- 系统自动标记异常登录（IP 归属地变化、连续失败≥5次）
- 异常登录记录高亮显示
- 支持按用户名、IP、时间范围搜索
- 支持导出异常登录报表（CSV）

---

### US-05：审计员追溯历史操作

> **作为** 审计员，
> **我希望** 能够搜索过去 90 天内所有删除操作记录，包括谁删除了什么资源、当时用的什么参数，
> **以便** 满足内部审计和合规检查要求。

**验收条件：**
- 审计日志页支持按操作类型="删除"筛选
- 支持按资源类型（VM/存储/网络等）筛选
- 每条记录包含：操作人、操作时间、IP、操作类型、资源类型、资源 ID、操作前/后快照
- 支持按时间范围（默认保留 180 天，可配置）
- 支持导出为 CSV/JSON
- 审计日志不可删除、不可修改

---

### US-06：租户管理员查看本租户资源使用情况

> **作为** 租户管理员，
> **我希望** 能够查看本租户各类资源的当前使用量和配额占比，
> **以便** 及时调整配额申请或清理闲置资源。

**验收条件：**
- 租户概览页显示 VM 使用数/配额、CPU 使用数/配额、内存使用/配额、存储使用/配额
- 使用率超过 80% 时黄色告警，超过 95% 时红色告警
- 可点击详情查看具体资源列表

## 四、详细需求描述（EARS 格式）

### 4.1 用户创建与管理

#### REQ-USER-001：创建用户

**While** 平台管理员或租户管理员处于用户管理页面，
**when** 点击"创建用户"按钮并填写用户信息，
**the system shall** 校验用户名唯一性、密码复杂度、邮箱格式，
**and shall** 创建用户记录并分配指定角色，
**and shall** 发送欢迎邮件（含首次登录链接）至用户邮箱。

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 用户名 | string(3-32) | 是 | 全局唯一，字母开头，支持字母/数字/下划线 |
| 显示名 | string(2-32) | 是 | 用户展示名称 |
| 密码 | string(8-64) | 是 | 符合密码策略，后端加密存储（bcrypt） |
| 邮箱 | email | 是 | 唯一，用于密码重置 |
| 手机号 | phone | 否 | 用于 MFA（二期） |
| 所属租户 | select | 是 | 平台管理员可分配任意租户，租户管理员仅限本租户 |
| 角色 | multi-select | 是 | 至少 1 个角色 |
| 描述 | text(200) | 否 | 备注信息 |

#### REQ-USER-002：用户登录

**When** 用户在登录页输入用户名和密码并提交，
**the system shall** 校验用户是否存在、是否被禁用、密码是否正确，
**and shall** 检查连续登录失败次数是否达到阈值，
**and shall** 成功后创建会话（JWT Token，有效期 2 小时，包含唯一 jti 标识），
**and shall** 将 jti 存入用户活跃 Token 集合（Redis Set，key=`user_tokens:{user_id}`），
**and shall** 记录登录日志（用户名、IP、User-Agent、时间、结果），
**and shall** 检测登录 IP 是否属于异地登录。

#### REQ-USER-002-1：Token 黑名单吊销机制

**The system shall** 在 Redis 中维护已吊销 Token 的黑名单，解决无状态 JWT 无法主动失效的问题。

**黑名单数据结构：**

| Key | Value | TTL | 说明 |
|-----|-------|-----|------|
| `blacklist:jti:{jti}` | 吊销原因（user_disabled/password_reset/user_deleted/admin_revoke） | Token 剩余有效期 | 单个被吊销的 Token |
| `user_tokens:{user_id}` | Set of jti | 随成员 TTL 自动清理 | 用户所有活跃 Token 的 jti 集合 |

**校验流程：**

**When** 每次 API 请求在 JWT 签名验证通过后，
**the system shall** 从 JWT 中提取 jti，
**and shall** 查询 Redis 黑名单 `blacklist:jti:{jti}` 是否存在，
**if** 命中黑名单，
**the system shall** 拒绝请求并返回 401 "Token 已被吊销"。

**吊销触发场景：**

| 场景 | 操作 |
|------|------|
| 用户被禁用 | 将该用户 `user_tokens:{user_id}` 中所有 jti 加入黑名单 |
| 用户被删除 | 将该用户所有活跃 Token 加入黑名单 |
| 密码被重置 | 将该用户所有活跃 Token 加入黑名单，强制重新登录 |
| 管理员主动踢出会话 | 将指定 jti 加入黑名单 |
| 角色被修改 | 将该用户所有活跃 Token 加入黑名单（权限变更需重新登录生效） |

**性能与一致性：**

- **生效延迟**：Token 吊销后最多 60 秒内生效（Redis 主从同步延迟），通常在同一数据中心内 < 10ms
- **Redis 不可用降级**：Redis 连接失败时记录告警日志，放行请求（可用性优先），Redis 恢复后补检
- **黑名单清理**：过期 Token 自动被 Redis TTL 清理，无需额外维护
- **内存估算**：假设日均 500 活跃用户、人均 2 个 Token，每个 jti 36 字节，黑名单内存 < 100KB

**The system shall** 定期（每小时）扫描 `user_tokens:{user_id}` 集合，移除已过期的 jti。

#### REQ-USER-003：密码策略

**When** 用户设置或修改密码，
**the system shall** 校验密码满足以下规则：
- 最小长度 8 位，最大 64 位
- 必须包含大写字母、小写字母、数字、特殊字符中的至少三类
- 不可包含用户名或邮箱
- 不可与最近 5 次历史密码重复

**When** 密码使用超过 90 天，
**the system shall** 在用户登录时强制跳转至密码修改页。

#### REQ-USER-004：登录失败锁定

**When** 同一用户名连续登录失败达到 5 次（可配置），
**the system shall** 锁定该账号 30 分钟（可配置），
**and shall** 锁定期间拒绝该账号的所有登录尝试，
**and shall** 在审计日志中记录锁定事件。

**When** 锁定时间到期，
**the system shall** 自动解除锁定，用户可正常登录。

#### REQ-USER-005：密码重置

**When** 管理员对某用户执行密码重置操作，
**the system shall** 生成临时密码（随机 16 位），
**and shall** 发送临时密码至用户邮箱，
**and shall** 标记该用户下次登录时强制修改密码，
**and shall** 将该用户所有活跃 Token 的 jti 加入黑名单（强制所有会话重新登录），
**and shall** 记录审计日志。

#### REQ-USER-006：会话超时

**When** 用户已登录且超过 30 分钟（可配置）无任何操作，
**the system shall** 自动将会话标记为过期，
**and shall** 用户下次操作时返回 401 状态码并跳转至登录页，
**and shall** 前端在超时前 5 分钟弹出续期提醒。

### 4.2 角色管理

#### REQ-ROLE-001：预定义角色

**The system shall** 在系统初始化时创建以下 6 个预定义角色，不可删除、不可修改名称和基础权限范围：

| 角色 | 标识 | 说明 |
|------|------|------|
| 平台管理员 | platform_admin | 全局管理所有租户、资源池、系统配置 |
| 安全管理员 | security_admin | 查看所有租户安全组和规则、批量封禁 IP、查看所有审计日志（不可管理用户和业务资源） |
| 租户管理员 | tenant_admin | 管理本租户的用户、角色、资源、配额 |
| 租户运维 | tenant_operator | 管理本租户的虚拟机、存储、网络等资源 |
| 租户开发者 | tenant_developer | 查看本租户资源，自助创建 VM/数据库 |
| 只读用户 | tenant_viewer | 仅查看本租户资源，无任何修改权限 |

#### REQ-ROLE-002：创建自定义角色

**When** 租户管理员点击"创建角色"，
**the system shall** 允许填写角色名称（本租户内唯一）、描述，
**and shall** 展示权限配置界面（资源类型 x 操作类型矩阵），
**and shall** 创建角色后即时生效。

**权限配置矩阵示例：**

| 资源类型 | 读 | 创建 | 修改 | 删除 | 管理 |
|---------|-----|------|------|------|------|
| 虚拟机 | ☑ | ☑ | ☑ | ☐ | ☐ |
| 存储 | ☑ | ☐ | ☐ | ☐ | ☐ |
| 网络 | ☑ | ☐ | ☐ | ☐ | ☐ |
| 用户 | ☐ | ☐ | ☐ | ☐ | ☐ |
| 角色 | ☐ | ☐ | ☐ | ☐ | ☐ |

#### REQ-ROLE-003：删除自定义角色

**When** 租户管理员尝试删除自定义角色，
**if** 该角色仍有用户绑定，
**the system shall** 提示"该角色下有 N 个用户，请先将用户迁移至其他角色后再删除"，
**and shall** 提供一键迁移功能（选择目标角色，批量迁移），
**and shall** 不允许直接删除有关联用户的角色。

### 4.3 多租户管理

#### REQ-TENANT-001：创建租户

**When** 平台管理员点击"创建租户"，
**the system shall** 校验租户名称全局唯一，
**and shall** 创建租户记录，
**and shall** 自动初始化该租户的 4 个预定义角色（租户管理员/运维/开发者/只读），
**and shall** 为租户创建默认资源配额（可在创建时修改），
**and shall** 在 Ceph 中创建该租户的独立存储池（RBD pool + RGW bucket namespace），
**and shall** 在 K8s 中创建该租户的独立 Namespace，
**and shall** 记录审计日志。

#### REQ-TENANT-002：资源配额管理

**When** 平台管理员修改租户配额，
**the system shall** 校验新配额不低于当前已使用量，
**if** 新配额低于已使用量，
**the system shall** 拒绝修改并提示"新配额不可低于当前已使用量"。

**配额字段：**

| 配额项 | 单位 | 默认值 | 说明 |
|--------|------|--------|------|
| VM 数量上限 | 个 | 50 | 租户可创建的 VM 总数 |
| CPU 核数上限 | 核 | 200 | 所有 VM 的 vCPU 总和 |
| 内存上限 | GB | 512 | 所有 VM 的内存总和 |
| 存储上限 | GB | 5120 | RBD + CephFS + RGW 总和 |
| 网络带宽上限 | Mbps | 1000 | 租户总出口带宽 |

#### REQ-TENANT-003：租户禁用

**When** 平台管理员禁用某租户，
**the system shall** 禁止该租户下所有用户执行任何写操作，
**and shall** 禁止该租户用户创建新会话（已登录会话保留），
**and shall** 不停止、不删除该租户的已有资源（VM 继续运行），
**and shall** 记录审计日志。

#### REQ-TENANT-004：资源可见性隔离

**When** 任何用户发起资源查询请求，
**the system shall** 在 API 中间件层注入租户过滤条件（tenant_id），
**and shall** 确保用户只能查询到所属租户的资源，
**and shall** 对跨租户访问请求返回"无权限"错误。

**When** 平台管理员查看全局资源，
**the system shall** 展示所有租户的资源（平台管理员不受租户隔离限制）。

### 4.4 权限策略

#### REQ-PERM-001：权限模型

**The system shall** 采用 RBAC 权限模型：

```
用户 (User) ──N:M──> 角色 (Role) ──1:N──> 权限策略 (Permission)
```

**权限策略定义：**

```json
{
  "effect": "allow",
  "resource": "vm",
  "actions": ["read", "create", "update"],
  "scope": {
    "type": "tenant",
    "tenant_id": "{{user.tenant_id}}"
  }
}
```

#### REQ-PERM-002：权限校验流程

**When** 用户发起任何 API 请求，
**the system shall** 经过以下校验链：

```
请求 → 认证中间件（JWT 校验）→ Token 黑名单检查（Redis）→ 授权中间件（RBAC 校验）
     → 租户隔离中间件（注入 tenant_id）
     → 配额校验中间件（创建/扩容操作）
     → 业务处理
     → 审计日志中间件（CUD 操作记录）
     → 响应
```

**When** 任一中间件校验失败，
**the system shall** 返回对应错误码（401/403/422）并拒绝请求。

#### REQ-PERM-003：权限决策

**When** 需要判断用户对某资源是否有某操作权限，
**the system shall** 执行以下决策逻辑：

1. 判断用户是否为平台管理员 → 有全部权限
2. 判断用户是否为租户管理员 → 对本租户资源有全部权限
3. 遍历用户的所有角色，合并所有权限策略
4. 判断请求的资源 + 操作是否在合并后的权限范围内
5. 判断资源是否属于用户所在租户（租户隔离）
6. 全部通过 → 允许；任一不通过 → 拒绝

### 4.5 操作审计

#### REQ-AUDIT-001：审计日志记录

**When** 任何用户执行 CUD（创建/修改/删除）操作，
**the system shall** 记录以下审计日志字段：

| 字段 | 说明 |
|------|------|
| id | 唯一 ID（UUID v7，按时间排序） |
| timestamp | 操作时间（UTC，精确到毫秒） |
| user_id | 操作用户 ID |
| username | 操作用户名 |
| tenant_id | 所属租户 ID |
| client_ip | 客户端 IP |
| user_agent | 客户端 User-Agent |
| action | 操作类型：create/update/delete |
| resource_type | 资源类型：vm/storage/network/user/role/tenant 等 |
| resource_id | 资源唯一 ID |
| resource_name | 资源名称（冗余，便于搜索） |
| request_params | 请求参数（脱敏后） |
| before_snapshot | 操作前资源快照（JSON，update/delete 时记录） |
| after_snapshot | 操作后资源快照（JSON，create/update 时记录） |
| result | 操作结果：success/failure |
| error_message | 失败原因 |
| trace_id | 请求链路追踪 ID |

#### REQ-AUDIT-002：审计日志存储

**The system shall** 将审计日志写入独立的数据存储（ClickHouse 或 Elasticsearch），
**and shall** 业务数据库与审计日志数据库物理分离，
**and shall** 审计日志仅支持追加写入（append-only），
**and shall** 不提供任何修改或删除审计日志的 API，
**and shall** 审计日志数据库使用独立账号，仅审计模块可写入。

#### REQ-AUDIT-003：审计日志搜索

**When** 用户在审计日志页进行搜索，
**the system shall** 支持以下筛选条件组合：

| 筛选条件 | 类型 | 说明 |
|---------|------|------|
| 时间范围 | 日期范围 | 默认最近 7 天，最大跨度 90 天 |
| 用户名 | 文本搜索 | 模糊匹配 |
| 操作类型 | 多选 | create/update/delete |
| 资源类型 | 多选 | vm/storage/network/user/role/tenant |
| 操作结果 | 单选 | 全部/成功/失败 |
| 租户 | 下拉 | 平台管理员可看所有租户，其他仅看本租户 |
| 资源名称 | 文本搜索 | 模糊匹配 |

**The system shall** 搜索结果按时间倒序排列，
**and shall** 支持分页（默认每页 20 条，可调至 50/100），
**and shall** 响应时间不超过 2 秒（99 分位）。

#### REQ-AUDIT-004：审计日志导出

**When** 用户点击"导出"按钮，
**the system shall** 将当前搜索结果导出为 CSV 或 JSON 文件，
**and shall** 单次导出上限 10000 条，
**and shall** 超出上限时提示"结果超过 10000 条，请缩小筛选范围后导出"，
**and shall** 记录导出操作本身到审计日志。

### 4.6 登录安全

#### REQ-SEC-001：异常登录检测

**When** 用户登录成功，
**the system shall** 比对本次登录 IP 归属地与上次登录 IP 归属地，
**if** 归属地城市不同，
**the system shall** 将该登录标记为"异地登录"，
**and shall** 向用户邮箱发送安全提醒邮件，
**and shall** 在登录历史中高亮显示。

#### REQ-SEC-002：登录历史

**When** 用户查看自己的登录历史，
**the system shall** 展示最近 30 天内的登录记录，包含：
- 登录时间
- 登录 IP
- IP 归属地（城市）
- 设备类型（浏览器/User-Agent）
- 登录结果（成功/失败/锁定）
- 是否异地登录（标记）

#### REQ-SEC-003：并发登录限制

**When** 用户成功登录，
**the system shall** 检查该用户当前活跃会话数（Redis 中 `user_tokens:{user_id}` 集合大小），
**if** 活跃会话数已达到 3 个（可配置），
**the system shall** 自动踢出最早创建的会话（将其 jti 加入黑名单），
**and shall** 在审计日志中记录"并发超限，最早会话已失效"。

**The system shall** 在登录成功后返回当前活跃会话列表（设备/IP/登录时间），
**and shall** 允许用户在自己的"安全设置"页面查看和主动踢出其他会话。

#### REQ-SEC-004：登录 IP 白名单

**When** 租户管理员为租户配置 IP 白名单（CIDR 格式，如 `10.0.0.0/8`），
**the system shall** 在用户登录时校验客户端 IP 是否在白名单范围内，
**if** 客户端 IP 不在白名单内，
**the system shall** 拒绝登录并返回"当前 IP 不在允许的登录范围内"。

**配置规则：**
- IP 白名单在租户级别配置，适用于该租户下所有用户
- 支持多条 CIDR 规则（如公司内网 + VPN 网段）
- 平台管理员 IP 不受白名单限制（避免管理员被自己锁在门外）
- 白名单变更即时生效，不影响已登录会话
- 白名单变更记录审计日志

**降级策略：**
**If** 白名单配置错误导致所有用户无法登录，
**the system shall** 平台管理员可通过物理服务器本地控制台或独立管理接口紧急关闭白名单。

### 5.1 新租户创建与初始化流程

```
平台管理员
    │
    ├── 1. 点击"创建租户"
    │      ├── 填写：租户名称、描述、配额（VM/CPU/内存/存储/带宽）

    │      └── 提交
    │
    ├── 2. 系统处理
    │      ├── 校验租户名称唯一性
    │      ├── 在数据库创建租户记录
    │      ├── 自动创建该租户的 4 个预定义角色
    │      ├── 在 Ceph 创建独立存储池（tenant-{id}-rbd, tenant-{id}-rgw）
    │      ├── 在 K8s 创建独立 Namespace（tenant-{id}）
    │      ├── 设置租户资源配额
    │      ├── 将指定用户绑定"租户管理员"角色
    │      ├── 发送通知邮件给租户管理员
    │      └── 记录审计日志
    │
    └── 3. 完成
           └── 租户管理员可登录并开始管理
```

### 5.2 用户登录与权限校验流程

```
用户
    │
    ├── 1. 登录页输入用户名+密码
    │      └── 前端提交 POST /api/v1/auth/login
    │
    ├── 2. 认证服务
    │      ├── 检查账号是否被禁用 → 禁用则返回 403
    │      ├── 检查是否在锁定状态 → 锁定则返回 423
    │      ├── 校验密码 → 失败则记录失败次数
    │      ├── 生成 JWT Token（含 user_id, tenant_id, roles, jti）
    │      ├── 将 jti 加入用户活跃 Token 集合（Redis Set）
    │      ├── 记录登录日志
    │      ├── 检测异地登录
    │      └── 返回 Token + 用户信息
    │
    ├── 3. 后续请求
    │      ├── 请求头携带 Authorization: Bearer <token>
    │      ├── API Gateway / 中间件
    │      │   ├── JWT 校验（有效性、过期）
    │      │   ├── Token 黑名单检查（Redis 查询 blacklist:jti:{jti}）
    │      │   ├── 提取 user_id, tenant_id, roles
    │      │   ├── RBAC 权限校验（资源+操作）
    │      │   ├── 租户隔离校验（tenant_id 注入）
    │      │   └── 配额校验（如适用）
    │      ├── 业务处理
    │      └── 审计日志记录（CUD 操作）
    │
    └── 4. 响应
```

### 5.3 敏感操作审计链（以创建 VM 为例）

```
开发者 创建 VM
    │
    ├── 1. 发起请求 POST /api/v1/tenants/{tid}/vms
    │      Body: { "name": "dev-nginx-01", "cpu": 4, "memory": 8192, ... }
    │      Headers: Authorization: Bearer <jwt>
    │
    ├── 2. 认证中间件
    │      └── 解析 JWT → user_id=1001, tenant_id=10, roles=["tenant_developer"]
    │
    ├── 3. 授权中间件
    │      └── 检查角色 tenant_developer 是否有 vm:create 权限 → 通过
    │
    ├── 4. 租户隔离中间件
    │      └── 校验 URL 中 tenant_id(10) == JWT 中 tenant_id(10) → 通过
    │
    ├── 5. 配额校验中间件
    │      ├── 查询租户 10 当前 VM 数量=12, 配额=50 → 通过
    │      ├── 查询租户 10 当前 CPU 使用=40, 配额=200 → 40+4=44 < 200 → 通过
    │      └── 查询租户 10 当前内存使用=128G, 配额=512G → 128+8=136 < 512 → 通过
    │
    ├── 6. 业务处理
    │      └── 调用 KVM 驱动创建 VM，返回 vm_id=5501
    │
    ├── 7. 审计中间件
    │      └── 写入审计日志：
    │          {
    │            "user_id": 1001,
    │            "username": "zhangsan",
    │            "tenant_id": 10,
    │            "client_ip": "10.0.1.55",
    │            "action": "create",
    │            "resource_type": "vm",
    │            "resource_id": "5501",
    │            "resource_name": "dev-nginx-01",
    │            "request_params": { "name": "dev-nginx-01", "cpu": 4, ... },
    │            "after_snapshot": { "id": 5501, "name": "dev-nginx-01", ... },
    │            "result": "success",
    │            "trace_id": "abc-123-def"
    │          }
    │
    └── 8. 响应
           └── 201 Created + VM 详情
```

**可追溯链路：** 任意审计日志记录 → 通过 `user_id` 找到操作人 → 通过 `trace_id` 关联所有相关日志 → 通过 `before_snapshot` / `after_snapshot` 查看资源变更详情。

## 七、边界场景与异常处理

### 7.1 删除仍有关联用户的租户

**场景：** 平台管理员尝试删除"研发部"租户，但该租户下仍有 45 个活跃用户。

**处理规则：**
1. 系统检测租户下是否有活跃用户
2. 若有，提示"该租户下有 45 个用户，请先将用户迁移至其他租户或禁用后再删除"
3. 提供"批量迁移用户"功能，选择目标租户，一键迁移所有用户
4. 迁移后重新校验
5. 若租户下仍有运行中的资源（VM/存储等），提示"该租户下仍有运行中的资源，请先清理"
6. 资源清空 + 用户清空后，才允许删除
7. 删除操作不可逆，需输入租户名称二次确认

### 7.2 禁用租户后其资源如何处理

**场景：** 数据部租户因业务调整被禁用。

**处理规则：**
1. 禁用租户后，该租户下所有用户的写操作被阻止（返回 403）
2. 已登录用户的会话保留，但下次 API 请求时被拒绝
3. 运行中的 VM **不自动关机**（避免业务中断风险），需人工确认后手动关机
4. 存储资源保留，不删除
5. 禁用 30 天后，系统自动发送提醒邮件给平台管理员
6. 平台管理员可选择：重新启用 / 清理资源后删除租户

### 7.3 并发修改同一用户权限

**场景：** 租户管理员 A 和租户管理员 B 同时编辑同一用户的角色分配。

**处理规则：**
1. 使用乐观锁（version 字段）处理并发
2. 先提交的请求成功
3. 后提交的请求检测到 version 不匹配，返回 409 Conflict
4. 提示"该用户权限已被其他管理员修改，请刷新后重试"
5. 页面展示最新数据，用户重新编辑后提交

### 7.4 审计日志存储满

**场景：** 审计日志数据库存储空间达到 90%。

**处理规则：**
1. 存储使用率达到 80% 时触发告警通知运维人员
2. 达到 90% 时触发严重告警
3. 系统自动触发归档策略：将超过 180 天的日志导出为压缩文件存入对象存储（Ceph RGW）
4. 归档完成后从在线存储删除已归档日志
5. 归档文件保留 3 年，可通过导入功能加载查看
6. 若归档后仍不足，暂停非关键审计日志写入（仅记录删除操作），等待扩容

### 7.5 登录失败多次锁定

**场景：** 用户 hacker 连续 5 次输入错误密码。

**处理规则：**
1. 第 1-4 次失败：返回"用户名或密码错误"，提示剩余尝试次数
2. 第 5 次失败：返回"账号已被锁定，请 30 分钟后重试"
3. 记录锁定事件到审计日志，标记为安全事件
4. 锁定期间，即使用户输入正确密码也拒绝登录
5. 锁定到期后自动解除，再次输入正确密码即可登录
6. 锁定期间，管理员可手动解锁（需审计日志记录）
7. 连续失败计数器在成功登录后清零

### 7.6 跨租户资源引用

**场景：** 租户 A 的用户尝试通过 API 直接访问租户 B 的 VM（如修改 URL 中的 tenant_id）。

**处理规则：**
1. API 中间件从 JWT Token 中提取用户的 tenant_id
2. 将请求中的 tenant_id 与 JWT 中的 tenant_id 比对
3. 不一致则返回 403 Forbidden："无权访问该租户的资源"
4. 记录该越权尝试到审计日志，标记为安全事件
5. 同一用户 1 小时内出现 3 次越权尝试，发送告警通知安全管理员

### 7.7 删除自定义角色（仍有用户绑定）

**场景：** 租户管理员尝试删除"高级开发者"角色，但仍有 8 个用户绑定该角色。

**处理规则：**
1. 系统检测角色关联用户数 > 0
2. 弹窗提示"该角色下有 8 个关联用户，不可直接删除"
3. 提供两个选项：
   - **迁移并删除**：选择目标角色 → 批量迁移用户 → 删除角色
   - **仅移除用户**：批量移除用户的所有角色绑定 → 删除角色
4. 迁移完成后记录审计日志（迁移了哪些用户、从哪个角色到哪个角色）
5. 迁移过程在事务中执行，确保原子性

### 7.8 会话过期时正在操作

**场景：** 用户正在填写创建 VM 的表单，会话已过期。

**处理规则：**
1. 前端在会话过期前 5 分钟弹出续期提醒
2. 用户可选择"续期"（重新获取 Token）或忽略
3. 若会话已过期，用户提交表单时收到 401 响应
4. 前端保存表单数据到 localStorage
5. 弹出登录对话框，用户重新登录后自动恢复表单数据
6. 登录成功后自动重放之前的请求（仅在表单未改变的情况下）

### 7.9 密码策略与弱密码

**场景：** 用户设置密码为 "Company@123"（包含公司名）。

**处理规则：**
1. 密码设置时进行弱密码检测
2. 检测规则：
   - 不可包含用户名或邮箱前缀
   - 不可包含公司名或常见单词（password/admin/root 等）
   - 不可为连续字符（abcdef/123456）或键盘序列（qwerty）
   - 不可为常见弱密码（维护弱密码黑名单库）
3. 检测到弱密码时拒绝设置，提示具体原因
4. 前端实时显示密码强度（弱/中/强），强密码才允许提交

### 7.10 LDAP 不可用时的降级登录

**场景：** 系统配置了 LDAP 认证，但 LDAP 服务器不可用。

**处理规则：**
1. LDAP 连接超时设置为 5 秒
2. 若 LDAP 不可用，降级到本地密码认证（使用用户上次同步的密码哈希）
3. 提示用户"LDAP 服务暂时不可用，已切换至本地认证"
4. 系统后台自动重连 LDAP（每 30 秒一次）
5. LDAP 恢复后自动切回
6. LDAP 不可用超过 5 分钟触发告警
7. 对于纯 LDAP 用户（无本地密码），LDAP 不可用时无法登录

### 7.11 删除平台管理员账号

**场景：** 有人尝试删除最后一个平台管理员账号。

**处理规则：**
1. 系统检测平台管理员角色下的用户数量
2. 若仅剩 1 个平台管理员，禁止删除，提示"至少需要保留 1 个平台管理员"
3. 禁止平台管理员删除自己
4. 禁止平台管理员禁用自己

### 7.12 大批量审计日志查询超时

**场景：** 查询过去 180 天所有操作日志，数据量超过 100 万条。

**处理规则：**
1. 默认查询时间范围限制为 90 天
2. 单次查询最多返回 10000 条
3. 查询超过 5 秒返回部分结果并提示"结果过多，请缩小查询范围"
4. 后端使用游标分页（search_after），避免深度分页性能问题
5. 大规模导出使用异步任务，完成后通知用户下载

**图例：**
- CRUDM = 创建(Create) + 读取(Read) + 修改(Update) + 删除(Delete) + 管理(Manage)
- CRUD = 创建 + 读取 + 修改 + 删除
- R = 仅读取
- - = 无权限

### 8.2 权限继承规则

1. **平台管理员** 拥有系统所有资源的全部权限，不受租户隔离限制
2. **安全管理员** 拥有全局安全审计和网络安全操作权限，可查看所有租户的安全组/规则/审计日志，可批量封禁 IP，但不可创建/删除网络和 CLB，不可管理用户和业务资源
3. **租户管理员** 拥有本租户所有资源的全部权限，可管理本租户的用户和角色
4. **多角色用户** 权限为所有角色权限的**并集**（最大权限原则）
5. **租户隔离** 所有非平台管理员角色均受租户边界限制（安全管理员可跨租户查看安全组和审计日志）

### 8.3 权限冲突解决

当用户拥有多个角色，且角色权限有冲突时：
- 所有角色的权限取**并集**（允许优先）
- 不支持显式的 deny 策略（简化设计，避免优先级混乱）
- 管理员可通过"移除角色"来缩小权限范围

### 8.4 特殊权限规则

| 规则 | 说明 |
|------|------|
| 仅能操作自己的资源 | 开发者创建的资源（如 VM），默认只有创建者和管理员可删除 |
| 禁止自我提权 | 用户不可修改自己的角色分配 |
| 禁止自我删除 | 用户不可删除自己的账号 |
| 安全操作二次确认 | 删除 VM/存储/租户 需输入资源名称确认 |
| 高权限操作审批 | 二期支持：删除生产环境资源需审批流程 |

## 十、验收标准

### 10.1 用户管理

- [ ] 平台管理员可创建用户并分配任意租户和角色
- [ ] 租户管理员可创建用户并仅能分配本租户角色
- [ ] 用户创建时校验用户名和邮箱唯一性
- [ ] 密码必须满足复杂度要求（8位+三类字符）
- [ ] 用户登录成功后获取 JWT Token，有效期内可正常请求 API
- [ ] 连续 5 次密码错误后账号锁定 30 分钟
- [ ] 锁定期间即使用户名密码正确也无法登录
- [ ] 30 分钟锁定到期后自动解除
- [ ] 密码过期（90 天）后登录强制跳转修改密码页
- [ ] 管理员可重置用户密码，用户下次登录强制修改
- [ ] 禁用用户后，该用户所有活跃 Token 加入黑名单，最多 60 秒内全部会话失效
- [ ] 会话 30 分钟无操作自动过期

### 10.2 角色管理

- [ ] 6 个预定义角色存在且不可删除、不可修改名称
- [ ] 租户管理员可创建本租户自定义角色
- [ ] 自定义角色可配置各资源类型的操作权限矩阵
- [ ] 删除自定义角色时，若有关联用户则提示迁移
- [ ] 角色权限变更后即时生效（不需要用户重新登录）
- [ ] 租户管理员不可创建跨租户角色

### 10.3 多租户管理

- [ ] 平台管理员可创建租户，自动初始化预定义角色
- [ ] 创建租户时自动在 Ceph 创建独立存储池
- [ ] 创建租户时自动在 K8s 创建独立 Namespace
- [ ] 可设置租户配额（VM/CPU/内存/存储/带宽）
- [ ] 配额低于已使用量时拒绝修改
- [ ] 禁用租户后，该租户用户写操作被拒绝
- [ ] 禁用租户不停止运行中的 VM
- [ ] 租户 A 的用户无法通过任何方式访问租户 B 的资源
- [ ] 平台管理员可查看所有租户资源

### 10.4 权限策略

- [ ] 用户只能执行其角色允许的操作
- [ ] 越权请求返回 403 并记录审计日志
- [ ] 平台管理员有全局权限
- [ ] 租户管理员对本租户有全部权限
- [ ] 只读用户无法执行任何写操作
- [ ] 开发者可创建 VM 但不能删除他人创建的 VM

### 10.5 操作审计

- [ ] 所有创建/修改/删除操作均记录到审计日志
- [ ] 审计日志包含完整的操作上下文（谁/何时/做什么/IP/前后快照）
- [ ] 审计日志不可删除、不可修改
- [ ] 可按用户/时间/操作类型/资源类型组合搜索
- [ ] 搜索结果响应时间 < 2 秒（10 万条以内）
- [ ] 支持导出为 CSV/JSON 格式
- [ ] 审计日志保留 180 天，过期自动归档
- [ ] 平台管理员可查看所有租户的审计日志
- [ ] 租户管理员仅可查看本租户的审计日志
- [ ] 日志详情展示操作前后 JSON 快照对比

### 10.6 登录安全

- [ ] 记录每次登录的时间/IP/设备/结果
- [ ] 异地登录（IP 归属城市变化）标记并发送邮件提醒
- [ ] 用户可查看自己最近 30 天登录历史
- [ ] 管理员可查看任意用户的登录历史
- [ ] 同一账号最多 3 个并发会话，超出时最早会话自动失效
- [ ] 租户管理员可配置本租户的登录 IP 白名单（CIDR 格式）
- [ ] IP 不在白名单内的用户被拒绝登录
- [ ] 平台管理员 IP 不受白名单限制

### 10.7 安全与合规

- [ ] 密码传输使用 HTTPS 加密
- [ ] 密码存储使用 bcrypt（cost ≥ 12）
- [ ] JWT Token 使用 RS256 签名，有效期 2 小时，含唯一 jti 标识
- [ ] Token 黑名单机制：禁用/删除/密码重置/角色变更时主动吊销 Token
- [ ] API 请求均有认证校验，无认证返回 401
- [ ] 敏感操作（删除 VM/存储/租户）需二次确认
- [ ] 审计日志数据库使用独立实例和独立账号
- [ ] 满足等保三级对身份鉴别、访问控制、安全审计的要求
