## 二、逐条评审结论

### 2.1 磁盘方案：PRD 假设 Ceph RBD vs 实际本地 qcow2

**结论：PRD 需大改，这是本次评审最严重的架构偏差。**

PRD 中镜像存储指向 CephFS（F-IMG-02）、虚拟机磁盘隐含依赖 Ceph RBD（附录术语表明确列出），但实际环境是 `/home/kvm/images/` 本地 qcow2 文件。这导致以下影响：

- **虚拟机创建流程**：PRD 的"镜像模板克隆系统盘"逻辑在 Ceph RBD 下是秒级 COW 克隆，在本地 qcow2 下需要物理复制 qcow2 backing file + 创建 overlay，耗时与磁盘大小成正比，3-5 分钟的创建时间承诺无法保证。
- **迁移**：PRD F-VM-19 标记为 P2 且需关机，这在本地 qcow2 下是正确的（需要复制磁盘文件到目标宿主机），但 PRD 未描述迁移时磁盘文件传输机制、增量迁移策略、以及 `/home/kvm/` 路径在跨节点时的一致性问题。
- **快照**：PRD 的快照方案基于 libvirt snapshot API（F-VM-13），这在本地 qcow2 下可行（snapshot 创建为 qcow2 内部快照或外部快照文件），但需明确：外部快照会产生快照链文件，PRD 第 308 行提到"合并数据到子快照"是正确的，但需补充 blockcommit 操作的详细说明。
- **克隆**：PRD F-VM-16 写"复制源虚拟机磁盘镜像"，在本地 qcow2 下是正确的，但未考虑 qcow2 backing chain 的处理——克隆时需要 flatten 整个 backing chain 还是保留 backing file 引用？

**必须修改**：全文将所有 Ceph RBD/CephFS 的存储假设替换为本地 qcow2 文件方案，明确各宿主机上 `/home/kvm/images/`、`/home/kvm/cloud-init/images/`、`/home/kvm/kvm_snap/`、`/home/kvm/kvm_backup/` 四个目录的用途和生命周期管理。

### 2.2 cloud-init 集成：PRD 覆盖不足

**结论：PRD 有提及但细节严重缺失，需补充完整的 cloud-init 创建流程设计。**

- **已有覆盖**：第 437 行提到"注入初始化配置（cloud-init 或 QGA）"，第 441-446 行列出创建后自动操作（设置主机名、配置网络 IP、注入 SSH 公钥、安装 QGA），方向正确。
- **缺失内容**：
  1. **镜像准备流程**：PRD 未说明 cloud-init 镜像如何准备——是预先在 `/home/kvm/cloud-init/images/` 中存放已安装 cloud-init 的 qcow2 模板镜像，还是上传 ISO 手动安装后再制作为模板？这与 F-IMG-02（上传 ISO）和 F-IMG-06（实例制镜像）的流程衔接不清晰。
  2. **user-data 注入机制**：第 437 行写"cloud-init 或 QGA"，但这两者是完全不同的机制。cloud-init 通过 ISO 挂载（cloud-init NoCloud datasource）或 nocloud 网络源注入，QGA 是在 VM 启动后通过 QMP 命令注入。PRD 应明确采用 **NoCloud ISO 方式**（生成含 meta-data/user-data 的 ISO 并挂载为 cdrom），这是本地 qcow2 环境下最可靠的做法。
  3. **SSH key 配置**：PRD 第 444 行仅写"注入 SSH 公钥（如有配置）"，未说明 SSH key 的来源——是用户创建时输入、从平台密钥对管理模块获取、还是使用租户默认密钥？user-data 中 `ssh_authorized_keys` 的写入格式需要明确。
  4. **网络配置**：第 443 行"配置网络 IP"过于笼统。cloud-init 下网络配置通过 user-data 中的 `network-config` 或 `write_files` + `runcmd` 实现，需明确是使用 cloud-init 原生网络配置 v1/v2 还是通过自定义脚本。
  5. **主机名规范**：第 442 行"设置主机名"，需明确主机名来源（用户输入 vs 自动生成），以及 cloud-init 中 `hostname` / `fqdn` 字段的对应关系。
  6. **错误处理**：cloud-init 初始化失败时（如网络不通导致 apt/yum 源不可达），VM 将启动但未完成初始化，PRD 未定义这种中间状态的检测和处理方式。

**必须修改**：新增"cloud-init 集成设计"专节，详细描述 NoCloud ISO 生成流程、user-data 模板结构、SSH key 注入、网络配置方式、初始化状态检测机制。

### 2.3 存量兼容：只读纳管 32 台存量 VM

**结论：方案基本可行，但存在 3 个需要修正的细节问题。**

PRD 第 397-406 行的存量纳管设计思路正确——通过 libvirt API 读取 domain XML、只读标记、限制修改操作。但结合实际环境需要修正：

- **UUID 匹配风险**：第 463 行"按 UUID 匹配"是正确做法，但需注意：如果存量 VM 是通过 `virt-clone` 或手动复制 qcow2 文件 + 修改 XML 创建的，可能出现 UUID 冲突。建议增加 **MAC 地址作为辅助匹配键**。
- **磁盘路径发现**：存量 VM 的 domain XML 中磁盘 source file 指向 `/home/kvm/images/xxx.qcow2`，纳管时需要解析此路径、校验文件是否存在、读取 qcow2 文件信息（虚拟大小、实际占用、backing file），并将这些信息写入平台数据库。PRD 第 466 行只写"读取 domain XML 获取配置信息（vCPU、内存、磁盘、网络）"，未细化磁盘发现步骤。
- **存量 VM 的 cloud-init 状态未知**：32 台存量 VM 的初始化方式可能不一致（部分手动安装、部分 cloud-init），纳管后平台无法保证这些 VM 与平台创建的 VM 在配置管理上的一致性。建议在纳管报告中标注"初始化方式未知"。
- **只读转平台管理的风险**：第 405-406 行允许将只读纳管转为平台管理模式。转换后平台将接管生命周期管理，但需注意：转换前平台必须对存量 VM 的磁盘文件、快照链、备份文件进行全面健康检查，防止接管后因磁盘文件状态异常导致操作失败。

**建议优化**：补充磁盘路径解析和 qcow2 文件信息采集的具体步骤，增加 MAC 地址辅助匹配，定义"只读转平台管理"的前置条件检查清单。

### 2.4 资源超分：CPU 4:1、内存不超分

**结论：方案在混部场景下风险较高，建议调整为更保守的比例并增加动态监控。**

PRD 第 385 行"CPU 超分比 4:1，内存 1:1（不超分）"：

- **CPU 4:1 超分评估**：6 台 R740 同时承载 KVM、Ceph OSD、K8s worker 三种负载。Ceph OSD 对 CPU 消耗不小（尤其在 recovery/backfill 时），K8s 容器化业务也有持续 CPU 消耗。实际可用于 KVM 的 CPU 资源需要扣除 Ceph + K8s 的基线消耗后才能计算超分比。建议先采集各节点 CPU 使用率基线数据（至少 7 天），然后按"（物理核数 - Ceph基线 - K8s基线）× 超分比"计算可用 vCPU。盲目 4:1 可能导致 Ceph 性能下降或 VM CPU steal time 飙升。
- **内存不超分（1:1）**：方向正确，但同样需要扣除 Ceph OSD 内存占用（通常建议每 TB OSD 预留 1GB 内存）、K8s 节点预留、以及系统自身开销。实际可分配内存 = 物理内存 - Ceph预留 - K8s预留 - 系统预留（建议至少 8GB/节点）。
- **缺乏超分安全边际**：PRD 第 387-388 行"超分比设置导致已分配超过物理×超分比时仅警告不阻止"，这在混部环境下过于宽松。建议增加**硬限制**：当 Ceph 集群处于 recovery 状态或 K8s 节点负载超过阈值时，自动暂停新 VM 创建，防止资源争抢导致所有服务雪崩。
- **规格过滤逻辑不完整**：第 202 行"规格资源需求超过宿主机可用资源时禁用"，但"可用资源"的计算公式未定义——是否已扣除混部负载基线？是否按超分比换算？这直接影响创建向导的资源展示准确性。

**必须修改**：明确混部环境下的可用资源计算公式（扣除 Ceph/K8s 基线），CPU 超分比建议下调至 2:1~3:1（待基线数据确认），增加混部负载感知的动态创建控制。

### 2.5 快照和备份：与现有目录的整合

**结论：PRD 快照方案与现有目录结构存在冲突，需重新设计整合方案。**

现有环境有 `/home/kvm/kvm_snap/` 和 `/home/kvm/kvm_backup/` 两个独立目录，而 PRD 仅基于 libvirt snapshot API 设计快照功能：

- **libvirt snapshot 的文件位置**：libvirt 外部快照（`--disk-only`）创建的 qcow2 快照文件默认位于虚拟机磁盘同目录（即 `/home/kvm/images/`），或由 XML 中指定的路径决定。现有 `/home/kvm/kvm_snap/` 目录说明团队已有手动快照管理实践，需要确认：存量快照是如何创建的（`virsh snapshot-create` vs 手动 `qemu-img create`）？是内部快照还是外部快照？
- **备份目录未纳入 PRD**：`/home/kvm/kvm_backup/` 目录在 PRD 中完全没有提及。PRD 功能清单中无"备份管理"功能。建议至少增加 **P1 级别的备份查看功能**（列出存量备份文件、关联到 VM、支持恢复操作），否则纳管后存量备份文件将成为"盲区"。
- **快照存储路径统一**：PRD 若继续使用 libvirt snapshot API，建议统一将快照文件写入 `/home/kvm/kvm_snap/`，与现有实践保持一致。需要设计存量快照的发现和导入机制——通过解析 snapshot XML 或扫描 qcow2 backing file chain 来识别已存在的快照关系。
- **备份方案缺失**：现有 `/home/kvm/kvm_backup/` 目录暗示团队已有备份实践（可能是 `qemu-img convert` 或 `cp` 产生的完整磁盘副本）。PRD 应补充备份管理功能（备份列表、从备份恢复、备份清理策略），否则平台上线后备份操作将退化为 SSH 手动操作。

**必须修改**：增加"备份管理"功能模块（至少 P1 查看+恢复），明确快照文件存储路径统一到 `/home/kvm/kvm_snap/`，设计存量快照和备份文件的发现与导入机制。

### 2.6 创建流程：5 步向导完整性

**结论：流程框架完整，但缺少 cloud-init 专属配置步骤，且磁盘配置缺失。**

PRD 第 180-225 行的 5 步创建流程：

- **缺失步骤：cloud-init 配置**。当前 5 步为"选宿主机→选镜像→选规格→选网络→确认"。在使用 cloud-init 模板镜像时，缺少关键的配置步骤：
  - SSH 公钥注入（从密钥对列表选择或粘贴）
  - 主机名设置
  - 用户自定义初始化脚本（可选，PRD 第 446 行提到但无对应 UI）
  - 建议将 Step 3（选规格）与 cloud-init 配置合并为"选规格与初始化配置"，或增加 Step 2.5 "cloud-init 配置"子步骤。
- **系统盘大小不可配**：第 215 行汇总信息包含"系统盘大小"，但前序步骤中没有任何地方让用户指定系统盘大小。实际需求中，用户可能需要选择大于镜像模板的系统盘（如模板 20GB，实际需要 100GB）。PRD 需在 Step 3 或 Step 5 中增加系统盘大小配置。
- **数据盘配置缺失**：第 215 行提到"数据盘配置（如有）"，但整个 5 步向导中没有数据盘创建步骤。需增加"添加数据盘"的可选步骤或子步骤（选择容量、存储池位置）。
- **镜像与 cloud-init 兼容性校验缺失**：第 195 行"选择 ISO 时提示手动安装"、第 196 行"选择模板时提示自动初始化"，但未校验模板镜像是否已安装 cloud-init。如果用户选择了一个未预装 cloud-init 的 qcow2 模板，创建后初始化将静默失败。

**必须修改**：增加 cloud-init 配置步骤（SSH key、主机名、自定义脚本），增加系统盘大小配置，增加可选的数据盘创建步骤，增加镜像 cloud-init 兼容性校验。

### 2.7 技术风险识别

**结论：识别以下 5 个关键技术风险。**

| 编号 | 风险描述 | 影响程度 | 建议措施 |
|------|---------|---------|---------|
| RISK-01 | **本地 qcow2 单点故障**：虚拟机磁盘文件存储在单台宿主机本地，宿主机故障将导致该节点所有 VM 不可用，且无法像 Ceph RBD 那样在其他节点快速拉起。6 台 R740 任一台宕机，影响面为 ~1/6 的 VM 容量。 | **高** | 短期：加强宿主机硬件监控和备件储备；中期：实现 qcow2 文件定时增量备份到其他节点或 CephFS；长期：评估是否迁移到 Ceph RBD。 |
| RISK-02 | **Ceph + KVM + K8s 混部资源争抢**：Ceph OSD recovery/backfill、K8s 容器突发流量、KVM 虚拟机负载三者叠加时，可能导致磁盘 IO 饥饿（Ceph 和 KVM 共享磁盘控制器）或 CPU 饥饿（所有服务争抢同一物理核）。 | **高** | 实施 cgroup 资源隔离（CPU set、内存 limit、IO 权重），配置 Ceph OSD 优先级和恢复限速，建立资源争抢的实时检测和告警机制。 |
| RISK-03 | **qcow2 backing chain 过长导致性能衰减**：cloud-init 模板镜像 → VM overlay → 快照 overlay → 快照 overlay ... 链过长后，每次 IO 需要穿透多层 qcow2，磁盘性能线性下降。PRD 第 308 行提到快照链管理但仅限制数量（20），未考虑合并策略。 | **中** | 实现自动 blockcommit 策略：快照链深度超过 5 层时自动合并（业务低峰期），或定期将运行时间超过 N 天的 VM 进行 backing file flatten。 |
| RISK-04 | **存量 VM 纳管的配置漂移**：32 台存量 VM 的 domain XML 可能包含非标准配置（手动修改的 CPU pinning、自定义设备、非标准网络桥接），纳管后平台对这些配置无感知，后续操作（如配置变更）可能破坏这些手动配置。 | **中** | 纳管时对 domain XML 进行完整性扫描和报告，标记非标准配置项，限制对此类 VM 的配置变更操作。 |
| RISK-05 | **cloud-init 初始化失败无感知**：VM 启动后，cloud-init 在 guest 内部执行初始化（网络配置、软件安装、脚本执行），如果初始化过程中某个步骤失败（如 yum 源不可达），libvirt 层面无法感知——VM 状态显示"运行中"但实际未完成初始化。 | **中** | 实现 cloud-init 状态检测机制：在 VM 启动后通过 QGA 或轮询 SSH 端口检测 cloud-init 是否完成（`cloud-init status`），超时未完成则标记为"初始化异常"状态。 |

## 评审总览

PRD 整体质量较高，结构完整，EARS 格式规范，边界场景考虑充分。但存在一个**关键定位偏差**——块存储模块的假设与实际情况严重不符，需要重大调整。以下逐条分析。

## 1. 块存储定位

### 现状分析

| 维度 | PRD 假设 | 实际情况 |
|------|---------|---------|
| VM 磁盘 | 基于 Ceph RBD | 本地 qcow2 文件（`/home/kvm/images/`） |
| 块存储驱动 | `rbd map` + libvirt attach | `virsh attach-disk` 使用本地路径 |
| 云盘定位 | 对标阿里云 EBS | 实际上 KVM 从未使用 Ceph RBD |

### 评审结论

**PRD 中块存储模块的定位需要做重大调整，当前方案与现状完全脱节。**

#### 问题拆解

1. **块存储模块存在的基础被架空**

   PRD 将块存储定位为"虚拟机持久化高性能块设备"，对标阿里云 EBS，并设计了完整的"创建云盘 → 挂载 VM → 扩容 → 通知 OS 识别"流程（REQ-BLK-001 ~ REQ-BLK-010）。但实际情况是：**所有 KVM 虚拟机磁盘使用本地 qcow2 文件**，Ceph RBD 从未被用作 VM 主存储。

   这意味着 PRD 中块存储模块的全部用户故事（US-03 "为虚拟机挂载云盘"）和 12 个功能点（BLK-01 ~ BLK-12）在当前的用户场景中**无对应需求**。

2. **仅有 `kvm-backup` pool 含 251 GiB RBD 数据**

   集群中唯一的 RBD pool 是 `kvm-backup`，251 GiB。其用途需要进一步确认（见第 2 条），但无论如何，这个量级远不足以支撑一个"云盘全生命周期管理"模块。

3. **存储池分层的前提不存在**

   PRD 在 REQ-BLK-003 中设计了"存储池选择"功能（ssd-pool / hdd-pool / archive-pool 三级分层），但实际集群**只有 8 块 HDD OSD**，不存在 SSD Pool 或归档 Pool。这个交互设计完全是空想。

4. **`virsh blockresize` 流程不适用于 qcow2**

   PRD 中扩容流程依赖 `rbd resize` + `virsh blockresize` / QEMU Guest Agent（REQ-BLK-007），这是 RBD 场景的流程。对于本地 qcow2 文件，扩容流程完全不同（`qemu-img resize` + VM 内部 resize2fs/xfs_growfs），需要重新设计。

### 调整建议

#### 方案 A：重新定位块存储模块（推荐）

将块存储模块从"Ceph RBD 云盘管理"调整为 **"虚拟机存储管理"**：

- **P0 核心功能**：展示 VM 的 qcow2 磁盘列表（路径、大小、实际占用、格式）、磁盘使用率监控、空间告警
- **P1 扩展功能**：qcow2 磁盘在线扩容（`qemu-img resize` + Guest Agent）、快照管理（qcow2 snapshot，非 RBD snapshot）
- **P2 远期规划**：Ceph RBD 作为 VM 存储后端的迁移方案（需与虚拟化团队协商）

#### 方案 B：明确 Ceph RBD 当前仅用于备份

如果团队确认 `kvm-backup` pool 仅用于备份场景，则：

- 将块存储模块降级为 **"备份存储管理"** 子模块
- 删除"挂载到虚拟机"、"扩容通知 OS"等流程
- 保留：RBD Image 列表查看、容量监控、快照管理（用于备份版本管理）

#### 方案 A vs B 对比

| 维度 | 方案 A（VM 存储管理） | 方案 B（备份存储管理） |
|------|----------------------|----------------------|
| 业务价值 | 高（解决真实痛点） | 中（功能单一） |
| 开发成本 | 中等（需适配 qcow2） | 低（大幅裁剪 PRD） |
| 远期扩展性 | 好（可平滑迁移到 RBD） | 差（后续需重构） |
| PRD 改动量 | 大（块存储全量重写） | 中（删除 VM 相关流程） |

**建议采用方案 A**，因为"虚拟机磁盘空间不足"才是运维日常的真实痛点，Ceph RBD 的云盘管理在无 VM 使用场景下是伪需求。

## 2. kvm-backup Pool

### 现状分析

- Pool 名称：`kvm-backup`
- 数据量：251 GiB
- 类型：副本池（3 副本）
- 关联：RBD（含 RBD Image）

### 评审结论

**Pool 命名表明其用途是 KVM 虚拟机备份，PRD 需要根据实际用途调整备份策略设计。**

#### 关键判断

1. **251 GiB 的数据量**说明已有一定规模的备份数据在此 pool 中。需要确认：
   - 备份方式：是 RBD Image 直接写入备份数据，还是 VM 通过某种备份工具将数据写入此 pool？
   - 备份频率和保留策略是什么？
   - 是否有自动清理机制？

2. **PRD 中未涉及备份策略设计**

   PRD 仅在基础设施概况表格中提到了 `kvm-backup` pool（1.3 节），但在功能清单和详细需求中完全没有针对此 pool 的专门设计。这是一个明显的遗漏。

3. **PRD 中块存储模块的快照功能（RBD snap）≠ 备份**

   PRD 设计的"云盘快照"（REQ-BLK-008 ~ REQ-BLK-010）是 RBD 快照，属于存储层的 Crash-Consistent Snapshot，与 `kvm-backup` pool 中可能存在的应用级备份是不同层面的东西。

### 调整建议

1. **明确 `kvm-backup` pool 的实际用途**

   需要运维团队确认：
   - 251 GiB 数据的具体内容（备份工具？手动导出的 qcow2 文件？）
   - 备份策略（增量/全量、保留周期）
   - 是否期望通过平台统一管理备份策略

2. **在 PRD 中新增"备份管理"子模块**

   至少包含以下功能：
   - 备份任务列表查看（来源 VM、备份时间、大小）
   - 备份空间使用趋势
   - 备份保留策略配置（如保留最近 N 份备份）
   - 备份过期自动清理
   - 备份恢复操作（恢复到指定 VM 或导出下载）

3. **将 PRD 中块存储的"快照"功能与备份功能区分**

   - **RBD 快照**：存储层的即时快照，用于快速回滚（保留在块存储模块）
   - **备份管理**：应用层的数据备份，涉及 `kvm-backup` pool（新建模块）

## 3. 对象存储方案

### 现状分析

- RGW 已启用，825 GiB 数据
- 4 个 RGW 用户：`manager-images`、`dashboard`、`inner-img`、`ark-video-backup`
- S3-compatible API 已可用

### 评审结论

**对象存储方案整体合理可行，PRD 覆盖了 RGW 管理的核心场景。但存在以下 5 个需要关注的问题。**

#### 问题 1：ACL 设计过于简化（中风险）

PRD 的 ACL 设计（REQ-OBJ-015）仅支持三个级别：私有、公共读、公共读写。这是**Bucket-level ACL**，粒度太粗。

实际 RGW 支持：
- Bucket Policy（JSON 策略，类似 AWS IAM Policy，可按用户/IP/操作精细化控制）
- IAM User Policy
- 对象级 ACL

**建议**：
- P0 保留三个预设级别（快速配置入口）
- P1 增加"高级策略编辑器"（直接编辑 Bucket Policy JSON），满足精细化权限需求
- 公共读写选项增加红色高危警告（PRD 已有提及，需强化）

#### 问题 2：RGW User 与平台用户的映射关系未定义（高风险）

PRD 中"创建 Bucket 后自动生成 AccessKey/SecretKey"（REQ-OBJ-005）暗示：**每个 Bucket 对应一个 RGW User**。

这与 RGW 的实际模型不一致：
- RGW User 是顶层实体，一个 User 可以拥有多个 Bucket
- 现有 4 个 RGW User 已经存在，每个 User 下可能有多个 Bucket
- 如果每个 Bucket 创建一个 RGW User，会导致 User 数量膨胀，管理混乱

**建议**：
- 明确 RGW User 的创建策略：每个**项目**对应一个 RGW User（而非每个 Bucket）
- 在该 User 下创建多个 Bucket，通过 Bucket Policy 实现项目内隔离
- 或者：每个**应用**对应一个 RGW User（更细粒度）
- 文档中补充 RGW User 与平台项目的映射关系图

#### 问题 3：用量统计的数据来源不明确（低风险）

PRD 要求展示"最近 24 小时请求次数趋势（GET/PUT/DELETE 分类）"（REQ-OBJ-007），但 RGW 原生不直接提供按 Bucket 维度的请求分类统计。

**建议**：
- 使用 RGW Usage Log（`radosgw-admin usage show`）获取用量数据
- 或启用 RGW 的 Bucket Logging 功能，将访问日志写入专门的 log bucket
- 或通过 Ceph exporter + Prometheus 采集 RGW 指标
- 在 PRD 中标注数据采集方案选型

#### 问题 4：文件上传大小限制过于乐观（中风险）

PRD 设定单文件最大 100 GiB（边界场景 1），但这是 RGW 配置的理论上限。在 8 块 HDD、混部环境的实际场景下：
- 100 GiB 文件通过 Web 上传需要数小时，期间网络波动、浏览器崩溃风险高
- 分片上传（10 MiB 分片）意味着 100 GiB 需要 10240 个分片，管理开销大

**建议**：
- Web 界面上传限制为 5 GiB（行业常见上限）
- 超过 5 GiB 的文件引导用户使用 S3 CLI/SDK 工具上传
- 或者 P1 实现"断点续传 + 分片上传"作为 Web 大文件上传方案

#### 问题 5：文件预览的技术边界不清晰（低风险）

PRD 要求"支持常见格式在线预览（图片、文本、PDF、视频）"（OBJ-12，P1），但没有定义：
- 预览是通过生成预签名 URL 直接在浏览器打开，还是后端转码后展示？
- 视频预览是否需要转码？（850 MiB 的 MP4 文件直接加载不现实）
- PDF 预览是否需要后端渲染？

**建议**：
- 图片/文本：生成预签名 URL，浏览器直接渲染（零成本）
- PDF：可集成 pdf.js 或后端转图片
- 视频：P1 仅支持小视频（< 50 MiB）的 HTML5 播放，大视频提供下载链接；P2 可考虑集成转码服务

## 4. Ceph 集群管理

### 现状分析

- 6 节点、5 MON + 3 MGR、8 OSD
- 单集群，规模不大
- Ceph Dashboard 已内置（Reef 版本自带）

### 评审结论

**自研 Dashboard 有其价值，但需明确与 Ceph 原生 Dashboard 的分工和权限边界。**

#### 4.1 Dashboard 嵌入 vs 自研

| 维度 | Ceph 原生 Dashboard | 自研 Dashboard |
|------|-------------------|----------------|
| 集群健康 | 原生支持，实时 | 需自建数据采集 |
| 性能图表 | 内置 Prometheus + Grafana | 需自建时序存储和图表 |
| 告警 | 内置 Alertmanager | 需自建告警引擎 |
| 用户体系 | 简单的 local user | 可与企业 IAM 集成 |
| 多集群 | 不支持 | PRD 设计支持（P2） |
| 操作审计 | 无 | PRD 设计了完整审计 |
| 审批流程 | 无 | PRD 设计了高危操作审批 |
| 中文界面 | 部分支持 | 完全自定义 |

**建议：分层策略**

| 层级 | 功能 | 方案 |
|------|------|------|
| **展示层** | 集群健康、容量、OSD/Pool 状态 | 自研 Dashboard（与企业 UI 统一） |
| **数据层** | IOPS、带宽、延迟等性能指标 | 复用 Ceph Prometheus Exporter + Prometheus + Grafana（嵌入 iframe），自研负责告警规则 |
| **操作层** | OSD out/in、Pool 创建/删除 | 自研封装 + 审批流程 |
| **管理入口** | 高级排障 | 提供"Ceph 原生 Dashboard"链接（SSO 跳转），保留原生工具作为兜底 |

**核心理由**：8 块 OSD 的单集群，自研全套性能监控（时序数据库 + 图表渲染 + 数据采集）的 ROI 很低。直接嵌入 Grafana iframe 是更务实的方案，将开发资源投入到审批流程、审计日志、IAM 集成等差异化能力上。

#### 4.2 权限边界评估

PRD 的权限矩阵（8.2 节）设计合理，但有以下建议：

| 操作 | PRD 设计 | 建议调整 | 理由 |
|------|---------|---------|------|
| 删除 OSD | 审批后执行 | **保持** | 正确，OSD 删除不可逆 |
| 删除 Pool | 审批后执行 | **保持** | 正确，需安全校验 + 关联资源检查 |
| OSD out/in | 存储管理员直接执行 | **增加操作窗口限制** | out 操作会触发数据重平衡，建议限制在业务低峰期（如凌晨 2-6 点）执行 |
| 修改 PG 数 | 仅允许增加 | **增加变更窗口限制** | PG 调整会触发大量数据迁移，同样需要低峰期窗口 |
| 创建 Pool | 存储管理员直接执行 | **保持** | 创建操作对集群影响较小 |
| MON 节点管理 | 仅查看 | **保持** | MON 操作应在原生环境执行 |

#### 4.3 OSD 故障处理设计评估

REQ-CLS-006（OSD 磁盘故障处理）的"一键隔离"按钮设计是亮点，但需注意：

- **"OSD Down > 5 分钟 → 高亮闪烁 + 一键隔离"** 是合理的。但需考虑：短暂网络抖动也可能导致 OSD Down 5 分钟，不一定代表磁盘故障
- **建议增加**：在"一键隔离"前展示 OSD 的 SMART 信息（如果可获取），帮助判断是否为硬件故障
- **自动 out 风险**：不建议自动执行 out 操作，因为 out 会触发数据重平衡，在错误判断的情况下代价很高。保持"手动确认"是正确的设计

## 5. 性能影响

### 现状分析

- 6 节点同时运行 KVM 虚拟机（qcow2）和 Ceph（OSD/MON/MGR/RGW）
- 8 块 HDD OSD，29 TiB 总容量
- Ceph 使用率仅 10.89%，负载较轻

### 评审结论

**当前负载下影响可控，但需在 PRD 中补充混部环境的操作约束。**

#### 5.1 各操作的影响评估

| 操作 | 对 KVM 影响 | 对 Ceph 影响 | 风险等级 |
|------|-----------|------------|---------|
| **Dashboard 查看** | 几乎为零 | 几乎为零（`ceph status` 是轻量操作） | 无风险 |
| **OSD out/in** | 中等（backfill/recovery 占用磁盘 IO 和网络） | 高（数据迁移期间性能下降） | **中风险** |
| **调整 PG 数** | 中等（同上） | 高（大量数据迁移） | **中风险** |
| **创建 Pool** | 几乎为零 | 低 | 低风险 |
| **RBD 快照** | 几乎为零 | 低（元数据操作） | 低风险 |
| **RGW 大文件上传** | 中等（占用网络带宽） | 中（占用 OSD IO） | 低风险 |
| **性能数据采集** | 几乎为零 | 低（60 秒间隔合理） | 无风险 |

#### 5.2 混部环境的关键风险

1. **OSD 操作触发 backfill/recovery 时**，HDD 的 IO 会被数据迁移大量占用，可能影响：
   - KVM 虚拟机的 qcow2 磁盘 IO（如果 qcow2 文件在同一个物理磁盘上）
   - RGW 的对象读写响应延迟

2. **磁盘带宽竞争**：8 块 HDD 在 backfill 期间，每个 OSD 的 IO 能力会显著下降

#### 5.3 调整建议

1. **PRD 中增加"混部环境操作约束"章节**

   ```
   - OSD out/in 操作限制在每日 02:00-06:00 窗口执行
   - PG 调整操作限制在周末低峰期执行
   - 执行数据迁移操作前，系统自动检测当前 IOPS 和带宽使用率，
     如果超过正常水平的 50%，延迟执行
   - 创建大量 RBD Image / 上传大量对象时，增加并发限流
   ```

2. **增加 backfill/recovery 的限速控制**

   在 Pool 配置中建议设置：
   ```
   osd_max_backfills = 1
   osd_recovery_max_active = 1
   osd_recovery_op_priority = 1
   ```
   在混部环境下，这些参数可以防止数据迁移"饿死"正常业务 IO。PRD 中可以增加"集群调优建议"章节。

3. **监控混部影响**

   在 Dashboard 中增加"节点级别的 CPU/内存/磁盘 IO/网络"指标（当前 PRD 只关注 Ceph 集群级别的指标），帮助发现混部资源竞争。

## 6. 存量 RGW 用户

### 现状分析

| RGW User | 推测用途 | 数据量 |
|----------|---------|--------|
| `manager-images` | 管理后台图片存储 | 未知 |
| `dashboard` | Dashboard 相关资源 | 未知 |
| `inner-img` | 内部图片服务 | 未知 |
| `ark-video-backup` | 方舟视频备份 | 未知 |

4 个用户共 825 GiB 数据。

### 评审结论

**存量 RGW 用户需要"纳管"而非"迁移"，但 PRD 的存量纳管流程（5.3 节）对 RGW 用户的处理过于笼统。**

#### 6.1 关键问题

1. **RGW User 与 Bucket 的关系需要扫描清楚**

   存量纳管流程（5.3 节 Step 3）提到"扫描 RGW Bucket（关联用户、用量）"，但没有说明如何建立 Bucket 与 User 的关联关系。需要补充：
   - 扫描每个 RGW User 下有哪些 Bucket
   - 扫描每个 Bucket 的 ACL / Policy
   - 将存量 Bucket 归属到对应的 RGW User

2. **AccessKey/SecretKey 已在外部使用**

   4 个现有 RGW User 的 AccessKey/SecretKey 可能已经被业务系统硬编码使用。如果平台"吊销"或"重新生成"这些凭证，会导致业务中断。

   **PRD 中的凭证管理（REQ-OBJ-016）需要明确**：
   - 存量 User 的凭证**仅查看，不可吊销**（或吊销前需确认业务影响）
   - 吊销操作增加"影响范围分析"：展示使用该凭证的 Bucket 列表和最近的访问记录

3. **纳管后的权限归属**

   存量 RGW User 对应的 Bucket 需要分配归属项目：
   - `ark-video-backup` → 方舟项目
   - `inner-img` → 内部图片服务项目
   - `manager-images` → 管理后台项目
   - `dashboard` → 运维项目

   分配后，项目成员才能看到对应 Bucket（符合 PRD 的项目隔离设计）。

#### 6.2 调整建议

1. **在存量纳管流程中增加 RGW User 专项步骤**

   ```
   Step 3.5: RGW 用户专项扫描
   - 列出所有 RGW User（radosgw-admin user list）
   - 获取每个 User 的详细信息（radosgw-admin user info --uid=xxx）
   - 建立 User → Bucket 的关联映射
   - 记录每个 User 的 AccessKey 列表（不暴露 SecretKey）
   - 标记 User 类型：存量纳管 / 平台创建
   ```

2. **增加"RGW 用户管理"页面**

   当前 PRD 只有 Bucket 列表，没有 RGW User 列表。建议增加：
   - RGW User 列表（用户 ID、显示名称、所属项目、Bucket 数量、用量汇总、创建时间）
   - 查看 User 详情（关联 Bucket、AccessKey 列表（脱敏）、配额、权限策略）
   - 创建平台 RGW User（自动关联到项目）

3. **迁移策略**

   - **不需要迁移数据**：存量 Bucket 中的 825 GiB 数据保持不动
   - **不需要迁移凭证**：存量 User 的 AK/SK 保持不变，业务不受影响
   - **需要纳管元数据**：将 User-Bucket 关联关系导入平台数据库，建立归属关系
   - **新增 Bucket 使用平台创建**：平台新建 Bucket 时创建新的 RGW User（或复用项目级 RGW User）

## 7. 容量规划

### 现状分析

| 指标 | 当前值 |
|------|--------|
| 总容量 | 29 TiB（8 × 3.6 TiB HDD） |
| 已用量 | 3.2 TiB（10.89%） |
| 剩余容量 | 25.8 TiB |
| 增长趋势 | 未知（需确认历史增长速率） |
| OSD 扩容能力 | 6 节点可继续增加 OSD |

### 评审结论

**当前容量充裕，但 PRD 的告警阈值设置需要调整，容量规划需要补充。**

#### 7.1 告警阈值分析

PRD 设计的容量告警阈值（REQ-CLS-003 / REQ-CLS-014）：

| 阈值 | PRD 行为 | 评估 | 建议 |
|------|---------|------|------|
| 80% | 容量环图变黄色 | **偏高** | 建议降为 **70%** 黄色预警（HDD 扩容周期长，需提前规划） |
| 90% | 容量环图变橙色 | **合理** | 保持，触发邮件通知 |
| 95% | 环图变红色 + 持久化横幅 | **合理** | 保持，触发紧急告警 + 限制写操作 |

**调整理由**：
- HDD 集群扩容涉及硬件采购、上架、OSD 部署、数据重平衡，全流程可能需要数周
- 70% 预警给运维团队留出充足的采购和部署时间
- Ceph 在容量超过 85% 时可能出现性能下降（nearfull ratio 默认为 85%）

#### 7.2 扩容路径评估

| 扩容方式 | 可行性 | 影响 | 建议 |
|---------|--------|------|------|
| 增加 HDD OSD | **可行** | 数据重平衡（backfill），持续数天 | 首选方案，6 节点均有空余盘位 |
| 替换大容量 HDD | 可行但风险高 | 逐个 OSD 替换，每块需完整重平衡 | 备选方案 |
| 增加节点 | 可行 | 需部署 MON/OSD/MGR，周期较长 | 长期方案 |
| 增加 SSD OSD（分层） | 可行 | 需创建 SSD Pool + CRUSH rule | P2 规划，可显著提升 RGW 小文件性能 |

#### 7.3 容量规划补充建议

PRD 中缺少容量规划功能，建议增加：

1. **容量趋势预测**

   基于历史用量数据（90 天），展示：
   - 过去 90 天用量增长趋势
   - 预计容量耗尽可能日期（基于线性/指数回归）
   - 按 Pool 维度的增长趋势分解

2. **OSD 扩容模拟**

   - 输入"新增 N 块 HDD（每块 X TiB）"，系统估算：
     - 扩容后总容量
     - 预计数据重平衡时长
     - 扩容后各 OSD 的使用率分布

3. **容量报表**

   - 按项目维度展示对象存储/块存储的用量趋势
   - 支持导出月度/季度容量报告

4. **Pool 级别的容量告警**

   当前 PRD 只有集群级别的容量告警（80%/90%/95%），建议增加：
   - 单个 Pool 使用率超过 70% → 警告
   - 单个 Pool 使用率超过 85% → 严重告警

## 一、网络模块评审

### 1.1 Linux Bridge + VLAN 方案：是否与现有环境兼容？有没有更好的方案？

**结论：方案兼容但存在显著局限，建议 P1 阶段迁移至 OVN/OVS。**

**兼容性分析：**

| 维度 | 评审结论 |
|------|---------|
| 现有环境兼容 | Linux Bridge 与现有 32 台 VM 的 KVM + qcow2 环境完全兼容，无需改造存量。Bridge 命名规范 `vn-{id前8位}` 避免了与现有 Bridge 冲突。 |
| 创建/销毁操作 | PRD 对 Bridge 的创建、NAT 规则配置、dnsmasq 启停、回滚机制描述清晰（4.1.2 节）。 |
| 存量共存 | PRD 1.3 节明确"不破坏现有 32 台 KVM 虚拟机的网络配置"，通过新建 Bridge + 路由互通实现隔离，策略正确。 |

**风险与不足：**

1. **缺少 VLAN 隔离机制**：PRD 中未提及 VLAN ID 的分配与管理。当前方案依赖纯 Bridge 隔离，二层域之间通过"不配置 NAT/路由"来实现隔离，这在高密度租户场景下存在 ARP 广播风暴风险。NAT 网络和隔离网络仅靠 Bridge 隔离，无 VLAN 标签，所有 Bridge 的广播域在物理网卡层面仍可能互相影响。
2. **跨宿主机通信**：PRD 在 TI-09 提到 agent 通过 gRPC 通信，但未说明不同宿主机上属于同一虚拟网络的 VM 之间如何实现二层互通。Linux Bridge 本身无法跨宿主机，需要 VXLAN/GRE overlay 或物理交换机 VLAN trunk。这是 PRD 中**最大的技术盲区**。
3. **推荐方案演进路径**：
   - **P0 阶段**：保持 Linux Bridge（满足 6 台 Dell R740 小规模场景）
   - **P1 阶段**：评估 OVS + VXLAN（利用 KVM 原生支持，跨宿主机二层互通，与 Ceph/K8s 混部兼容）
   - **P2 阶段**：引入 OVN（原生多租户、ACL 即安全组、分布式 DHCP、与 K8s CNI 对接）

**建议补充：**

- 明确跨宿主机二层互通方案（VXLAN 隧道或物理交换机 VLAN trunk）
- 增加 VLAN ID 字段到 virtual_networks 表，避免广播域污染
- 在桥接网络创建时增加"VLAN ID"可选参数

### 1.2 安全组实现：iptables/nftables 方案在混部环境下的性能影响？

**结论：方案可行，但存在性能衰减和规则爆炸风险，建议补充压测指标和 nftables 优先策略。**

**详细分析：**

| 维度 | 评审结论 |
|------|---------|
| 方案正确性 | PRD TI-02/03 描述的 chain 命名规范（`SG-{sg_id}-{ingress/egress}`）和原子下发（临时 chain → 替换 → 回滚）设计合理，是业界成熟做法。 |
| 规则生成逻辑 | 5.3 节流程图描述清晰：遍历安全组规则 → 按优先级排序 → 生成 iptables → agent 下发 → 原子替换。 |
| 多安全组叠加 | AC-SG-08 验收条件覆盖了多安全组叠加场景。 |

**性能风险：**

1. **规则膨胀问题**：每台 VM 最多 5 个安全组，每个安全组可能包含几十条规则。在最坏情况下，单台 VM 的 iptables 规则可达数百条。Linux Bridge 混部场景下（KVM + Ceph + K8s 共享 6 台宿主机），iptables 规则匹配是 O(n) 的线性遍历，大量规则会显著增加网络延迟。
2. **iptables vs nftables**：PRD 中提到"iptables/nftables"但未明确优先级策略。nftables 在内核 4.x+ 中性能优于 iptables（规则匹配从 O(n) 优化为集合查找），且支持原子规则替换。建议**以 nftables 为主、iptables 为 fallback**。
3. **Ceph 存储网络干扰**：Ceph OSD 之间的复制流量、K8s pod 网络流量与 VM 流量共享物理网卡。安全组规则在宿主机 netfilter 层处理，大量 VM 的高频规则匹配可能挤占 CPU，影响 Ceph 的延迟敏感性。
4. **缺失性能基准**：PRD 未给出安全组规则数量与吞吐量/延迟的压测指标。建议补充：
   - 单 VM 100 条规则时的吞吐量衰减率
   - 单宿主机 50 VM 时的 iptables 规则总数上限
   - P99 网络延迟在规则加载后的变化

**安全风险：**

1. **默认规则安全**：PRD 4.2.1 明确默认入方向 DROP、出方向 ALLOW，符合最小权限原则。但需确保默认 DROP 规则在宿主机层面正确配置（不仅是 VM 内部 iptables）。
2. **规则绕过风险**：安全组规则应用于 VM 网卡对应的 Linux Bridge 接口。需确认规则是在 Bridge 的 FORWARD chain 还是 VM tap 接口的 INPUT/OUTPUT chain。建议在 Bridge 的 FORWARD chain 层面控制，避免 VM 内部绕过。
3. **存量 VM 保护**：需明确存量 32 台 VM 的默认安全策略。建议在引入安全组时，为存量 VM 自动绑定一个"存量兼容"安全组（入方向全放行），防止误拦截现有业务。

**建议补充：**

- 明确 nftables 优先策略，iptables 仅作为内核兼容性 fallback
- 增加性能基准指标：100/500/1000 条规则时的吞吐量衰减
- 明确安全组规则生效位置（Bridge FORWARD chain vs VM tap interface）
- 为存量 VM 设计兼容迁移策略

### 1.3 CLB 方案：Nginx/HAProxy 作为内网 CLB 是否合理？高可用如何保证？

**结论：方案合理但高可用设计严重不足，P0 阶段即存在单点故障风险。**

**合理性分析：**

| 维度 | 评审结论 |
|------|---------|
| 代理选型 | Nginx 适合七层 HTTP/HTTPS（URL 转发、SSL 卸载），HAProxy 适合四层 TCP/UDP 和七层 HTTP。PRD 同时支持两种代理类型（TI-07），设计合理。 |
| 配置管理 | PRD 7.13 节明确 `nginx -t` / `haproxy -c` 语法检查 + 回滚机制，graceful reload（4.3.3），安全可靠。 |
| SSL 管理 | 私钥 AES-256-GCM 加密存储（TI-08），证书过期提醒（4.3.5），安全合规。 |
| 健康检查 | 完整覆盖 TCP/HTTP/HTTPS，健康/不健康阈值可配置，全不健康时返回 503（7.4 节）。 |

**高可用风险（严重）：**

1. **单点故障**：PRD 7.6 节明确承认"V1 阶段不自动迁移，需运维人工介入"，并计划"P2 迭代考虑 VRRP 方案"。**这是整个网络模块最大的风险点**——CLB 实例部署在单台宿主机，宿主机故障 = CLB 不可用 = 所有后端服务不可达。
2. **6 台 R740 场景下的影响**：如果 CLB 部署在 node-01，node-01 故障后所有依赖该 CLB 的业务中断，直到人工介入重建。对于生产环境不可接受。
3. **VIP 绑定方式不明**：PRD 提到"配置 VIP 绑定"但未说明 VIP 是如何绑定到 CLB 宿主机的。如果是 `ip addr add` 到 lo 或物理网卡，VIP 不具备漂移能力。需要明确：
   - VIP 是绑定到 loopback 还是物理接口？
   - 跨宿主机迁移时如何更新 ARP 表？
4. **缺失冗余方案**：即使是 V1 阶段，也应至少支持以下方案之一：
   - **方案 A（推荐 P0）**：主备模式 — 在不同宿主机部署 CLB 主备实例，通过 keepalived VRRP 实现 VIP 漂移
   - **方案 B（推荐 P1）**：多实例 + DNS 轮询 — 多个 CLB 实例绑定不同 VIP，通过内部 DNS 做负载均衡
   - **方案 C（推荐 P2）**：基于 OVS/OVN 的分布式 LB（类似 K8s kube-proxy IPVS 模式）

**建议补充（P0 优先级）：**

- 将 CLB 高可用方案从 P2 提升至 P0：至少实现主备 + keepalived VRRP
- 明确 VIP 绑定机制和漂移流程
- 增加"CLB 主备切换"验收条件

### 1.4 浮动 IP 设计：跨宿主机漂移的 ARP/GARP 方案在 Linux Bridge 环境下是否可行？

**结论：方案原理正确，但实现细节缺失严重，跨宿主机漂移在纯 Linux Bridge 环境下面临核心挑战。**

**原理分析：**

| 维度 | 评审结论 |
|------|---------|
| 设计对标 | 对标阿里云 EIP 的浮动 IP 设计理念正确（4.4 节），IP 与 VM 生命周期解耦。 |
| ARP/GARP 方案 | 4.4.5 节提到"发送 GARP 通知网络交换机更新 MAC 地址表"，这在二层网络（同一 VLAN/Bridge 域）下是标准做法。 |
| 绑定/解绑流程 | 4.4.2-4.4.3 节的 `ip addr add/del` + ARP 配置流程描述合理。 |

**核心问题：**

1. **跨宿主机漂移的核心矛盾**：浮动 IP 跨宿主机漂移的关键是二层可达性。在纯 Linux Bridge（无 VXLAN overlay）环境下，不同宿主机上的 VM 属于不同 Bridge 域，浮动 IP 从一个宿主机的 Bridge 迁移到另一个宿主机的 Bridge 后，**其他宿主机的 ARP 缓存不会自动更新**。GARP 只能更新同一二层域内的交换机 MAC 表，但无法解决跨 Bridge 的路由问题。
2. **物理交换机依赖**：PRD 假设"发送 GARP 通知网络交换机"，但这要求：
   - 所有宿主机连接到同一物理交换机
   - 虚拟网络使用桥接模式（Bridge 绑定物理网卡）
   - 交换机支持并正确响应 GARP
   如果使用 NAT 网络或隔离网络，GARP 无法跨越 Bridge 边界。
3. **漂移时间目标不现实**：PRD 设定"整个漂移过程目标在 10 秒内完成"，在 Linux Bridge 环境下，ARP 缓存老化时间通常为 30-300 秒，10 秒内完成二层路径切换需要额外手段（如主动清理所有相关宿主机的 ARP 缓存）。
4. **IP 冲突风险**：绑定期间使用 `ip addr add` 添加辅助 IP，如果同一 IP 被意外配置到两台 VM（agent 通信失败等异常场景），将导致 IP 冲突和网络不可用。PRD 的"强制解绑"场景（4.4.3）可能导致此问题。
5. **缺失实现细节**：
   - 浮动 IP 漂移时如何确保旧宿主机不再响应 ARP？（需要 agent 清理 ARP 条目）
   - 如何通知同网络内其他 VM 更新 ARP 缓存？（需要广播 ARP announcement）
   - NAT 网络下浮动 IP 漂移后，SNAT 规则如何更新？

**可行性评估矩阵：**

| 漂移场景 | 可行性 | 条件 |
|---------|--------|------|
| 同宿主机不同 VM | 可行 | 仅需本地 ARP 更新 |
| 跨宿主机 — 桥接网络 | 可行 | 依赖物理交换机 GARP 支持，需确认交换机型号 |
| 跨宿主机 — NAT 网络 | 不可行（V1） | 需要 overlay 网络（VXLAN）或集中式网关 |
| 跨宿主机 — 隔离网络 | 不可行（V1） | 二层隔离域无法跨宿主机 |

**建议：**

- 明确浮动 IP 的适用网络类型限制（P0 仅支持桥接网络）
- 增加跨宿主机漂移的前置条件校验
- 漂移流程中增加：旧宿主机 ARP 条目清理 → 新宿主机 IP 绑定 → GARP 广播（3次，间隔1s）→ 确认可达性
- 将漂移时间目标调整为 30 秒（更现实）
- 考虑引入分布式 ARP 代理（如 OVN 的 ARP responder）解决跨宿主机问题

### 1.5 与存量网络共存：不破坏现有 32 台 VM 的网络，方案是否足够安全？

**结论：PRD 声明了存量兼容原则，但缺乏具体的迁移策略和安全隔离验证机制。**

**安全性分析：**

| 维度 | 评审结论 |
|------|---------|
| 设计原则 | PRD 1.3 节明确"不破坏现有 32 台 KVM 虚拟机的网络配置，新网络与存量网络通过路由互通"。 |
| 命名规范 | TI-01 定义 Bridge 命名 `vn-{network_id前8位}`，可避免与现有 Bridge 冲突。 |
| CIDR 冲突检测 | 4.1.2 节实现 CIDR 重叠检测，防止新建网络与存量网络 IP 冲突。 |

**安全风险：**

1. **存量 VM 的安全组真空**：32 台存量 VM 没有安全组保护，在新平台上创建的新 VM 均有安全组。如果新老 VM 在同一路由域内，攻击者可能通过存量 VM 作为跳板攻击新 VM。需要为存量 VM 提供安全组绑定路径。
2. **存量网络拓扑未建模**：PRD 假设存量网络与新建网络"通过路由互通"，但未描述如何发现和建模存量网络的拓扑。如果存量 VM 使用默认的 virbr0 或自定义 Bridge，管理平台需要能够发现这些网络并展示。
3. **iptables 规则冲突**：存量环境可能已有手工配置的 iptables 规则（防火墙/NAT）。新平台下发安全组规则时可能与存量规则冲突或互相覆盖。PRD 7.10 节覆盖了 VM 迁移场景，但未覆盖存量规则冲突场景。
4. **DHCP 冲突**：存量 VM 可能依赖已有的 dnsmasq 实例。新平台的 dnsmasq 实例需确保监听在正确的 Bridge 接口上，不与存量 dnsmasq 冲突。
5. **路由表污染**：新平台在宿主机上配置 NAT 规则（SNAT/MASQUERADE），可能影响存量 VM 的出站路由行为。

**建议补充：**

- 增加"存量网络发现与导入"功能：扫描宿主机上现有 Bridge/iptables/dnsmasq，自动建模
- 为存量 VM 提供"导入并绑定默认安全组"的一键操作
- 增加 iptables 规则冲突检测（在 agent 下发前执行 `iptables -C` 检查）
- 定义新平台使用的 iptables chain 命名空间（如统一前缀 `NP-`），与存量规则隔离
- 增加存量兼容性验收条件

## 二、权限与审计模块评审

### 2.1 多租户隔离粒度：资源池/网络/数据三层隔离是否充分？

**结论：三层隔离设计充分，落地细节较为完整，但部分隔离边界存在隐患。**

**隔离层次分析：**

| 隔离层 | PRD 实现 | 评审结论 |
|--------|---------|---------|
| **资源池隔离** | Ceph 独立 RBD pool + RGW namespace（REQ-TENANT-001）、K8s 独立 Namespace | 充分。存储和计算在底层物理隔离，防止数据泄露和数据面攻击。 |
| **网络隔离** | 虚拟网络按租户归属、安全组按租户可见（PRD-03 8.1 节） | 充分。但需注意：网络模块 PRD 中安全组支持跨租户只读引用（3.1 节），可能成为信息泄露渠道——租户 A 可通过引用租户 B 的安全组来推断租户 B 的 VM 数量。 |
| **数据隔离** | API 中间件注入 tenant_id（REQ-TENANT-004）、审计日志独立存储 | 充分。但需补充数据库级别的隔离验证（如 PostgreSQL Row-Level Security）。 |

**隐患与建议：**

1. **安全组跨租户引用风险**：PRD-03 允许安全组规则引用其他租户的安全组作为流量来源。这虽然是合理的网络互通需求，但存在信息泄露风险——攻击者可通过创建引用规则来探测目标安全组是否存在（时序侧信道）。建议：
   - 引用前需目标租户管理员审批
   - 限制引用频率，防止枚举攻击
2. **数据库层面隔离**：PRD 描述的是应用层 tenant_id 过滤。建议增加 PostgreSQL Row-Level Security（RLS）策略作为纵深防御，防止应用层 bug 导致的跨租户数据泄露。
3. **配额隔离**：REQ-TENANT-002 的配额校验在中间件层，但未描述配额计算是实时查询还是缓存。如果使用缓存（如 PRD-03 提到的 Redis 权限缓存），可能存在配额检查的时间窗口漏洞——两个并发创建请求可能同时通过配额检查导致超额分配。建议配额校验使用数据库行级锁或 Redis 原子操作。

### 2.2 5 个预定义角色：权限边界是否合理？有没有遗漏？

**结论：5 个预定义角色边界清晰、符合最小权限原则，但缺少安全管理员独立角色。**

**角色设计评审：**

| 角色 | 权限范围 | 评审结论 |
|------|---------|---------|
| 平台管理员 (platform_admin) | 全局所有权限 | 合理。需确保至少有 1 个活跃平台管理员（PRD 7.11 节已覆盖）。 |
| 租户管理员 (tenant_admin) | 本租户全部权限 + 用户/角色管理 | 合理。权限继承规则（8.2 节）清晰。 |
| 租户运维 (tenant_operator) | 本租户资源管理 | 合理。与开发者权限边界清晰（运维可管理全部资源，开发者仅管理自己创建的）。 |
| 租户开发者 (tenant_developer) | 本租户查看 + 自助创建 | 合理。"仅能删除自己创建的资源"（8.4 节）是正确的设计。 |
| 只读用户 (tenant_viewer) | 仅查看 | 合理。 |

**遗漏角色：**

1. **安全管理员 (security_admin)**：PRD-03 的权限矩阵（8.1 节）中存在"安全管理员"角色——"可查看所有租户的安全组和规则，可批量封禁 IP，但不可创建/删除网络和 CLB"。但在 PRD-04 的预定义角色列表（REQ-ROLE-001）中，安全管理员**并未列为预定义角色**。两个 PRD 存在不一致。
   - 如果安全管理员是独立角色，需要加入预定义角色列表并定义完整权限矩阵
   - 如果安全管理员是平台管理员的职责子集，需要在平台管理员权限中明确安全操作边界
2. **审计员 (auditor)**：PRD-04 的 US-05 提到"审计员"角色，但在预定义角色和权限矩阵中均未出现。审计员应独立于平台管理员，仅具有审计日志查看权限，无任何资源操作权限。这是等保三级的重要要求（三权分立：系统管理员、安全管理员、审计管理员）。
3. **网络管理员 (network_admin)**：如果网络运维需要独立角色（仅管理网络/安全组/CLB，不管理 VM 和存储），当前角色体系无法满足。建议作为可扩展的自定义角色，暂不纳入预定义。

**建议：**

- 将"安全管理员"和"审计员"纳入预定义角色列表，实现三权分立
- 统一 PRD-03 和 PRD-04 中的角色定义
- 在权限矩阵中为安全管理员和审计员补充完整行

### 2.3 审计日志方案：独立存储、不可篡改的方案是否可行？

**结论：方案设计总体可行且安全，但 ClickHouse/ES 选型需澄清，append-only 的实现细节需补充。**

**安全性分析：**

| 维度 | 评审结论 |
|------|---------|
| 存储分离 | REQ-AUDIT-002 要求审计日志与业务数据库物理分离、独立账号、仅审计模块可写。符合等保三级要求。 |
| 不可篡改 | 仅支持 append-only，不提供修改/删除 API。设计正确。 |
| 数据完整性 | 使用 UUID v7（按时间排序），便于顺序校验和完整性检查。 |
| 保留策略 | 180 天在线 + 归档到 Ceph RGW + 3 年保留（7.4 节）。合规且可落地。 |

**实现风险：**

1. **ClickHouse vs Elasticsearch 选型未定**：PRD 同时提到 ClickHouse 和 Elasticsearch。两者架构差异显著：
   - ClickHouse：列存、压缩率高、时序查询快，但更新/删除能力弱（恰好符合 append-only 需求）
   - Elasticsearch：全文搜索强、支持复杂聚合，但存储成本高、运维复杂
   - **推荐 ClickHouse**：更符合审计日志"写多读少、时序查询"的特征，且 6 台 R740 的资源有限，ClickHouse 的压缩率更有优势
2. **append-only 的完整性校验**：PRD 未提及如何防止审计日志被绕过直接操作数据库。建议：
   - 在 ClickHouse 表上设置 `ENGINE = MergeTree()` + 禁止 DELETE/UPDATE 的权限
   - 使用区块链式哈希链（每条日志包含前一条日志的哈希），实现防篡改验证
   - 定期导出日志快照并签名，存入离线存储
3. **审计日志写入的可靠性**：审计日志写入失败不应阻塞业务操作。PRD 需明确写入策略：
   - 同步写入（阻塞业务，但保证完整性）vs 异步写入（不阻塞，但可能丢失）
   - 写入失败时的重试和降级策略
   - **推荐方���**：异步写入 + 本地缓冲队列 + 失败告警
4. **敏感信息脱敏**：审计日志中 request_params 包含"脱敏后"的请求参数（REQ-AUDIT-001），但未定义脱敏规则。密码、Token、私钥等敏感字段必须在写入审计日志前脱敏。建议明确脱敏字段清单和规则。

**建议补充：**

- 明确审计日志存储选型为 ClickHouse（推荐）
- 增加审计日志完整性校验机制（哈希链或 Merkle Tree）
- 明确脱敏字段清单（密码/Token/私钥/手机号/邮箱）
- 定义审计日志写入可靠性策略（异步 + 缓冲 + 告警）

### 2.4 LDAP 预留接口：设计是否合理？

**结论：接口预留思路正确，降级方案务实，但集成细节有改进空间。**

**合理性分析：**

| 维度 | 评审结论 |
|------|---------|
| 降级策略 | 7.10 节设计了 LDAP 不可用时的本地认证降级，并明确 5 秒超时和自动重连。设计务实。 |
| 用户导入 | REQ-USER 功能列表中包含"LDAP/AD 导入用户，同步组织架构"（P1），方向正确。 |
| 密码策略兼容 | 本地密码策略（90 天过期、复杂度校验）与 LDAP 域策略可能存在冲突，需定义优先级。 |

**风险与建议：**

1. **LDAP 用户的 tenant 映射**：LDAP 用户导入时，如何映射到平台租户？是通过 LDAP 组织单元（OU）映射还是手动指定？PRD 未说明。建议：
   - 支持基于 LDAP 属性（如 `department` 或 `memberOf`）自动映射到租户
   - 提供映射规则配置界面
2. **密码策略冲突**：LDAP 域控制器通常有自己的密码策略。当平台本地策略（90 天过期）与 AD 策略不一致时，以谁为准？建议明确：LDAP 用户密码策略以域控制器为准，本地策略仅用于纯本地用户。
3. **LDAP 同步周期**：未说明用户和组的同步频率（实时/定时）。建议：
   - 实时同步：用户登录时验证 LDAP 凭据并更新本地信息
   - 定时全量同步：每日凌晨执行，同步新增/删除用户和组织架构变更
4. **纯 LDAP 用户的 fallback**：7.10 节正确指出"纯 LDAP 用户（无本地密码），LDAP 不可用时无法登录"。建议为 LDAP 用户可选配置本地密码哈希同步（用户首次 LDAP 登录成功后缓存密码哈希），提升高可用性。
5. **LDAPS/TLS**：未提及 LDAP 通信是否加密。**必须使用 LDAPS（636 端口）或 StartTLS**，明文 LDAP 不可接受。

**建议补充：**

- 明确 LDAP 用户到租户的映射规则
- 明确 LDAP 密码策略与本地策略的优先级
- 补充 LDAPS/TLS 要求
- 增加 LDAP 用户本地密码哈希缓存选项

### 2.5 会话安全和登录策略：是否足够？

**结论：基本安全措施到位，但 JWT 设计存在风险，MFA 缺失是明显短板。**

**安全措施评审：**

| 安全措施 | PRD 实现 | 评审结论 |
|---------|---------|---------|
| 密码策略 | 8位+三类字符+90天过期+历史5次（REQ-USER-003） | 符合基线要求。建议增加密码黑名单检测（如 Have I Been Pwned API）。 |
| 登录锁定 | 5次失败/30分钟（REQ-USER-004） | 合理。建议增加递增锁定时间（第1次锁定5分钟，第2次15分钟，第3次30分钟）。 |
| 会话超时 | 30分钟无操作 + 前端5分钟续期提醒（REQ-USER-006） | 合理。 |
| 异地登录检测 | IP 归属地城市变化 + 邮件告警（REQ-SEC-001） | 务实。纯内网环境（10.x IP）的异地检测意义有限，但对 VPN/远程接入场景有价值。 |
| JWT Token | RS256 签名 + 2小时有效期（10.7 节） | 存在风险，见下文。 |

**JWT 安全风险（重要）：**

1. **无 Token 刷新机制细节**：PRD 附录 B 列出了 `/api/v1/auth/refresh` 接口，但未说明 refresh token 的存储和失效策略。refresh token 通常需要更长的有效期和更强的保护（httpOnly cookie 或 secure storage）。
2. **无 Token 吊销机制**：JWT 是无状态的，2 小时有效期内无法主动吊销。如果管理员禁用用户或修改角色，用户的旧 Token 仍然有效（直到过期）。PRD 虽然提到"禁用用户后已登录会话在下次请求时失效"（10.1 节），但这与无状态 JWT 的设计矛盾——除非每次请求都查数据库验证用户状态。
3. **建议**：
   - 引入 Token 黑名单（Redis 缓存被吊销的 Token ID，TTL = Token 剩余有效期）
   - 或使用有状态会话（Session ID + Redis），彻底解决吊销问题
   - 如果坚持无状态 JWT，将有效期缩短至 15 分钟 + refresh token 1 小时，降低吊销窗口
4. **RS256 vs HS256**：PRD 选择 RS256（非对称签名），这是正确的选择（便于微服务间验证，私钥集中管理）。但需确保私钥安全存储（如 Vault 或加密配置文件）。

**缺失的安全措施：**

1. **MFA 多因素认证**：PRD 将 MFA 标记为 P2，但考虑到平台管理员和租户管理员的高权限，**建议将 MFA 提升至 P1**（至少对管理员角色强制启用）。
2. **IP 白名单**：PRD 将 IP 白名单标记为 P2。考虑到内网运维平台的敏感性，建议 P1 支持租户级别的 IP 访问控制（如仅允许公司内网 IP 段访问）。
3. **操作频率限制**：PRD 未提及 API 速率限制。缺少速率限制可能导致暴力破解（虽然已有登录锁定）和资源滥用。建议在 API Gateway 层增加速率限制。
4. **会话并发控制**：未提及是否允许同一用户多处同时登录。建议支持配置：允许/禁止并发会话，默认允许但限制数量（如最多 3 个设备）。

**建议补充：**

- 明确 JWT 吊销机制（Token 黑名单或缩短有效期）
- 将 MFA 从 P2 提升至 P1（至少对管理员角色）
- 增加 API 速率限制（全局 + 用户级别）
- 增加会话并发控制配置

### 3.1 角色定义不一致

| 问题 | 详情 |

### 3.2 租户创建流程不一致

PRD-04 REQ-TENANT-001 描述创建租户时自动在 Ceph 创建存储池和在 K8s 创建 Namespace。但 PRD-03（网络模块）未描述创建租户时是否需要自动创建默认虚拟网络。建议明确：创建租户时是否自动创建默认 NAT 网络？或者首次创建 VM 时才创建？

### 3.3 权限校验链路描述不一致

PRD-04 REQ-PERM-002 描述的中间件链路包含"配额校验中间件"，但 PRD-03 的网络模块未描述配额限制（仅租户级别有 VM/CPU/内存配额，网络资源如虚拟网络数量、安全组数量、CLB 数量是否有配额？）。
