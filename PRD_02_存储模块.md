## 一、模块概述

### 1.1 模块定位

存储模块是运维内网统一管理平台的三大核心基础设施模块之一，旨在为用户提供**一站式的企业级存储资源管理能力**。该模块基于自研轻量内核 + 插件化驱动架构，以只读纳管方式接入现有 Ceph 集群（Reef 18.2.5），在不影响存量业务的前提下，提供以下核心能力：

- **对象存储**：对标阿里云 OSS，面向应用的弹性、安全、海量非结构化数据存储
- **块存储**：Ceph RBD 存储池管理 + 虚拟机磁盘存储概览，提供 RBD 镜像管理、Pool 使用情况监控和虚拟机磁盘的存储全景视图
- **备份管理**：基于 `kvm-backup` Pool 的虚拟机备份管理，支持备份策略配置、任务监控和恢复操作
- **集群管理**：对标 Ceph Dashboard，面向运维管理员的集群健康与性能监控

### 1.2 与对标产品的关系

| 子模块 | 对标产品 | 核心对标能力 | 差异化定位 |
|--------|----------|-------------|-----------|
| 对象存储 | 阿里云 OSS | Bucket 管理、对象 CRUD、ACL 访问控制、用量统计 | 基于 Ceph RGW 的私有化部署，零网络出口成本；与内部 IAM 体系原生集成 |
| 块存储 | — | RBD 存储池管理、RBD 镜像列表与监控、虚拟机磁盘存储概览 | 定位调整：虚拟机磁盘（qcow2/RBD）由计算模块管理，存储模块聚焦 Ceph RBD 层面的存储池管理和磁盘存储全景视图 |
| 备份管理 | — | 备份策略配置、备份任务列表、备份恢复、备份空间趋势 | 基于 `kvm-backup` Pool 的虚拟机备份统一管理，弥补原 PRD 缺失的备份管理能力 |
| 集群管理 | Ceph Dashboard | 集群健康、OSD/Pool/MON 状态、性能监控、告警 | 增强的多集群纳管能力 + 自定义告警规则 + 与企业运维体系（工单、通知）深度整合 |

### 1.3 现有基础设施概况

| 指标 | 当前值 |
|------|--------|
| Ceph 版本 | Reef 18.2.5 |
| 集群状态 | HEALTH_OK |
| 节点数 | 6 台（ceph-node1 ~ ceph-node6） |
| MON 节点 | 5 个（node2 为 active MGR） |
| OSD 总数 | 8 块 HDD |
| 总容量 | 29 TiB |
| 已用量 | 3.2 TiB（10.89%） |
| Pool 总数 | 12 个 |
| RBD 状态 | 已启用（kvm-backup pool，251 GiB） |
| RGW 状态 | 已启用（825 GiB 数据，4 用户） |
| CephFS 状态 | 已启用（ark-CephFS，27 MiB） |

### 1.4 核心约束

- **只读纳管**：对接存量 Ceph 集群时，默认以只读模式导入，不可自动修改存量资源配置
- **不可中断**：所有管理操作不得影响现有业务正常运行
- **渐进式接管**：新增资源（Bucket、云盘、Pool）通过平台创建，存量资源逐步迁移纳入平台管理
- **多租户隔离**：资源按项目/租户进行隔离，基于 IAM 权限体系控制访问

## 二、用户故事

### US-01：运维管理员监控集群健康状态

> **作为** 运维管理员  
> **我想要** 在一个统一的 Dashboard 上查看 Ceph 集群的健康状态、容量使用趋势、IOPS 和带宽指标  
> **以便于** 我可以第一时间发现集群异常并快速响应，避免业务中断

**验收条件**：
- Dashboard 首页展示集群整体健康状态（HEALTH_OK / HEALTH_WARN / HEALTH_ERR）
- 展示总容量、已用容量、剩余容量的数值与百分比环图
- 展示最近 24 小时的 IOPS 和带宽趋势折线图
- 展示 PG 状态分布（active+clean / degraded / inconsistent 等）

---

### US-02：开发人员创建并使用对象存储 Bucket

> **作为** 应用开发人员  
> **我想要** 在平台上自助创建 Bucket、上传和管理文件  
> **以便于** 我可以快速为应用搭建文件存储能力，无需找运维手动配置

**验收条件**：
- 可在界面创建 Bucket，指定名称、所属项目、访问权限
- 创建后自动生成 S3-compatible 的 Endpoint、AccessKey、SecretKey
- 支持通过 Web 界面上传/下载/删除文件，支持拖拽上传
- 支持创建文件夹、批量删除

---

### US-03：虚拟化管理员为虚拟机挂载云盘

> **作为** 虚拟化管理员  
> **我想要** 为指定虚拟机创建并挂载云盘、按需扩容  
> **以便于** 我可以灵活管理虚拟机的存储资源，应对业务增长

**验收条件**：
- 可从可用存储池中创建指定大小的云盘
- 云盘可挂载到已纳管的虚拟机，支持指定设备名
- 支持在线扩容（仅允许扩大），扩容后自动通知虚拟机 OS 识别新容量
- 云盘卸载前二次确认，已挂载的云盘不可删除

---

### US-04：运维管理员配置告警规则

> **作为** 运维管理员  
> **我想要** 配置自定义告警规则（容量阈值、OSD 状态、PG 异常等）  
> **以便于** 我可以在问题发生前收到预警，或在故障发生时第一时间被通知

**验收条件**：
- 支持创建多条告警规则，每条规则指定监控指标、阈值、通知渠道
- 当集群使用率超过 70%/85%/95% 时自动触发告警
- 当 OSD 状态变为 Down 时触发告警
- 当 PG 出现 degraded/inconsistent/stuck 状态时触发告警
- 告警通知支持企业微信/邮件/短信

---

### US-05：存储管理员管理 OSD 和 Pool

> **作为** 存储管理员  
> **我想要** 查看所有 OSD 的详细状态、管理 Pool 的配置  
> **以便于** 我可以及时发现故障磁盘，合理规划存储池资源

**验收条件**：
- OSD 列表展示所有 OSD 的 ID、节点、状态、容量、使用率、PG 数
- 支持按状态筛选（up/down/in/out）
- Pool 列表展示所有 Pool 的名称、类型、副本数、PG 数、用量
- 支持查看 Pool 详情（PG 状态分布、关联的 RBD/RGW 资源）
- OSD 和 Pool 的操作（创建、删除、修改）需二次确认并记录审计日志

---

### US-06：项目负责人查看存储用量与成本

> **作为** 项目负责人  
> **我想要** 查看本项目的对象存储和块存储用量统计  
> **以便于** 我可以合理规划存储预算，识别资源浪费

**验收条件**：
- 按项目维度展示对象存储总用量、Bucket 数量、请求次数
- 按项目维度展示块存储总容量、云盘数量、快照占用
- 支持按时间范围（7天/30天/自定义）查看用量趋势

## 四、详细需求描述（EARS 格式）

### 4.1 对象存储

#### 4.1.1 Bucket 管理

**REQ-OBJ-001 — Bucket 列表加载**

> **Event-driven**：当用户进入 Bucket 列表页时，系统应根据当前用户的项目权限，加载并展示其有权访问的 Bucket 列表。

**REQ-OBJ-002 — Bucket 列表展示字段**

> **Ubiquitous**：Bucket 列表应始终展示以下字段：Bucket 名称、所属项目、区域（RGW Zone）、存储类型（标准/低频）、存储用量（GiB/TiB）、对象数量、访问权限（私有/公共读/公共读写）、创建时间、创建者。

**REQ-OBJ-003 — 创建 Bucket 表单校验**

> **Event-driven**：当用户提交创建 Bucket 表单时，系统应校验 Bucket 名称符合 DNS 命名规范（3-63 字符、仅小写字母/数字/连字符、不以连字符开头或结尾），且名称在全局唯一。

**REQ-OBJ-004 — 创建 Bucket 权限设置**

> **Event-driven**：当用户创建 Bucket 时，系统应默认设置访问权限为"私有"，并允许用户选择"公共读"或"公共读写"。

**REQ-OBJ-005 — Bucket 创建成功反馈**

> **Event-driven**：当 Bucket 创建成功后，系统应自动生成 S3-compatible 的 AccessKey 和 SecretKey，并在结果页展示 Endpoint 地址，同时提供一键复制凭证功能。

**REQ-OBJ-006 — 删除 Bucket 前置校验**

> **Event-driven**：当用户发起删除 Bucket 操作时，系统应首先检查 Bucket 是否为空（无任何对象和分片上传碎片）。
>
> **Unwanted**：如果 Bucket 非空，则系统应拒绝删除并提示用户"Bucket 内存在 N 个对象，请先清空 Bucket 后再删除"。
>
> **Event-driven**：如果 Bucket 为空，系统应弹出二次确认对话框，要求用户输入 Bucket 名称确认删除意图。

**REQ-OBJ-007 — Bucket 用量统计**

> **State-driven**：在 Bucket 详情页中，系统应展示以下用量信息：
> - 当前存储用量（GiB/TiB）
> - 对象总数
> - 最近 30 天存储用量趋势折线图
> - 最近 24 小时请求次数趋势（GET/PUT/DELETE 分类）

---

#### 4.1.2 对象管理

**REQ-OBJ-008 — 文件列表展示**

> **Event-driven**：当用户进入 Bucket 详情页时，系统应以文件浏览器风格展示 Bucket 内第一层级的对象和文件夹列表。
>
> **Ubiquitous**：文件列表应始终展示以下字段：名称、类型（文件夹/文件/通过扩展名图标区分）、大小（GiB/MiB/KiB 自适应单位）、最后修改时间、存储类型。

**REQ-OBJ-009 — 文件夹导航**

> **Event-driven**：当用户点击文件夹时，系统应进入该文件夹层级，展示该路径下的对象列表，并在顶部显示面包屑导航（Bucket > dir1 > dir2 > ...），支持点击面包屑任意层级跳转。

**REQ-OBJ-010 — 创建文件夹**

> **Event-driven**：当用户点击"创建文件夹"按钮时，系统应弹出输入框要求输入文件夹名称。
>
> **Event-driven**：当用户确认创建时，系统应在当前路径下创建一个以"/"结尾的零大小对象，模拟文件夹结构。

**REQ-OBJ-011 — 文件上传**

> **Event-driven**：当用户触发文件上传（点击上传按钮选择文件或拖拽文件到上传区域）时，系统应执行以下行为：
> - 如果单个文件 ≤ 100 MiB：使用普通 PUT 上传
> - 如果单个文件 > 100 MiB：自动启用分片上传（Multipart Upload），分片大小默认 10 MiB
>
> **Ubiquitous**：文件上传应始终显示实时进度条，包括：上传速度、已上传/总大小、剩余时间估算。
>
> **Unwanted**：如果上传过程中网络中断，则系统应支持断点续传——在网络恢复后自动从上次中断的分片继续上传，最多重试 3 次。
>
> **Event-driven**：当上传完成后，系统应刷新当前文件列表，并在页面右上角显示"上传成功"的绿色提示。

**REQ-OBJ-012 — 文件下载**

> **Event-driven**：当用户点击单个文件的"下载"按钮时，系统应生成预签名 URL（Presigned URL，有效期 5 分钟），并通过浏览器触发文件下载。
>
> **Event-driven**：当用户选中多个文件并点击"批量下载"时，系统应后台异步将选中文件打包为 ZIP，ZIP 生成完毕后通过通知提示用户下载，ZIP 文件在 24 小时后自动清理。

**REQ-OBJ-013 — 文件删除**

> **Event-driven**：当用户选中一个或多个对象并点击"删除"时，系统应弹出确认对话框，列出即将删除的对象名称和数量。
>
> **Event-driven**：当用户确认删除后，系统应执行删除操作并刷新列表。
>
> **Unwanted**：如果删除操作部分失败（如部分对象被锁定），则系统应展示详细的成功/失败列表。

**REQ-OBJ-014 — 文件搜索**

> **Optional**：当文件搜索功能启用时，系统应支持以下搜索方式：
> - 按文件名前缀匹配（Prefix Search）
> - 按文件名后缀/扩展名过滤
> - 支持分页展示搜索结果，每页默认 100 条

#### 4.1.3 访问控制

**REQ-OBJ-015 — ACL 设置**

> **Event-driven**：当用户在 Bucket 详情页修改访问权限时，系统应通过 RGW Admin API 更新 Bucket 的 ACL 策略。
>
> **Ubiquitous**：Bucket 访问权限应始终支持以下三种级别：
> - **私有（Private）**：仅 Bucket Owner 和授权用户可访问
> - **公共读（Public Read）**：所有人可读取对象，仅 Owner 可写入
> - **公共读写（Public Read/Write）**：所有人可读写（不推荐，设置时需弹窗警告）

**REQ-OBJ-016 — 访问凭证管理**

> **Event-driven**：当用户在 Bucket 详情页查看访问凭证时，系统应展示当前 Bucket 关联的 AccessKey 列表（仅显示 Key 前缀 + 创建时间），SecretKey 仅在创建时展示一次。
>
> **Event-driven**：当用户点击"吊销凭证"时，系统应弹出二次确认，确认后通过 RGW Admin API 移除该 AccessKey。

#### 4.1.4 RGW User 纳管

> **背景说明**：集群现有 4 个存量 RGW 用户（`manager-images`、`dashboard`、`inner-img`、`ark-video-backup`），共计 825 GiB 数据。这些用户的 AccessKey/SecretKey 可能已被业务系统使用，纳管时需保护现有凭证不被意外吊销。

**REQ-OBJ-017 — 存量 RGW User 扫描**

> **Event-driven**：当平台首次接入 Ceph 集群时，系统应自动执行 RGW User 专项扫描：
> - 列出所有 RGW User（`radosgw-admin user list`）
> - 获取每个 User 的详细信息（`radosgw-admin user info --uid=xxx`），包括：User ID、显示名称、邮箱、配额、AccessKey 列表（仅记录 Key 前缀，不记录 SecretKey）、关联 Bucket 列表
> - 建立 User → Bucket 的关联映射关系
> - 标记 User 类型为"存量纳管"
>
> **Ubiquitous**：存量纳管扫描完成后，系统应始终在扫描结果摘要中展示 RGW User 的发现数量。

**REQ-OBJ-018 — RGW User 列表页**

> **Event-driven**：当用户进入 RGW User 列表页时，系统应展示所有 RGW User 的列表，包含以下字段：User ID、显示名称、类型（存量纳管/平台创建）、所属项目（未分配则显示"—"）、关联 Bucket 数量、存储总用量、AccessKey 数量（不暴露 SecretKey）、创建时间。

**REQ-OBJ-019 — RGW User 详情**

> **Event-driven**：当用户点击某个 RGW User 进入详情页时，系统应展示：
> - 基本信息：User ID、显示名称、邮箱、配额、创建时间
> - 关联 Bucket 列表：Bucket 名称、用量、权限
> - 访问凭证列表：AccessKey 前缀、创建时间、最后使用时间（如有）、状态（活跃/已吊销）
>
> **Ubiquitous**：存量纳管 User 的 SecretKey 应始终不可查看（仅展示 Key 前缀），平台创建的 User 的 SecretKey 仅在创建时展示一次。

**REQ-OBJ-020 — 存量 User 凭证保护**

> **Unwanted**：如果用户尝试吊销存量纳管 User 的 AccessKey，则系统应弹出高风险警告——"该用户为存量纳管用户，其 AccessKey 可能被以下业务系统使用：[关联 Bucket 列表 + 最近访问记录]。吊销凭证将导致相关业务中断。确认继续？"，并要求用户输入 User ID 确认。

> **Event-driven**：吊销操作前，系统应执行影响范围分析，展示使用该凭证的所有 Bucket 列表和最近 7 天的访问记录摘要。
>
> **Event-driven**：确认吊销后，系统应记录审计日志，并通过通知渠道告知相关项目管理员。

**REQ-OBJ-021 — RGW User 与项目映射**

> **Event-driven**：当管理员将存量 RGW User 分配到项目时，系统应建立 User → 项目 → Bucket 的归属关系。分配后，该项目下的成员可查看和使用该 User 关联的 Bucket。
>
> **Ubiquitous**：存量 RGW User 分配项目后，其 Bucket 列表应始终对项目成员可见（遵循项目隔离原则）。
>
> **建议映射关系**：
> - `ark-video-backup` → 方舟项目
> - `inner-img` → 内部图片服务项目
> - `manager-images` → 管理后台项目
> - `dashboard` → 运维项目

**REQ-OBJ-022 — 创建平台 RGW User**

> **Event-driven**：当管理员创建平台 RGW User 时，系统应：
> 1. 通过 RGW Admin API 创建 RGW User（`radosgw-admin user create`）
> 2. 自动生成 AccessKey 和 SecretKey
> 3. 将 User 关联到指定项目
> 4. 展示创建结果（User ID、AccessKey、SecretKey（仅一次））
>
> **Ubiquitous**：平台创建的 RGW User 应始终标记类型为"平台创建"，与存量纳管 User 区分。

**REQ-OBJ-023 — RGW User 与 Bucket 创建策略**

> **Ubiquitous**：新建 Bucket 时应始终遵循以下策略：
> - 默认使用项目级 RGW User（一个项目对应一个 RGW User），在该 User 下创建 Bucket
> - 或允许用户选择使用已有的项目级 RGW User 或创建新的应用级 RGW User
>
> **Event-driven**：当用户创建 Bucket 时，系统应在创建表单中展示"RGW User"选项：默认关联项目级 RGW User，可下拉选择其他已关联项目的 RGW User。

### 4.2 块存储

> **定位说明**：虚拟机磁盘（本地 qcow2 或 Ceph RBD）的创建、挂载、扩容、卸载等生命周期管理由**计算模块**负责，用户在创建虚拟机时自由选择存储后端。存储模块的块存储子模块聚焦于 **Ceph RBD 层面的存储池管理和虚拟机磁盘存储概览**，提供 RBD 镜像的全局视图和 Pool 使用情况监控。

#### 4.2.1 RBD 存储池管理

**REQ-BLK-001 — RBD Pool 列表加载**

> **Event-driven**：当用户进入 RBD Pool 列表页时，系统应加载并展示所有启用了 RBD 的存储池列表。

**REQ-BLK-002 — RBD Pool 列表展示字段**

> **Ubiquitous**：RBD Pool 列表应始终展示以下字段：Pool 名称、Pool ID、类型（副本/EC）、副本数/EC Profile、PG 数、总分配容量（含 RBD Image 总大小）、实际占用容量、使用率（百分比进度条）、RBD Image 数量、压缩状态。

**REQ-BLK-003 — RBD Pool 容量告警**

> **Event-driven**：当单个 RBD Pool 使用率超过 70% 时，该 Pool 行应显示黄色警告标记。
>
> **Event-driven**：当单个 RBD Pool 使用率超过 85% 时，该 Pool 行应显示红色警告标记，并触发严重告警通知。
>
> **Unwanted**：如果 Pool 使用率超过 90%，则系统应拒绝在该 Pool 中创建新的 RBD Image，并提示"存储池 {pool_name} 使用率已达 {usage}%，为避免性能下降，已暂停新建操作。请联系管理员扩容。"

**REQ-BLK-004 — Pool 使用情况详情**

> **Event-driven**：当用户点击某个 RBD Pool 进入详情页时，系统应展示：
> - Pool 基本信息（名称、ID、类型、副本数、PG 数、压缩配置）
> - 该 Pool 下所有 RBD Image 的汇总统计（总数量、总分配容量、实际占用容量）
> - 最近 30 天的 Pool 使用量趋势折线图
> - 该 Pool 下所有 RBD Image 的列表（名称、大小、快照数、关联虚拟机）

#### 4.2.2 RBD 镜像管理

**REQ-BLK-005 — RBD 镜像列表加载**

> **Event-driven**：当用户进入 RBD 镜像列表页时，系统应根据当前用户的项目权限，加载并展示其有权查看的 RBD Image 列表。

**REQ-BLK-006 — RBD 镜像列表展示字段**

> **Ubiquitous**：RBD 镜像列表应始终展示以下字段：Image 名称、所属 Pool、大小（GiB）、实际占用（GiB）、关联虚拟机（名称，无关联则显示"—"）、磁盘后端类型（Ceph RBD / 本地 qcow2）、快照数量、创建时间。

**REQ-BLK-007 — RBD 镜像筛选**

> **Ubiquitous**：RBD 镜像列表应始终支持以下筛选条件：按 Pool 筛选、按关联虚拟机状态筛选（已关联/未关联）、按磁盘后端类型筛选、按名称搜索。

**REQ-BLK-008 — RBD 镜像详情**

> **Event-driven**：当用户点击某个 RBD Image 进入详情页时，系统应展示：
> - 基本信息：名称、Pool、大小、实际占用、创建时间、磁盘后端类型
> - 关联信息：关联虚拟机名称、设备名（如有）、虚拟机状态
> - 快照列表：快照名称、创建时间、大小
> - 性能指标：最近 24 小时的 IOPS、吞吐量、延迟趋势图

#### 4.2.3 虚拟机磁盘存储概览

**REQ-BLK-009 — 虚拟机磁盘存储概览**

> **Event-driven**：当用户进入虚拟机磁盘存储概览页时，系统应展示所有虚拟机磁盘的汇总视图：
> - **概览卡片**：虚拟机总数、使用 Ceph RBD 的 VM 数、使用本地 qcow2 的 VM 数、磁盘总容量、磁盘总占用
> - **按项目维度**：各项目的虚拟机磁盘容量汇总（RBD vs qcow2 分开展示）
> - **按存储后端维度**：Ceph RBD 和本地 qcow2 的容量分布饼图
>
> **Ubiquitous**：概览页应始终支持按项目筛选和按存储后端类型筛选。

**REQ-BLK-010 — 虚拟机磁盘列表**

> **Event-driven**：当用户进入虚拟机磁盘列表页时，系统应展示所有虚拟机磁盘的列表，包含以下字段：虚拟机名称、磁盘路径/Pool、存储后端类型（Ceph RBD / 本地 qcow2）、磁盘大小（GiB）、实际占用（GiB）、使用率、挂载点/设备名、所属项目。
>
> **Ubiquitous**：磁盘列表应始终支持按存储后端类型筛选（全部/Ceph RBD/本地 qcow2）和按项目筛选。

#### 4.2.4 RBD 快照管理

**REQ-BLK-011 — 创建快照**

> **Event-driven**：当用户在 RBD Image 详情页点击"创建快照"时，系统应弹出表单：
> - 快照名称（必填，2-64 字符）
> - 描述（选填）
>
> **Event-driven**：当用户确认创建后，系统应通过 RBD 快照命令创建快照（\）。
>
> **Unwanted**：如果 RBD Image 正在进行其他操作（如扩容），则系统应拒绝创建快照并提示当前操作状态。

**REQ-BLK-012 — 删除快照**

> **Event-driven**：当用户删除快照时，系统应弹出二次确认，确认后执行 \。

**REQ-BLK-013 — 回滚快照**

> **Event-driven**：当用户点击"回滚快照"时，系统应弹出严重警告——"回滚操作将覆盖 RBD Image 当前数据，此操作不可逆！"，并要求用户输入 Image 名称确认。
>
> **Unwanted**：如果 RBD Image 已挂载到虚拟机，则系统应拒绝回滚并提示"请先在计算模块中卸载该磁盘后再执行回滚操作"。

#### 4.2.5 RBD Image 操作

**REQ-BLK-014 — 创建 RBD Image**

> **Event-driven**：当用户点击"创建 RBD Image"时，系统应弹出创建表单，包含以下字段：
> - Image 名称（必填，2-64 字符，支持中英文/数字/连字符/下划线）
> - 目标 Pool（必填，下拉选择已启用的 RBD Pool，展示每个 Pool 的可用容量和使用率）
> - 容量大小（必填，最小 1 GiB，最大不超过所选 Pool 可用容量的 90%，步长为 1 GiB）
> - 描述（选填，最多 256 字符）
>
> **Unwanted**：如果所选 Pool 使用率已超过 90%，则系统应拒绝创建并提示"存储池 {pool_name} 使用率已达 {usage}%，已暂停新建操作"。

**REQ-BLK-015 — 删除 RBD Image**

> **Event-driven**：当用户发起删除 RBD Image 操作时，系统应首先检查该 Image 是否关联虚拟机。
>
> **Unwanted**：如果 Image 关联了虚拟机，则系统应拒绝删除并提示"RBD Image 已关联虚拟机 {vm_name}，请先在计算模块中卸载并解除关联后再删除"。
>
> **Event-driven**：如果 Image 未关联虚拟机，系统应弹出二次确认对话框，要求用户输入 Image 名称确认删除意图。
>
> **Event-driven**：当用户确认删除后，系统应先检查 Image 是否存在快照。
>
> **Unwanted**：如果 Image 存在快照，则系统应提示"Image 存在 N 个快照，删除将同时删除所有快照，确认继续？"，需用户再次确认。

**REQ-BLK-016 — 扩容 RBD Image**

> **Event-driven**：当用户对 RBD Image 点击"扩容"时，系统应弹出扩容表单，展示当前容量，并允许用户输入新的容量值。
>
> **Ubiquitous**：扩容操作应始终遵循"仅允许扩大，不允许缩小"的约束。
>
> **Unwanted**：如果 Image 已挂载到虚拟机且虚拟机 Guest Agent 不可达，则系统应提示"扩容完成后需手动在虚拟机内执行文件系统扩容"。

### 4.3 Ceph 集群管理

#### 4.3.1 集群 Dashboard

**REQ-CLS-001 — Dashboard 概览**

> **Event-driven**：当用户进入 Ceph 集群 Dashboard 时，系统应加载并展示以下信息：
> - **集群健康状态**：HEALTH_OK（绿色）/ HEALTH_WARN（黄色）/ HEALTH_ERR（红色），含状态描述文本
> - **容量概览**：环形进度图展示总容量、已用容量、剩余容量，含数值和百分比
> - **IOPS 趋势图**：最近 24 小时的读/写 IOPS 折线图
> - **带宽趋势图**：最近 24 小时的读/写带宽（MiB/s）折线图
> - **PG 状态分布**：饼图展示 active+clean / degraded / peering / inconsistent / stuck 等状态的 PG 数量和比例
> - **节点状态概览**：MON 节点健康数/总数、OSD up 数/总数、MGR 节点状态
>
> **State-driven**：当集群状态为 HEALTH_WARN 时，Dashboard 应在健康状态卡片中展示具体的 Warning 信息列表（如"1 OSDs are down"）。
>
> **State-driven**：当集群状态为 HEALTH_ERR 时，Dashboard 应以醒目的红色横幅展示 Error 详情，并自动刷新间隔缩短至 10 秒。

**REQ-CLS-002 — Dashboard 自动刷新**

> **Ubiquitous**：Dashboard 应始终以 30 秒间隔自动刷新数据。用户可手动点击"刷新"按钮立即刷新，也可通过下拉菜单切换刷新间隔（10秒/30秒/60秒/暂停）。

**REQ-CLS-003 — 容量预警**

> **Event-driven**：当集群总使用率超过 70% 时，容量环图应变为黄色警告色，并触发告警通知（警告级别）。
>
> **Event-driven**：当集群总使用率超过 85% 时，容量环图应变为橙色，并触发严重告警通知。注意：Ceph 集群 nearfull ratio 默认为 85%，超过此阈值可能触发集群自我保护机制。
>
> **Event-driven**：当集群总使用率超过 95% 时，容量环图应变为红色，并在 Dashboard 顶部显示持久化警告横幅，同时触发紧急告警并自动限制写操作（仅允许只读查询）。
>
> **设计理由**：HDD 集群扩容涉及硬件采购、上架、OSD 部署、数据重平衡，全流程可能需要数周。将黄色预警从 80% 下调至 70%，为运维团队留出充足的采购和部署时间。

**REQ-CLS-003-A — 容量趋势预测**

> **Event-driven**：当用户查看容量趋势时，系统应基于历史用量数据（最近 90 天）展示以下信息：
> - 过去 90 天的集群用量增长趋势折线图
> - 基于线性回归的预计容量耗尽可能日期（如"按当前增长速度，预计 2026-12-15 达到 85% 容量"）
> - 按 Pool 维度的增长趋势分解（各 Pool 的用量增长贡献占比）
> - 过去 30 天日均增长量和过去 7 天日均增长量（用于识别加速趋势）
>
> **State-driven**：当预测耗尽日期小于 90 天时，系统应在容量趋势卡片中显示警告标记。
>
> **State-driven**：当预测耗尽日期小于 30 天时，系统应触发严重告警通知。

**REQ-CLS-003-B — Pool 级容量告警**

> **Event-driven**：当单个 Pool 使用率超过 70% 时，系统应在 Pool 列表中显示黄色警告标记，并触发告警通知（警告级别）。
>
> **Event-driven**：当单个 Pool 使用率超过 85% 时，系统应触发严重告警通知。

#### 4.3.2 OSD 管理

**REQ-CLS-004 — OSD 列表**

> **Event-driven**：当用户进入 OSD 列表页时，系统应加载并展示所有 OSD 的列表，包含以下字段：OSD ID、所属节点（主机名）、状态（up/down）、In/Out 状态、容量（TiB/GiB）、使用量（TiB/GiB）、使用率（百分比进度条）、PG 数量、权重（CRUSH Weight）、设备路径、守护进程版本。
>
> **Ubiquitous**：OSD 列表应始终支持按状态筛选（全部/up/down/in/out）和按节点筛选。

**REQ-CLS-005 — OSD 状态变更**

> **Event-driven**：当用户对 OSD 执行 out 操作（标记为 out）时，系统应弹出二次确认——"标记 OSD.{id} 为 out 将触发数据重新平衡（backfill/recovery），可能影响集群性能。确认继续？"。
>
> **Event-driven**：当用户对 out 状态的 OSD 执行 in 操作时，系统应直接执行，无需确认。
>
> **Unwanted**：如果用户尝试删除 OSD，则系统应触发高危操作审批流程——生成审批工单，需管理员审批通过后才执行，并先确保 OSD 已标记为 out 且数据已完成迁移。

**REQ-CLS-006 — OSD 磁盘故障处理**

> **Unwanted**：如果检测到 OSD 状态变为 Down 且持续时间超过 5 分钟，则系统应：
> 1. 自动触发告警通知
> 2. 在 OSD 列表中高亮标记该 OSD（红色背景闪烁）
> 3. 在 OSD 操作列显示"一键隔离"按钮（标记 out + 通知管理员）

---

#### 4.3.3 Pool 管理

**REQ-CLS-007 — Pool 列表**

> **Event-driven**：当用户进入 Pool 列表页时，系统应加载并展示所有 Pool 的列表，包含以下字段：Pool 名称、Pool ID、类型（replicated/erasure-coded）、副本数（replicated）/ EC Profile（erasure-coded）、PG 数量（当前/建议）、存储用量、压缩状态（启用/禁用）、压缩算法、关联资源类型（RBD/RGW/CephFS）。
>
> **Ubiquitous**：Pool 列表应始终支持按类型和关联资源类型筛选。

**REQ-CLS-008 — 创建 Pool**

> **Event-driven**：当用户点击"创建 Pool"时，系统应弹出创建表单：
> - Pool 名称（必填，符合 Ceph Pool 命名规范）
> - 类型（必选：副本池 / EC 池）
> - PG 数量（必填，系统应根据 OSD 数量自动计算推荐值，允许用户修改）
> - 副本数（副本池必选，默认 3，支持 1/2/3/4）
> - EC Profile（EC 池必选，下拉选择已有 EC Profile）
> - 压缩策略（选填：无/force/aggressive，默认无）
> - 所属项目（选填，关联到项目用于后续权限管理）
>
> **Event-driven**：创建成功后，系统应自动刷新 Pool 列表。

**REQ-CLS-009 — 修改 Pool 配置**

> **Event-driven**：当用户修改 Pool 的 PG 数时，系统应校验新 PG 数不小于当前值（PG 数仅允许增加）。
>
> **Unwanted**：如果用户尝试减少 PG 数，则系统应拒绝并提示"PG 数仅允许增加，不允许减少，以避免数据迁移风险"。

**REQ-CLS-010 — 删除 Pool**

> **Event-driven**：当用户发起删除 Pool 操作时，系统应执行以下安全检查：
> 1. 检查 Pool 中是否存在 RBD Image（如有则拒绝删除）
> 2. 检查 Pool 是否关联 RGW Bucket（如有则拒绝删除）
> 3. 检查 Pool 是否为 CephFS 的 data/metadata Pool（如是则拒绝删除）
>
> **Unwanted**：如果任一检查不通过，则系统应拒绝删除并明确提示原因。
>
> **Event-driven**：如果所有检查通过，系统应弹出高危操作确认——要求用户输入 Pool 名称，并提供"我确认删除此 Pool，所有数据将永久丢失"的勾选确认。

#### 4.3.4 MON 节点管理

**REQ-CLS-011 — MON 节点列表**

> **Event-driven**：当用户查看 MON 节点列表时，系统应展示所有 MON 节点的状态信息：节点名称、IP 地址、Quorum 参与情况（leader/peon）、版本号、运行时长。
>
> **Ubiquitous**：MON 节点列表应始终以绿色圆点标识 Quorum 参与节点，灰色圆点标识非 Quorum 节点。

---

#### 4.3.5 性能监控

**REQ-CLS-012 — 性能图表**

> **Event-driven**：当用户查看性能监控图表时，系统应展示以下指标的趋势图：
> - 集群读/写 IOPS（ops/s）
> - 集群读/写带宽（MiB/s）
> - 集群读/写延迟（ms）
> - 各 Pool 的 IOPS 和带宽分布（堆叠图）
>
> **Ubiquitous**：所有性能图表应始终支持时间范围切换：最近 1 小时、6 小时、24 小时、7 天、30 天。

**REQ-CLS-013 — 性能数据采集**

> **Ubiquitous**：系统应始终以 60 秒间隔从 Ceph 集群采集性能数据（通过 `ceph pg stat`、`ceph osd perf`、Prometheus exporter 等方式），数据保留周期为 90 天。

#### 4.3.6 告警配置

**REQ-CLS-014 — 告警规则配置**

> **Event-driven**：当用户创建告警规则时，系统应提供以下配置项：
> - 规则名称（必填）
> - 监控指标（必选）：集群使用率 / OSD Down 数量 / PG Degraded 比例 / PG Stuck 数量 / MON 节点 Down / MGR 节点 Down / Pool 使用率
> - 触发条件（必填）：阈值（如使用率 > 70%）、持续时间（如持续 5 分钟）
> - 告警级别（必选）：严重 / 警告 / 通知
> - 通知渠道（必选，多选）：企业微信 / 邮件 / 短信 / 站内通知
> - 通知对象（必填）：用户/用户组
> - 启用状态（默认启用）

**REQ-CLS-015 — 告警触发**

> **Event-driven**：当监控指标达到告警规则设定的阈值并持续指定时间后，系统应触发告警：
> 1. 在告警历史中创建一条告警记录（状态：触发中）
> 2. 通过配置的通知渠道发送告警消息
> 3. 在 Dashboard 顶部展示告警横幅（严重级别为红色，警告级别为黄色）

**REQ-CLS-016 — 告警恢复**

> **Event-driven**：当触发告警的指标恢复到正常范围并持续 5 分钟后，系统应：
> 1. 将告警记录状态更新为"已恢复"
> 2. 发送恢复通知
> 3. 移除 Dashboard 顶部的对应告警横幅

**REQ-CLS-017 — 告警确认/静默**

> **Event-driven**：当用户对告警点击"确认"时，系统应将告警状态更新为"已确认"，停止重复通知，但告警横幅保留。
>
> **Event-driven**：当用户设置告警静默时（指定静默时长），系统应在静默期间暂停该规则的通知发送，静默到期后自动恢复。

### 4.4 备份管理

> **定位说明**：基于 Ceph 集群中已有的 `kvm-backup` Pool（251 GiB），统一管理虚拟机备份策略、任务和恢复流程。与块存储模块的 RBD 快照（存储层即时快照）区分——备份管理面向应用层的备份数据生命周期管理。

#### 4.4.1 备份策略

**REQ-BKP-001 — 备份策略配置**

> **Event-driven**：当用户创建备份策略时，系统应提供以下配置项：
> - 策略名称（必填，2-64 字符）
> - 备份周期（必选）：每日 / 每周 / 自定义 Cron 表达式
> - 备份时间窗口（必填）：指定可执行备份的时间段（如"每日 02:00-06:00"），避免影响业务
> - 保留份数（必填，最小 1，默认 7）：超过保留份数的旧备份自动标记为可清理
> - 目标存储池（必填，默认 `kvm-backup`）
> - 关联虚拟机（必填，支持多选）：应用该策略的虚拟机列表
> - 启用状态（默认启用）

**REQ-BKP-002 — 备份策略编辑**

> **Event-driven**：当用户修改备份策略时，系统应校验：
> - 保留份数不可减少到低于当前实际保留的备份数量
> - 修改备份周期后，下一次备份时间按新周期重新计算
>
> **Unwanted**：如果用户尝试删除仍有关联虚拟机的备份策略，则系统应拒绝并提示"该策略已关联 {N} 台虚拟机，请先解除关联后再删除"。

---
#### 4.4.2 备份任务

**REQ-BKP-003 — 备份任务列表**

> **Event-driven**：当用户进入备份任务列表页时，系统应根据当前用户的项目权限，加载并展示其有权查看的备份任务列表。
>
> **Ubiquitous**：备份任务列表应始终展示以下字段：任务 ID、源虚拟机名称、备份策略名称（手动备份则显示"手动"）、开始时间、结束时间、备份大小（GiB/MiB）、耗时、状态（进行中/成功/失败/已清理）、操作（查看详情/恢复/清理）。

**REQ-BKP-004 — 手动备份**

> **Event-driven**：当用户点击"手动备份"时，系统应弹出表单：
> - 源虚拟机（必填，下拉选择已纳管的虚拟机列表）
> - 备份名称（必填，2-64 字符）
> - 描述（选填）
> - 目标存储池（必填，默认 `kvm-backup`）
>
> **Event-driven**：提交后，系统应创建备份任务并立即执行，备份任务列表中新增一条状态为"进行中"的记录。
>
> **Unwanted**：如果目标存储池使用率超过 90%，则系统应拒绝创建并提示"存储池 {pool_name} 容量不足，请清理过期备份或联系管理员扩容"。

**REQ-BKP-005 — 自动备份执行**

> **State-driven**：系统应根据各备份策略的配置，在指定时间窗口内自动触发备份任务。每个备份窗口内，系统应按策略优先级顺序执行备份任务，同一时刻最多执行 2 个并发备份任务（避免对 Ceph 集群和源虚拟机造成过大负载）。
>
> **Unwanted**：如果备份任务执行失败，则系统应：
> 1. 记录失败原因（如"虚拟机不可达"、"存储池空间不足"、"备份超时"）
> 2. 触发告警通知（警告级别）
> 3. 自动重试 1 次（间隔 30 分钟），重试仍失败则标记为最终失败
>
> **Event-driven**：备份完成后，系统应根据保留策略检查是否需要清理过期备份——如果当前备份数量超过策略配置的保留份数，则自动标记最早的备份为"待清理"。

**REQ-BKP-006 — 备份详情**

> **Event-driven**：当用户点击某个备份任务查看详情时，系统应展示：
> - 基本信息：任务 ID、备份名称、源虚拟机、备份策略、开始/结束时间、耗时、大小
> - 备份内容摘要：备份的磁盘列表（磁盘名称、大小）
> - 存储信息：存储池、RBD Image 名称、实际占用大小
> - 状态历史：创建 → 进行中 → 成功/失败 的状态流转

---
#### 4.4.3 备份恢复

**REQ-BKP-007 — 从备份恢复**

> **Event-driven**：当用户对某个备份任务点击"恢复"时，系统应弹出恢复表单：
> - 目标虚拟机（必填，下拉选择已纳管的虚拟机列表，默认选择源虚拟机）
> - 恢复确认（必填，勾选"我确认将覆盖目标虚拟机的当前磁盘数据，此操作不可逆"）
>
> **Unwanted**：如果目标虚拟机处于运行状态，则系统应拒绝恢复并提示"请先关闭目标虚拟机后再执行恢复操作"。
>
> **Event-driven**：确认恢复后，系统应创建恢复任务，展示恢复进度。
>
> **Unwanted**：如果恢复任务执行失败，则系统应记录失败原因，保留原备份数据不变，并通知用户。

---
#### 4.4.4 备份空间管理

**REQ-BKP-008 — 备份空间趋势**

> **Event-driven**：当用户进入备份空间概览页时，系统应展示：
> - `kvm-backup` Pool 的容量概览：总容量、已用容量、使用率
> - 最近 90 天的备份空间使用趋势折线图
> - 预计容量耗尽可能日期（基于历史增长趋势的线性预测）
> - 按策略维度：各备份策略的备份数据量汇总
> - 按虚拟机维度：各虚拟机的备份数据量排行

**REQ-BKP-009 — 备份清理**

> **Event-driven**：当用户手动清理备份时，系统应弹出二次确认——"清理备份 {backup_name} 将永久删除备份数据，此操作不可逆。确认继续？"，并要求用户输入备份名称确认。
>
> **State-driven**：系统应根据各备份策略的保留份数配置，自动标记超出保留份数的最早备份为"待清理"状态。
>
> **Event-driven**：管理员可在"待清理"列表中批量确认清理或手动保留（标记为"保留"）。
>
> **Unwanted**：如果备份正在被恢复任务使用，则系统应拒绝清理并提示"该备份正在执行恢复操作，请等待恢复完成后再清理"。

---
#### 4.4.5 审计

**REQ-BKP-010 — 备份操作审计**

> **Ubiquitous**：所有备份相关操作（创建策略/修改策略/删除策略/手动备份/自动备份/恢复/清理）应始终记录审计日志，包含：操作人、操作时间、操作类型、操作对象、操作结果。

## 七、边界场景与异常处理

### 场景 1：上传超大文件（>5 GiB）

| 维度 | 描述 |
|------|------|
| **触发条件** | 用户通过 Web 界面上传超过 5 GiB 的单个文件 |
| **系统行为** | 自动启用分片上传（Multipart Upload），分片大小 10 MiB。上传过程中实时展示分片级别的进度。 |
| **异常处理** | 如果单个分片上传失败，自动重试该分片（最多 3 次）。如果累计失败分片超过总数的 10%，则终止上传并提示用户"上传失败率过高，请检查网络后重试"。 |
| **边界限制** | 单文件最大 100 GiB（受 RGW 配置限制）。超过此限制的文件在拖拽时即被拦截并提示"文件大小超过 100 GiB 限制"。 |

### 场景 4：云盘扩容后通知 OS 识别新容量

### 场景 14：混部环境 — OSD out/in 操作窗口限制

| 维度 | 描述 |
|------|------|
| **触发条件** | 用户尝试在非低峰期执行 OSD out/in 操作 |
| **背景** | 6 个节点同时运行 KVM 虚拟机（qcow2）和 Ceph（OSD/MON/MGR/RGW），OSD out/in 操作会触发数据重平衡（backfill/recovery），大量占用 HDD IO 和网络带宽，可能影响 KVM 虚拟机磁盘 IO 和 RGW 对象读写延迟。 |
| **系统行为** | 1. 系统检查当前时间是否在允许的操作窗口内（默认每日 02:00-06:00）<br>2. 如果在窗口外，弹出提示并引导用户选择"立即执行（风险自担）"或"预约到下一窗口执行"<br>3. 如果选择"立即执行"，需要输入操作理由并记录审计日志 |
| **用户提示** | "OSD out/in 操作将触发数据重平衡，可能影响集群性能和 KVM 虚拟机 IO。当前不在建议操作窗口（02:00-06:00）内。建议 [预约到下一窗口执行] 或 [立即执行（风险自担）]。<br>预约操作将在 {next_window_time} 自动执行。" |
| **预约执行** | 预约操作存储在任务队列中，到达窗口时间后自动执行。执行前再次检查集群状态（HEALTH_OK 才执行）。 |

### 场景 15：混部环境 — backfill/recovery 限速

| 维度 | 描述 |
|------|------|
| **触发条件** | OSD out/in 或 PG 调整操作触发数据迁移（backfill/recovery） |
| **背景** | 在混部环境中，如果 backfill/recovery 不限速，HDD 的 IO 带宽会几乎全部被数据迁移占用，导致 KVM 虚拟机磁盘 IO 严重下降和 RGW 请求超时。 |
| **系统行为** | 1. 执行可能触发数据迁移的操作前，系统自动应用限速参数：<br>&nbsp;&nbsp;&nbsp;- `osd_max_backfills = 1`（每个 OSD 同时只做 1 个 backfill）<br>&nbsp;&nbsp;&nbsp;- `osd_recovery_max_active = 1`（每个 OSD 同时只做 1 个 recovery）<br>&nbsp;&nbsp;&nbsp;- `osd_recovery_op_priority = 1`（降低 recovery 优先级，保障客户端 IO）<br>2. 数据迁移期间，Dashboard 展示迁移进度（已完成/总量/速度/预计剩余时间）<br>3. 系统持续监控客户端 IOPS 和延迟，如果客户端性能下降超过 50%，自动暂停迁移 |
| **用户提示** | "数据迁移进行中（已完成 {N} GiB / 总量 {M} GiB，速度 {speed} MiB/s，预计剩余 {eta}）。已自动启用限速保护，客户端 IO 优先级高于数据迁移。" |
| **迁移完成后** | 迁移完成后自动恢复默认参数，发送通知告知管理员 |

### 场景 16：混部环境 — PG 调整操作窗口限制

| 维度 | 描述 |
|------|------|
| **触发条件** | 用户尝试执行 PG 数调整操作 |
| **背景** | PG 调整会触发大量数据迁移，影响范围比单个 OSD out/in 更大。在混部环境下，PG 调整需要更严格的时间窗口限制。 |
| **系统行为** | 1. PG 调整操作仅允许在周末低峰期窗口（周六/周日 00:00-08:00）执行<br>2. 操作前自动检测当前 IOPS 和带宽使用率，如果超过正常水平的 50%，延迟执行<br>3. 执行前系统自动应用限速参数（同场景 15）<br>4. 仅允许增加 PG 数，不允许减少 |
| **用户提示** | "PG 数调整将触发大量数据迁移。此操作仅允许在周末低峰期窗口（周六/周日 00:00-08:00）执行。当前时间不满足条件，建议 [预约执行]。" |

---
### 场景 17：混部环境 — 大量并发写入保护

| 维度 | 描述 |
|------|------|
| **触发条件** | 短时间内创建大量 RBD Image（>10 个/分钟）或上传大量大文件（>50 个并发分片上传） |
| **背景** | 在混部环境中，大量并发写入操作可能挤占 KVM 虚拟机的磁盘 IO 资源。 |
| **系统行为** | 1. 系统对以下操作实施并发限流：<br>&nbsp;&nbsp;&nbsp;- RBD Image 创建：最大并发 3 个<br>&nbsp;&nbsp;&nbsp;- RGW 分片上传：最大并发分片数 100 个<br>2. 超过并发限制的请求排队等待，按 FIFO 顺序执行 |
| **用户提示** | "当前并发操作已达上限，您的请求已加入队列（位置 {position}），预计 {eta} 后开始执行。" |

### 8.3 权限规则说明

1. **项目隔离**：标记"本项目"的权限仅对用户所属项目的资源生效。超级管理员和存储管理员可跨项目访问。
2. **高危操作审批**：删除 OSD、删除 Pool、回滚快照等不可逆操作，即使有权限也需要触发审批流程，审批通过后才执行。

4. **只读用户**：仅可查看监控数据和资源状态，不可执行任何写操作。
