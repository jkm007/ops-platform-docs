### T003 — 统一错误码 + 响应格式 + 请求ID

**输入（依赖）**：T001

**输出（交付物）**：
- `backend/internal/dto/response.go`：统一响应结构体
- `backend/internal/dto/errors.go`：错误码枚举
- `backend/internal/middleware/request_id.go`：请求ID中间件

错误码枚举：

```go
const (
    CodeSuccess         = 0
    CodeBadRequest      = 1001  // 参数错误
    CodeUnauthorized    = 1002  // 未授权
    CodeForbidden       = 1003  // 无权限
    CodeNotFound        = 1004  // 资源不存在
    CodeConflict        = 1005  // 资源冲突
    CodeKVMError        = 2001  // KVM 操作失败
    CodeCephError       = 2002  // Ceph 操作失败
    CodeNetworkError    = 3001  // 网络操作失败
    CodeInternalError   = 5000  // 内部错误
)
```

**技术要点**：
- `request_id` 由中间件自动生成（UUID），注入 context，记录到日志和响应
- Handler 层统一返回 `dto.Response{Code, Msg, Data, RequestId}`
- 定义 `dto.PageRequest` 和 `dto.PageResponse` 通用分页结构
- 错误码常量用 `const` + iota 分组定义，便于扩展

**验收标准**：
- 所有 API 响应格式统一为 `{"code":0,"msg":"success","data":{},"request_id":"uuid"}`
- 错误响应格式统一，错误码对应含义正确
- 请求ID 在日志中可追踪

**预估工时**：1 人天

### T004 — JWT 认证 + Redis Token 黑名单

**输入（依赖）**：T001, T002

**输出（交付物）**：
- `backend/internal/middleware/jwt.go`：JWT 中间件
- `backend/internal/handler/auth.go`：登录/登出/当前用户 Handler
- `backend/internal/service/auth_service.go`：登录逻辑（密码验证、Token 签发）
- `backend/internal/model/user.go`：User 模型（含 password_hash bcrypt 存储）

API 端点：
```
POST /api/v1/auth/login    # 登录 {username, password} → {token, expires_at, user_info}
POST /api/v1/auth/logout   # 登出 {token} → 加入 Redis 黑名单
GET  /api/v1/auth/me       # 获取当前用户信息（从 JWT 解析）
```

**技术要点**：
- JWT 使用 `golang-jwt/jwt/v5`，签名算法 HS256
- Token 有效期：Access Token 2小时，Refresh Token 7天（一期可先不做Refresh）
- 登出时将 Token 加入 Redis 黑名单，TTL = Token 剩余有效期
- 中间件从 Authorization Header 提取 Bearer Token
- 中间件先检查 Redis 黑名单，再验证 JWT
- 中间件将 `user_id`, `tenant_id`, `username` 注入 context

**验收标准**：
- 使用预置 admin 账号登录成功，返回 Token
- 使用错误密码登录返回 1002 错误
- 携带 Token 访问受保护接口成功
- 无 Token 访问受保护接口返回 1002
- 登出后，原 Token 访问受保护接口返回 1002

**预估工时**：2 人天

### T005 — RBAC 权限中间件 + 6个预定义角色

**输入（依赖）**：T004

**输出（交付物）**：
- `backend/internal/middleware/rbac.go`：RBAC 权限检查中间件
- `backend/internal/service/permission_service.go`：权限验证逻辑
- `backend/migrations/000001_init.up.sql`（T002中已含）角色预置数据：

6个预定义角色：

| 角色 | 标识 | 权限范围 |
|------|------|---------|
| 超级管理员 | super_admin | `*:*` — 所有资源所有操作 |
| 平台管理员 | platform_admin | `host:*`, `image:*`, `spec:*`, `user:read`, `audit:read` |
| 安全管理员 | security_admin | `user:*`, `role:*`, `tenant:*`, `audit:*` |
| 租户管理员 | tenant_admin | 本租户内所有资源的所有操作 |
| 租户运维 | tenant_operator | 本租户内资源的日常运维操作（不含删除/纳管/配额管理） |
| 只读用户 | readonly | 所有资源的只读操作 |

**技术要点**：
- 权限格式：`resource:action`（如 `vm:create`, `host:delete`, `*:*`）

- 中间件从 context 获取 `user_id` → 查 `user_roles` → 查 `roles.permissions` → 匹配请求的 `resource:action`
- 租户级别权限：非平台级角色只能操作 `tenant_id` 匹配的资源
- 定义中间件工厂函数 `RequirePermission(resource, action string)` 用于路由注册

**验收标准**：
- super_admin 可访问所有接口
- readonly 用户调用 POST /api/v1/vms 返回 1003
- 租户A用户查看租户B的虚拟机返回 1003 或空列表
- 角色权限 JSON 格式正确，可解析

**预估工时**：2 人天

### T037 — 虚拟机创建向导完整流程（7步）API

- `backend/internal/service/resource_validator.go`：资源校验

创建流程 API 内部逻辑：

```
Step 1 - 选择宿主机：校验宿主机在线、资源充足（考虑超分比）
Step 2 - 选择镜像：校验镜像可用、cloud-init 支持检测
Step 3 - cloud-init 配置：生成 meta-data + user-data
Step 4 - 选择规格：校验规格可用、资源匹配
Step 5 - 选择存储类型：本地 qcow2（校验 /home/kvm/images/ 空间）或 Ceph RBD（校验 Pool 空间）
Step 6 - 选择网络：校验网络可用、IP 分配（DHCP 或手动指定）
Step 7 - 确认创建：
  1. 原子性资源预留（SELECT ... FOR UPDATE 锁宿主机资源记录）
  2. 生成 cloud-init ISO（通过 Agent）
  3. 创建磁盘（qemu-img create 或 rbd create，通过 Agent）
  4. 构建 domain XML
  5. 定义 + 启动 VM（通过 Agent）
  6. 更新数据库（VM + disk 记录）
  7. 释放资源预留锁
```

**技术要点**：
- 创建流程支持"预校验"和"正式提交"两个阶段
